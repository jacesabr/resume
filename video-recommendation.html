Video Recommendation Engine

Building a two-stage recommendation system for video platforms — candidate generation, ranking, cold start, and the filter bubble problem.

9 sections · Collaborative filtering, content-based, Two-Tower, cold start, evaluation

The Problem: 'What Should You Watch Next?'

500 hours of video uploaded to YouTube every minute. 800 million videos on TikTok. The recommendation engine must narrow billions of candidates to a ranked list of 10-20 in <200ms. This is a two-stage system: candidate generation (retrieve 100s from millions) → ranking (score and order the 100s).

Scale

YouTube: 2B+ users, 800M+ videos. Must generate personalized recommendations for each user in real-time. Candidate generation must filter 800M → 500 in <50ms.

Business Impact

Recommendations drive 70%+ of watch time on YouTube and Netflix. A 0.1% improvement in recommendation quality = millions in ad revenue and subscriber retention.

Mathematical Foundation

\[P(\text{watch} \mid u, v, c) = f(u, v, c; \theta)\]

Core prediction: probability user u watches video v in context c. The ranking function.

\[\text{score}(u, v) = \mathbf{u}^T \mathbf{v}\]

Dot product of user embedding and video embedding — the foundation of candidate generation. Fast approximate nearest neighbor search.

\[\text{Loss} = -\sum_{(u,v^+)} \log\sigma(\mathbf{u}^T\mathbf{v}^+) - \sum_{(u,v^-)} \log\sigma(-\mathbf{u}^T\mathbf{v}^-)\]

Contrastive loss: push user embedding close to watched videos (v+) and away from unwatched (v-).

\[\text{E}[\text{watch time}] = P(\text{click}) \times \text{E}[\text{duration} \mid \text{click}]\]

YouTube's actual objective: expected watch time, not click probability. Prevents clickbait optimization.

Production Architecture: Two-Tower Neural Model

YouTube, TikTok, and Netflix use two-tower models: one tower encodes the user, one encodes the item. The dot product of the two embeddings is the relevance score. This allows pre-computing item embeddings offline.

Python 3
# Two-Tower Model — production recommendation architecture
"kw">class="kw">import torch
"kw">class="kw">import torch.nn "kw">class="kw">as nn

"kw">class TwoTowerModel(nn.Module):
    "kw">class="kw">def __init__(self, n_users, n_videos, n_categories, embed_dim=64):
        super().__init__()
        
        # User tower
        self.user_embed = nn.Embedding(n_users, embed_dim)
        self.user_tower = nn.Sequential(
            nn.Linear(embed_dim + 3, 128),  # +3 "kw">class="kw">for user features
            nn.ReLU(),
            nn.Linear(128, embed_dim)
        )
        
        # Video tower
        self.video_embed = nn.Embedding(n_videos, embed_dim)
        self.category_embed = nn.Embedding(n_categories, 16)
        self.video_tower = nn.Sequential(
            nn.Linear(embed_dim + 16 + 4, 128),  # +category +video features
            nn.ReLU(),
            nn.Linear(128, embed_dim)
        )
    
    "kw">class="kw">def forward(self, user_ids, user_features, video_ids, video_cats, video_features):
        # User representation
        u = self.user_embed(user_ids)
        u = torch.cat([u, user_features], dim=1)
        u = self.user_tower(u)
        u = torch.nn.functional.normalize(u, dim=1)
        
        # Video representation
        v = self.video_embed(video_ids)
        vc = self.category_embed(video_cats)
        v = torch.cat([v, vc, video_features], dim=1)
        v = self.video_tower(v)
        v = torch.nn.functional.normalize(v, dim=1)
        
        # Dot product similarity
        "kw">class="kw">return (u * v).sum(dim=1)

# Key advantage: video embeddings computed OFFLINE
# At serving time: only compute user tower + ANN lookup
# This is how YouTube serves recommendations in <200ms

"kw">class="kw">print("Video tower runs offline → FAISS index")

Failure Mode ✗ — Filter Bubbles & Popularity Bias

The echo chamber problem. If the model only recommends what users already like, it creates filter bubbles. If it only recommends popular content, niche creators never surface. Both hurt long-term platform health.

Key Takeaways & Production Checklist

When This Approach Works

✅ Two-stage architecture: fast retrieval (ANN) → precise ranking (ML model)
✅ Collaborative filtering for warm users, content-based for cold start
✅ Optimize for watch time, not clicks — prevents clickbait problem
✅ Two-tower models enable pre-computing video embeddings offline
✅ A/B test everything — offline NDCG gains don't always improve user satisfaction

Common Pitfalls

❌ Optimizing only for engagement → filter bubbles, misinformation amplification
❌ Popularity bias — power law means top 1% get 50%+ of recommendations
❌ Cold start: new users and new videos have no interaction data
❌ Position bias: users click top results regardless of quality
❌ Training on implicit feedback (views) without accounting for exposure bias
