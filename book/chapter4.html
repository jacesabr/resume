4.1

Layer Normalisation

Before any attention or feedforward operation touches the residual stream, GPT-2 normalises it. Chapter 3 (§3.4) derives the mechanism from first principles; here we implement it and focus on the practical details that matter when building a GPT model. Without normalisation, training a 12-layer transformer at a constant learning rate fails within the first thousand steps — activations compound multiplicatively through each block, sending gradients into overflow.

GPT-2 uses Pre-LN: normalise before the sublayer, add the raw residual after. Xiong et al. 2020 proved Pre-LN converges 40% faster than Post-LN. The formula: subtract the mean, divide by standard deviation, apply learned scale γ and shift β. At init, LayerNorm is an exact identity (γ=1, β=0). RMSNorm (used in LLaMA/T5/Gemma) drops mean-centering for a 7% speedup.

LAYER NORMALISATION — Ba et al. 2016 (arXiv 1607.06450) LN(x) = γ · (x − μ) / √(σ² + ε) + β μ = mean over d_model features  |  σ² = variance  |  γ,β ∈ ℝd learned  |  ε = 1e-5
Pre-LN vs Post-LN: Pre-LN (GPT-2) normalises the input before the sublayer; the raw residual is added after. Post-LN (Vaswani 2017) normalises after the residual addition. Pre-LN gives a gradient highway — the bypass gradient is always 1.0, so early layers always receive a usable signal regardless of depth. Convergence is 40% faster and warmup requirements are lower (Xiong et al. 2020).
layer_norm.py
import math

class LayerNorm:
    def __init__(self, emb_dim, eps=1e-5):
        self.eps   = eps
        self.scale = [1.0] * emb_dim  # ↑ gamma: init=1 so LN is identity at training start
        self.shift = [0.0] * emb_dim  # ↑ beta: init=0 so LN is identity at training start

    def forward(self, x):
        # x: list of floats, length emb_dim (one token's 768-dim embedding vector)
        n    = len(x)
        mean = sum(x) / n                          # ↑ mu: mean across feature dim (NOT batch dim)
        var  = sum((v - mean)**2 for v in x) / n  # ↑ sigma^2: variance across 768 features
        std  = math.sqrt(var + self.eps)            # ↑ add epsilon BEFORE sqrt: prevents div-by-zero
        norm = [(v - mean) / std for v in x]       # ↑ zero-mean, unit-variance across feature dim
        return [self.scale[i]*norm[i] + self.shift[i]
                for i in range(n)]              # ↑ affine: gamma * x_norm + beta (both learned)

# RMSNorm (Zhang & Sennrich 2019, arXiv 1910.07467) -- used in LLaMA, T5, Gemma
class RMSNorm:
    def __init__(self, emb_dim, eps=1e-5):
        self.eps   = eps
        self.scale = [1.0] * emb_dim  # ↑ only gamma -- no beta; simpler, 7% faster than LayerNorm

    def forward(self, x):
        n   = len(x)
        rms = math.sqrt(sum(v**2 for v in x) / n + self.eps)  # ↑ root mean square; no centering
        return [self.scale[i] * x[i] / rms for i in range(n)]    # ↑ scale by gamma/RMS only

4.2

The GELU Activation

Without a nonlinear activation function, stacking linear layers collapses to a single linear transformation regardless of depth. A 12-layer transformer without nonlinearity would be mathematically equivalent to one matrix multiplication — no representational hierarchy, no capacity to approximate complex functions. Nonlinearity is the mechanism by which depth provides exponentially more expressive power than width alone.

GPT-2 uses GELU — Gaussian Error Linear Unit — introduced by Hendrycks and Gimpel in 2016 (arXiv 1606.08415). The defining idea is a probabilistic gate: GELU(x) = x × Φ(x), where Φ is the Gaussian CDF. For large positive x, Φ(x)→1 and GELU(x)→x. For large negative x, Φ(x)→0 and GELU(x)→0. The transition is smooth and differentiable everywhere. Compare to ReLU(x) = max(0,x): a hard gate whose gradient is exactly 0 for all negative inputs. Once a ReLU neuron goes negative it receives zero gradient and cannot recover — the “dying ReLU” problem. GELU(-1)≈-0.159, not 0; the neuron still participates weakly and can recover via that nonzero gradient.

The exact GELU requires the Gaussian CDF, computed with a tanh approximation in GPT-2 (error < 0.0005 across practical inputs). SwiGLU (Shazeer 2020, arXiv 2002.05202) goes further: the FFN up-projection is doubled, one half is passed through swish(x) = x⋅σ(x), and gates the other half elementwise. SwiGLU consistently outperforms plain GELU at matched parameter counts and is used in LLaMA-2, LLaMA-3, and PaLM, with inner dimension 8/3×d_model instead of 4× to keep totals equivalent.

GELU ACTIVATION — Hendrycks & Gimpel 2016 (arXiv 1606.08415) GELU(x) = x · Φ(x) ≈ 0.5x · (1 + tanh(√(2/π) · (x + 0.044715x³))) Φ(x) = Gaussian CDF.   GELU(0)=0   GELU(-1)≈-0.159   GELU(1)≈0.841   GELU(+∞)→+∞

ReLU(x) = max(0, x)

Hard gate. Gradient is exactly 0 for all negative inputs. “Dying ReLU”: once a neuron goes negative it receives no gradient and cannot recover. Simple and fast but brittle in very deep networks at high learning rates.

ReLU(-1) = 0.000   ReLU(1) = 1.000

Smooth probabilistic gate. Negative values suppressed but not zeroed — gradient flows for recovery. Always differentiable. Used in GPT-2, GPT-3, BERT, Mistral. LLaMA-2/3 uses SwiGLU variant.

activations.py
import math

def gelu(x):
    # GPT-2 tanh approximation of GELU (Hendrycks & Gimpel 2016)
    inner = math.sqrt(2.0/math.pi) * (x + 0.044715*x**3)  # ↑ polynomial approx of Gaussian CDF argument
    return 0.5 * x * (1.0 + math.tanh(inner))             # ↑ x * P(Gaussian <= x): smooth probabilistic gate

def relu(x):
    return max(0.0, x)  # ↑ hard gate: negative inputs silenced, gradient = 0 there (dying ReLU risk)

def swish(x):
    # Used inside SwiGLU (Shazeer 2020, LLaMA-2)
    return x / (1.0 + math.exp(-x))  # ↑ x * sigmoid(x): smooth gate, always differentiable

def gelu_grad(x):
    # d/dx GELU(x) = Phi(x) + x * phi(x)  (product rule: d/dx [x * Phi(x)])
    cdf = 0.5*(1.0+math.tanh(math.sqrt(2.0/math.pi)*(x+0.044715*x**3)))
    pdf = math.exp(-x**2/2) / math.sqrt(2*math.pi)
    return cdf + x*pdf  # ↑ nonzero even for negative x -- no dying neuron problem

The FeedForward Sublayer

Chapter 3 (§3.10) introduced the feedforward sublayer conceptually — how it handles computation within each position while attention handles communication between positions. Here we implement it and count every parameter.

The FFN is a two-layer MLP: linear expansion by 4×, GELU elementwise, then contraction back to d_model. For GPT-2 small (d_model = 768): 768 → 3,072 → 768. The 4× factor traces to Vaswani et al. 2017 and has proven so reliable that virtually every subsequent transformer kept it. Geva et al. 2021 showed these layers function as key-value memory stores: each W1 row detects a pattern; the corresponding W2 row stores a value. Factual knowledge lives here.

Parameter dominance: GPT-2 small: W1 = 768×3072 = 2,359,296 + 3,072 biases; W2 = 3072×768 = 2,359,296 + 768 biases. Total per FFN: 4,722,432. Across 12 blocks: 56,669,184 — 45.6% of the entire model. Attention QKV+O projections hold only 28.5%. The FFN is where GPT-2 stores most of its knowledge.
feedforward.py
import math

def gelu(x):
    inner = math.sqrt(2.0/math.pi) * (x + 0.044715*x**3)
    return 0.5 * x * (1.0 + math.tanh(inner))

class FeedForward:
    def __init__(self, d_model, expansion=4):
        self.d  = d_model
        self.dh = expansion * d_model  # ↑ d_ff = 4*d: Vaswani 2017 convention kept by all GPT variants
        # W1: (d_model, d_ff)  -- up-projection: each row is a "key" pattern detector
        # W2: (d_ff, d_model)  -- down-projection: each row is a "value" to retrieve

    def param_count(self):
        d, h = self.d, self.dh
        return {"W1": d*h, "b1": h, "W2": h*d, "b2": d, "total": d*h+h+h*d+d}

    def forward_1token(self, x, W1, b1, W2, b2):
        # x: one token's d_model-dim vector -- no cross-token communication here
        h = [gelu(sum(x[j]*W1[i][j] for j in range(self.d)) + b1[i])
             for i in range(self.dh)]  # ↑ 4x wider: each neuron detects a different feature pattern
        return [sum(h[i]*W2[j][i] for i in range(self.dh)) + b2[j]
                for j in range(self.d)]  # ↑ read out "values" for activated key neurons (Geva 2021)

Shortcut Connections

Chapter 3 (§3.11) proved why residual connections solve the vanishing gradient problem. Here we implement the pattern and see how it enables GPT-2’s 12-layer stack — and GPT-3’s 96-layer stack — to train at all.

The mechanism: each block computes F(x) and returns x + F(x). The gradient of the bypass term is always exactly 1.0, guaranteeing a gradient highway from output to input regardless of depth. At initialisation with small weights, F(x)≈0, so x + F(x)≈x — the entire block stack begins as an identity function. Each block learns to add a small correction rather than a full new representation. Elhage et al. 2021 formalised this as the residual stream view (covered in §4.11).

Gradient highway: ∂L/∂x = ∂L/∂(x+F(x)) × (1 + ∂F/∂x). The “1” guarantees a minimum gradient of ∂L/∂(x+F(x)) regardless of ∂F/∂x. This single term is why GPT-3 (96 layers) and LLaMA-3 (80 layers) are trainable without specialised gradient clipping or layer-by-layer learning rate schedules.
residual.py
# Pre-LN residual pattern -- appears twice per transformer block in GPT-2

def block_forward(x, attn, ffn, ln1, ln2):
    x = x + attn(ln1(x))  # ↑ residual 1: bypass x has gradient 1.0 -- always flows unchanged
    x = x + ffn(ln2(x))   # ↑ residual 2: same gradient highway guarantee for the FFN path
    return x               # ↑ residual stream enriched with two additive corrections from this block

# Backprop gradient through residual 1:
#   dL/dx_in = dL/dx_out * (1 + dAttn/dx_in)
#   The "1 +" term is the bypass -- minimum gradient is always dL/dx_out.
#
# At initialisation (small weights):
#   attn(ln1(x)) ~= 0  and  ffn(ln2(x)) ~= 0
#   => full block stack starts as identity: output ~= input
# Each block learns to ADD a correction, not replace the signal from scratch.

The Transformer Block

One transformer block assembles LayerNorm, multi-head causal self-attention, FeedForward, and two residual connections into a single reusable unit. The block takes a tensor of shape (batch_size, seq_len, d_model) and returns a tensor of the same shape. This shape invariance is what makes blocks trivially stackable — block N+1 receives precisely the same typed input that block N received, with no glue layers between them.

The forward pass through one block follows a strict order: (1) LayerNorm1 normalises each token’s 768-dim vector; (2) multi-head causal self-attention mixes information across positions — the only place in the block where tokens communicate; (3) attention output is added back to the pre-norm residual (skip 1); (4) LayerNorm2 normalises the updated residual stream; (5) FeedForward processes each position independently with 4× expansion; (6) FFN output is added back (skip 2). The block is complete.

For GPT-2 small, each block has 7,087,872 parameters: two LayerNorms (3,072), QKV projection (1,771,776), output projection (590,592), FFN (4,722,432). Twelve blocks account for 85 million of GPT-2’s 124 million total parameters. Blocks are architecturally identical but have independently learned weights. Early blocks tend to learn syntactic patterns (part-of-speech, named entity boundaries); middle blocks learn semantic associations; late blocks specialise in task-specific next-token distributions. This functional hierarchy emerges from training, not from any architectural difference between blocks.

Division of labour: Attention is the only place information crosses position boundaries — token at position i can read from 0…i-1. FFN processes each position in isolation. Alternating attention→FFN layers create a computation pattern: mix information globally (attention), then process each position’s resulting representation locally (FFN). This interleaving produces a representation hierarchy that grows richer with depth.

class TransformerBlock:
    def __init__(self, emb_dim, n_heads, drop_rate=0.1):
        self.ln1  = LayerNorm(emb_dim)                       # ↑ pre-norm 1: applied before attention
        self.attn = MultiHeadCausalAttention(emb_dim, n_heads)  # ↑ 12 heads x 64 dim, causal mask
        self.ln2  = LayerNorm(emb_dim)                       # ↑ pre-norm 2: applied before FFN
        self.ff   = FeedForward(emb_dim)                     # ↑ 4x expansion, GELU, per-token
        self.drop = drop_rate                                # ↑ dropout 0.1 for regularisation

    def forward(self, x):
        x = x + dropout(self.attn(self.ln1(x)), self.drop)
        # ↑ residual 1: x passes through unchanged if attn learns nothing useful
        x = x + dropout(self.ff(self.ln2(x)), self.drop)
        # ↑ residual 2: x passes through unchanged if FFN learns nothing useful
        return x  # ↑ same shape (B, T, d_model) -- ready for the next block

Stacking Blocks

The GPT architecture stacks N identical transformer blocks sequentially. GPT-2 small uses 12, medium 24, large 36, XL 48. The residual stream — a tensor of shape (batch, seq_len, d_model) — flows through each block in order. Each block reads the current stream, computes two additive updates (from attention and FFN), and writes them back via the residual additions. The next block receives the enriched stream as its input.

The blocks are architecturally identical but they are not identical in function. Each block has its own independently initialised and trained weight matrices. Empirically, early blocks (layers 1–4) learn syntactic patterns: part-of-speech transitions, named entity boundaries, punctuation structure. Middle blocks (layers 5–8) encode semantic associations, coreference, and world knowledge. Late blocks (layers 9–12) specialise in distributional next-token patterns for the specific training domain. This hierarchy emerges from training; it is not hardcoded. Levine et al. 2020 showed that for a fixed parameter budget, deeper models (more blocks) generally outperform wider models (larger d_model), making depth a preferred scaling axis.

In PyTorch, stacking blocks uses nn.ModuleList rather than a Python list. The distinction is critical: nn.ModuleList registers every contained module as a named submodule of the parent, causing their parameters to appear in model.parameters() and be tracked by the optimiser. A plain Python list of modules is invisible to PyTorch’s parameter tracking — none of those parameters would be updated during training. Gradient checkpointing (Chen et al. 2016) is used during training of large models to trade compute for memory: activations are discarded during the forward pass and recomputed during backprop, saving ~60% of activation memory at the cost of ~33% more compute.

Depth scaling law: For the same total parameter budget, deeper models (more blocks) consistently outperform wider models (larger d_model). GPT-3’s 96 layers and LLaMA-2’s 80 layers exploit this. The Chinchilla scaling laws (Hoffmann et al. 2022) quantify the optimal depth/width ratio for a given compute budget, but depth is almost always favoured over raw width increase.

The Full GPT Model

Assembling the complete model is almost anticlimactic after the component work — the pieces click together without surprises. The forward pass is: token embedding lookup + position embedding lookup → add + dropout → N transformer blocks → final LayerNorm → LM head linear projection to vocabulary size. Input is a sequence of integer token IDs; output is logits over the 50,257-token vocabulary for every position in the sequence.

Token embeddings convert integer token IDs into d_model-dimensional vectors via a simple lookup table — a (vocab_size × d_model) weight matrix where each row corresponds to one token. This is exactly equivalent to one-hot encoding multiplied by the weight matrix, but stored as a direct table for efficiency. Position embeddings are identical in structure: a (context_len × d_model) matrix where each row encodes one position index. GPT-2 uses learned position embeddings, not the sinusoidal ones from the original Transformer. Both embedding tables are added elementwise to form the initial residual stream; this is the only point in the entire model where position information enters.

The final LayerNorm appears after all transformer blocks but before the LM head. This “output norm” is present in GPT-2 (Pre-LN models add it explicitly) but absent from the original Post-LN Transformer. Without it, the raw residual stream variance after 12 blocks can be large, making the pre-softmax logits numerically unstable. The LM head is a Linear(d_model, vocab_size) with no bias; in GPT-2 its weight matrix is tied to the token embedding matrix (see §4.8), adding zero additional parameters.

Weight Tying

GPT-2’s LM head — the final Linear(768, 50257) projection — shares its weight matrix with the token embedding table. In PyTorch: self.lm_head.weight = self.tok_emb.weight. This single assignment eliminates 50,257 × 768 = 38,597,376 parameters from the model — nearly 147 MB at float32. It is not merely a memory optimisation; it improves model quality by enforcing a geometric consistency that two separate matrices would need to learn independently.

The intuition: the score for predicting token t at some position is computed as h · W_lm[t], where h is the position’s residual stream after the final LayerNorm. If W_lm is tied to the token embedding W_emb, then W_lm[t] = W_emb[t] — the embedding vector for token t. The score is therefore h · W_emb[t]: the dot product between the current representation and token t’s embedding. Tokens whose embeddings are geometrically close to h score highest. This is exactly the right inductive bias: “predict the tokens whose learned representation is most similar to the current state.” Without tying, the LM head could learn a completely different geometry from the embeddings, which would be redundant and harmful.

Press and Wolf introduced weight tying in 2017 (arXiv 1608.05859) and measured a ~1.4 perplexity improvement on Penn Treebank. Since then it has become standard in virtually every language model: BERT, GPT-2, GPT-3, OPT, and Falcon all use it. LLaMA notably does not — it uses a separate unshared lm_head, sacrificing the parameter saving in exchange for more expressive freedom in the output projection. Gradients flow into the tied matrix from both directions during training: the embedding receives signal from the embedding loss path and the LM head loss path simultaneously, providing more diverse gradient signal than an untied matrix would receive.

Counting Parameters

GPT-2 “small” has 124 million parameters. That number is cited everywhere but rarely fully explained. Here is the exact breakdown. Understanding it reveals where the model’s capacity is actually located, why the FFN dominates, and why scaling depth is more efficient than scaling vocabulary size.

The largest single component is the token embedding matrix: 50,257 × 768 = 38,597,376 parameters, or 31% of the model. The position embedding adds 1,024 × 768 = 786,432. Then come 12 transformer blocks: each block contains approximately 7,087,872 parameters (see §4.5), giving 12 × 7,087,872 = 85,054,464 total. The final LayerNorm contributes 1,536. The LM head is weight-tied: zero additional parameters. Sum: 38,597,376 + 786,432 + 85,054,464 + 1,536 = 124,439,808 — approximately 124M.

The dominant component by far is the FFN layers: 12 × 4,722,432 = 56,669,184 — 45.6% of the model. This is why the Geva et al. “key-value memory” interpretation (arXiv 2012.14913) is plausible: there is enormous parameter capacity in those matrices to store factual associations. The attention mechanism, despite its conceptual centrality, holds only 12 × (1,771,776+590,592) = 28,348,416 — 22.8% of the model. Per transformer block: attention is 4d² parameters and FFN is 8d² parameters (plus biases), so FFN always holds exactly twice as many parameters as attention at any scale.

Generating Text

Text generation is the forward pass used as a loop. Given an initial sequence of token IDs (the prompt), run a forward pass to get logits for the next token at the last position, select or sample one token, append it to the sequence, and repeat. Each iteration extends the context by one token. This autoregressive loop underlies every production language model application: chatbots, code completion, creative writing, and instruction following.

The simplest strategy is greedy decoding: argmax over the logits at each step. This is deterministic and fast but produces repetitive, low-diversity text — the highest-probability continuation tends to keep selecting the same high-frequency phrases. Temperature divides all logits by a scalar T before softmax: T<1 sharpens the distribution (higher-ranked tokens get relatively more probability), T>1 flattens it (more random). T→0 recovers greedy decoding. Top-k restricts sampling to only the k highest-probability tokens (GPT-2’s default k=50), preventing very unlikely tokens from occasionally being selected. Top-p (nucleus sampling), introduced by Holtzman et al. 2020 (arXiv 1904.09751), keeps tokens until cumulative probability reaches p (typically 0.9), dynamically adjusting vocabulary size per step based on how peaked the distribution is. Min-p (2023) is an alternative: compute p_min = p_max × min_p_ratio and remove tokens below that threshold; it outperforms top-p at low temperatures.

A crucial engineering optimisation is the KV cache: during generation, the key and value tensors for all previous positions are cached and reused rather than recomputed. Without the cache, each generation step requires a full forward pass over all T tokens (O(T²) attention per step, O(T³) total for T tokens). With the cache, each step only computes attention for the new token against cached K,V (O(T) per step, O(T²) total). At GPT-2’s 1,024 context, the KV cache provides ~500× speedup by step 512.

Attention for the new token against cached K,V is O(T) rather than O(T²). Generation goes from O(T³) total FLOPs to O(T²). At context length 1024, this is a 512× speedup by the last step. Every production inference system uses KV caching.

import math, random

def generate(model, token_ids, max_new=50, temperature=1.0, top_k=50):

    for _ in range(max_new):
        ctx = token_ids[-model.context_len:]            # ↑ truncate to context window (1024 for GPT-2)
        logits = model.forward(ctx)[0][-1]             # ↑ last position logits: shape (50257,)

        logits = [l / temperature for l in logits]      # ↑ temperature: T<1 sharpens, T>1 flattens

        if top_k is not None:
            idx = sorted(range(len(logits)), key=lambda i:-logits[i])
            threshold = logits[idx[top_k - 1]]
            logits = [l if l >= threshold else -float("inf") for l in logits]
            # ↑ top-k: keep only top k tokens; -inf -> probability 0 after softmax

        m = max(logits)
        exps = [math.exp(l - m) for l in logits]
        s = sum(exps)
        probs = [e / s for e in exps]                  # ↑ softmax: sum-to-1 probability distribution

        r, acc = random.random(), 0.0
        next_token = 0
        for i, p in enumerate(probs):
            acc += p
            if r <= acc: next_token = i; break         # ↑ inverse CDF sampling

        token_ids.append(next_token)                     # ↑ extend context with sampled token
        if next_token == model.eos_id: break           # ↑ stop at <|endoftext|> token (ID 50256)

    return token_ids

The Residual Stream View

All the components described so far can be unified under a single conceptual frame: the residual stream. Every transformer block reads from a shared tensor of shape (B, T, d_model), computes a correction, and writes it back additively. Nothing is ever replaced; everything is accumulated. This view, formalised by Elhage et al. in 2021 (transformer-circuits.pub), transforms an otherwise opaque stack of operations into an interpretable system of simultaneous read/write operations on a common information bus.

The residual stream starts as the sum of token embeddings and position embeddings. At every subsequent operation — each attention sublayer, each FFN sublayer — a delta is computed from the current stream and added back. By the time the stream reaches the final LayerNorm after 12 blocks, it carries the cumulative result of all 24 read/write operations (2 per block × 12 blocks). The LM head then reads from this final stream to produce vocabulary logits. Crucially, the stream is never consumed: every layer reads its full content and every layer writes back to the full content. Earlier layers’ contributions persist throughout.

This framing has deep implications. It means each attention head and each FFN layer is best understood not as transforming a representation, but as adding a specific component to the stream. Mechanistic interpretability research (Anthropic, DeepMind) decomposes models by identifying which components contribute which types of information to the stream at each layer. The residual stream view also explains why residual connections cannot be removed: they are not a gradient trick alone — they are the architecture. The stream is the shared state that all layers communicate through.

def gpt_as_residual_stream(token_ids, tok_emb, pos_emb, blocks, final_ln, lm_head):

    return logits

Attention Head Patterns

GPT-2 small has 12 attention heads per block, 12 blocks deep — 144 attention heads total. Each head independently computes its own Q, K, V projections from the residual stream, attends across positions, and writes a contribution back to the stream. The 144 heads do not all do the same thing. Research by Clark et al. (2019) and Olsson et al. (2022) identified that individual heads specialise into distinct, interpretable functions that can be characterised by their attention patterns.

The most well-studied specialisation is the induction head (Olsson et al. 2022, arXiv 2209.11895). An induction head implements in-context copying: if the sequence contains the pattern [A][B]…[A], the head attends to the position after the first [A] when processing the second [A], predicting [B] as the next token. This mechanism is responsible for most of GPT-2’s ability to repeat patterns seen earlier in the context. Induction heads emerge reliably in all transformers above 2 layers and are thought to be one of the fundamental computational primitives of transformer circuits.

Other documented head types include: previous-token heads (attend to position i-1, copy recent context forward), duplicate-token heads (suppress repeated tokens), positional heads (attend to fixed relative offsets regardless of content), and syntactic heads (track subject-verb agreement or noun-modifier dependencies). Not all heads are interpretable — many attend diffusely or serve unclear functions. The distribution of head types varies between layers: early layers tend to have more positional and syntactic heads; late layers have more semantic and task-specific heads.

Induction Heads

Pattern: [A][B]...[A] → attends back to first [A]’s successor to predict [B]. Enables in-context pattern copying. Emerges in all transformers above 2 layers (Olsson et al. 2022).

Previous-Token Heads

Always attends to position i-1. Effectively copies the token one step back into the current position’s representation. Provides short-range sequential context to later heads.

Duplicate-Token Heads

Detects and signals repeated tokens. Used by downstream components to suppress or penalise repetitions. Helps explain why transformers with nucleus sampling produce less repetitive text than greedy decoding.

Syntactic Heads

Track grammatical dependencies: subject-verb agreement, noun-adjective relationships, coreference. Tend to appear in early-to-middle layers. Documented in BERT probing studies (Clark et al. 2019).

Heads are composable: Elhage et al. 2021 showed that a 2-layer transformer’s behaviour can be decomposed into a sum of three terms: the direct path (embedding straight to LM head), single-layer paths (each head writes to LM head directly), and two-hop paths (head in layer 1 composes with head in layer 2). This composability extends to deeper models and is the foundation of circuit-level analysis.

import math

def causal_attention_pattern(Q, K, head_dim, seq_len):

    # Q, K: (seq_len, head_dim) -- query and key matrices for one head
    scale = math.sqrt(head_dim)           # ↑ scale by sqrt(d_k) to control dot product magnitude
    attn = []
    for i in range(seq_len):

        row = []
        for j in range(seq_len):

            if j > i:   # causal mask: future positions are -inf
                row.append(-float("inf"))  # ↑ -inf -> probability 0 after softmax: token i cannot see j>i
            else:
                qk = sum(Q[i][k]*K[j][k] for k in range(head_dim)) / scale
                row.append(qk)             # ↑ scaled dot product: how much does position i attend to j?
        # softmax over unmasked positions
        m = max(r for r in row if r != -float("inf"))
        exp_row = [math.exp(r - m) if r != -float("inf") else 0.0 for r in row]
        s = sum(exp_row)
        attn.append([e/s for e in exp_row])  # ↑ softmax: attention weights sum to 1 over unmasked positions
    return attn  # (seq_len, seq_len) lower-triangular attention pattern

GPT-2 in Numbers: A Complete Parameter Audit

This section provides the definitive, exhaustive parameter count for every size in the GPT-2 family. The formula for total parameters is: (vocab × d) + (ctx × d) + n_layers × [2 × (2d) + 3d² + 3d + d² + d + 2 × (2d) + d × 4d + 4d + 4d × d + d] + 2d. The weight-tied LM head contributes 0 additional parameters in all GPT-2 variants.

Per transformer block the exact parameter count is: two LayerNorms (4d), QKV projection (3d² + 3d), output projection (d² + d), FFN W1 (4d² + 4d), FFN W2 (4d² + d). Summing: 12d² + 13d per block. For d=768: 12×768² + 13×768 = 7,077,888 + 9,984 = 7,087,872. For d=1024: 12×1024² + 13×1024 = 12,582,912 + 13,312 = 12,596,224. These are the numbers that determine how GPT-2 family models fit into VRAM: float16 uses 2 bytes per parameter; activations during forward pass add approximately batch_size × seq_len × d × n_layers × 4 bytes.

To run GPT-2 small at float32: 124M × 4 bytes = 496MB weights, plus activations. At float16: 248MB. At int8: 124MB. GPT-2 XL at float16: 1.5B × 2 = 3GB, comfortably within a single modern GPU. LLaMA-2 7B at float16: 7B × 2 = 14GB, requiring one A100-40GB. LLaMA-2 70B at int4 (4 bits per weight): 70B × 0.5 = 35GB — fits on two A100-40GB or one A100-80GB. def exact_params(n_layers, d, n_heads, vocab=50257, ctx=1024, tied=True):

    dff = 4 * d
    tok_emb  = vocab * d             # ↑ token embedding table: vocab_size x d_model

    ln_final = 2 * d                 # ↑ final LayerNorm: scale + shift

    per_block = (
        2 * 2 * d +          # ↑ 2x LayerNorm (scale+shift each): 4d

        d*d + d +             # ↑ output projection weight + bias: d^2 + d
