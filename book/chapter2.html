2.1

The Tokenization Problem

Neural networks operate on vectors of real numbers. Text is a sequence of Unicode characters. Bridging that gap is tokenization — the choice of how to divide text into discrete units that can each be assigned a number. This decision is not cosmetic. It determines the effective context length, the model's ability to handle rare words, the difficulty of learning morphology, and even the security of the deployed system.

Three strategies have been tried seriously. Character-level tokenization uses a fixed alphabet of ~256 byte values. Any text can be represented, OOV (out-of-vocabulary) is impossible, but sequences become 4× longer than word-level — making it much harder for attention mechanisms to connect distant ideas. Word-level tokenization splits on whitespace and punctuation, using a fixed vocabulary of ~170,000 English words. Compact and interpretable, but it fails on unseen compounds, multilingual text, code, emoji, and morphologically rich languages (Finnish can form millions of distinct word-forms from a handful of roots). Subword tokenization is the practical sweet spot: common words stay whole, rare words decompose into recognizable pieces, and the vocabulary stays fixed and manageable. Every modern LLM uses a subword approach.

The same phrase tokenized three ways shows the stakes. "unbelievably" becomes 12 character tokens, 1 word token (which fails on unseen compounds like "unbelievably-fast"), or 3 subword tokens: ["un", "believ", "ably"] — pieces that transfer their meaning to hundreds of other words the model has seen. The concrete tradeoff: GPT-2's tokenizer encodes English prose at approximately 1.3 tokens per word. Chinese prose averages 1.5 tokens per character. Python code averages 0.8 tokens per character. The model never sees words — only tokens.

Key insight: Tokenization is irreversible. A model trained with one tokenizer cannot use another without full retraining. This is why the tokenizer is considered part of the model — it ships with it, is versioned with it, and its design choices echo through every downstream capability.

import re

def char_tokens(text):
    return list(text.encode('utf-8'))       # byte-level character tokenization

def word_tokens(text):
    return re.findall(r"[a-zA-Z0-9']+|[^\w\s]", text)  # simple word split

def bpe_approx(text):
    # approximate subword split using common English affixes
    affixes = ["un","re","pre","dis","ing","tion","ness","ment","ful","less","ably","ed","er","est","ly"]
    tokens = []
    for word in text.split():
        split = False
        for af in affixes:
            if word.lower().startswith(af) and len(word) > len(af) + 2:
                tokens += [word[:len(af)], word[len(af):]]
                split = True; break
            if word.lower().endswith(af) and len(word) > len(af) + 2:
                tokens += [word[:-len(af)], word[-len(af):]]
                split = True; break
        if not split:
            tokens.append(word)
    return tokens

]

2.2

Byte Pair Encoding

BPE was invented in 1994 as a data compression algorithm by Philip Gage. The observation: the most redundant information in a file is adjacent byte pairs that repeat — compress them, and file size shrinks. Sennrich, Haddow, and Birch (2016) applied this idea to tokenization for neural machine translation and transformed the field. Every major LLM released since 2019 uses some variant of BPE.

The training algorithm is elegant in its simplicity. Start with a corpus where every word is pre-split into individual characters (with a special end-of-word marker, </w>, to preserve word boundaries). Count every adjacent pair across the entire corpus. Find the most frequent pair. Merge every occurrence of that pair into a single new token. Repeat for k merge operations. The result is an ordered list of k merge rules plus the final vocabulary.

GPT-2 made one critical refinement: byte-level BPE. Instead of starting with Unicode characters (some of which may never appear in the training corpus), start with all 256 raw byte values. This guarantees the tokenizer can encode any byte sequence — including emoji, arbitrary Unicode, binary data, and broken encodings — without ever needing an unknown-token fallback. GPT-2's vocabulary: 256 base bytes + 50,000 merges + 1 special token (<|endoftext|>) = 50,257 tokens.

Why byte-level? Unicode has 1.1 million code points. A corpus-level character vocabulary omits any character not seen in training — fail on Japanese if trained only on English. Byte-level BPE starts from the 256 byte alphabet (which covers every possible Unicode character when UTF-8 encoded) and merges upward. Zero OOV problems by construction.

▶ Cell 2 — Trace BPE merge steps on a toy corpus 
from collections import Counter

def get_stats(vocab):
    """Count every adjacent pair across all word entries."""
    pairs = Counter()
    for word, freq in vocab.items():
        syms = word.split()
        for i in range(len(syms) - 1):
            pairs[(syms[i], syms[i+1])] += freq
    return pairs

def merge_vocab(pair, vocab):
    """Replace every occurrence of the pair with a merged token."""
    src = ' '.join(pair)         # 'e s' as the string key to replace
    dst = ''.join(pair)          # 'es' as the merged token
    return {w.replace(src, dst): f for w, f in vocab.items()}

# Classic Sennrich 2016 example corpus (word → frequency)
vocab = {
    'l o w ': 5,
    'l o w e r ': 2,
    'n e w e s t ': 6,
    'w i d e s t ': 3,
}

print("Initial vocabulary (word: frequency):")
for w, f in vocab.items():
    print(f"  {f}×  {w}")

NUM_MERGES = 10
print(f"\nApplying {NUM_MERGES} BPE merges:\n")

all_tokens = set()
for i in range(NUM_MERGES):
    pairs = get_stats(vocab)
    if not pairs: break
    best = max(pairs, key=pairs.get)
    vocab = merge_vocab(best, vocab)
    merged = ''.join(best)
    all_tokens.add(merged)
    print(f"  Merge {i+1:2d}: ({best[0]!r} + {best[1]!r}) → {merged!r:12}  count={pairs[best]}")

print("\nFinal token sequences:")
for w, f in vocab.items():
    print(f"  {f}×  [{', '.join(repr(t) for t in w.split())}]")

# collect all learned tokens
all_toks = set()
for w in vocab:
    all_toks.update(w.split())
print(f"\nVocabulary: {sorted(all_toks)}")
print(f"Size grew from 8 chars → {len(all_toks)} tokens after {NUM_MERGES} merges.")
print("Try: change NUM_MERGES or the vocab frequencies.")

2.3

Building BPE from Scratch

The toy example above is the core algorithm. A production BPE tokenizer adds two things: a regex pre-tokenizer that splits text into word-like units before BPE runs (so the tokenizer never merges characters across word or punctuation boundaries), and a fast encoder that applies the saved merge list to any new string at inference time.

The pre-tokenization regex is often overlooked but critically important. GPT-2 uses a regex that correctly handles English contractions ("don't" → ["don", "'t"]), attaches a leading space to most words (so "cat" and " cat" can become different tokens), and separates numbers and punctuation into their own tokens. Without pre-tokenization, BPE might merge letters across a space — turning "lower" and " older" into "lower older" and merging "r " as a token, which would be confusing and wasteful.

Here is the complete working byte-level BPE — the same logic Karpathy uses in minbpe. train_bpe takes a string and vocabulary size; encode and decode round-trip perfectly:

bpe.py 
from collections import Counter

def get_stats(ids):
    # count every consecutive pair in the flat token list
    return Counter(zip(ids, ids[1:]))
    # ↑ zip(ids, ids[1:]) pairs each token with its right neighbour
    #   e.g. [72, 101, 108] → {(72,101):1, (101,108):1}

def merge(ids, pair, new_id):
    # replace every occurrence of pair with the merged token new_id
    out, i = [], 0
    while i < len(ids):
        if (i + 1 < len(ids) and
                ids[i] == pair[0] and ids[i+1] == pair[1]):
            out.append(new_id)
            # ↑ found the pair: emit the merged token, skip both
            i += 2
        else:
            out.append(ids[i]); i += 1
    return out

def train_bpe(text, vocab_size=300):
    # encode text as raw bytes, then run BPE for (vocab_size - 256) merges
    ids = list(text.encode('utf-8'))
    # ↑ byte-level: every byte is a valid initial token (IDs 0–255)
    vocab = {i: bytes([i]) for i in range(256)}
    # ↑ initial vocab: byte 65 = b'A', byte 32 = b' ', etc.
    merges = {}   # (pair) → new_token_id, in training order

    for i in range(vocab_size - 256):
        stats = get_stats(ids)
        if not stats: break
        best = max(stats, key=stats.get)
        # ↑ greedy: always merge the most frequent pair first
        new_id = 256 + i
        # ↑ new tokens are assigned IDs starting at 256
        ids = merge(ids, best, new_id)
        merges[best] = new_id
        vocab[new_id] = vocab[best[0]] + vocab[best[1]]
        # ↑ new token's bytes = left bytes + right bytes concatenated

    return merges, vocab   # save these two objects; they ARE the tokenizer

def encode(text, merges):
    # tokenize new text by applying merge rules in training order
    ids = list(text.encode('utf-8'))
    for pair, new_id in merges.items():
        ids = merge(ids, pair, new_id)
        # ↑ MUST apply in same order as training — order matters!
    return ids

def decode(ids, vocab):
    # convert token IDs back to text via the vocab byte map
    raw = bytes().join(vocab[i] for i in ids)
    return raw.decode('utf-8', errors='replace')
The merge table is the tokenizer. Two objects are all you need: the ordered list of merges (applied in sequence during encoding) and the vocab map (ID → byte sequence, used during decoding). No neural network parameters. Encoding "hello world" is O(k·n) where k = number of merges and n = number of tokens.

2.4

tiktoken in Practice

tiktoken is the tokenizer OpenAI uses for all GPT-3.5, GPT-4, and Codex models. Built in Rust with a Python wrapper, it encodes at roughly 1 GB/s — 3–6× faster than HuggingFace tokenizers for the same vocabulary, fast enough that tokenization is never the bottleneck in a training or inference pipeline.

tiktoken ships with three main encoding schemes. r50k_base (also called "gpt2") has 50,257 tokens — the original GPT-2 vocabulary. p50k_base extends it to 50,281 tokens and adds a small set of additional code-related tokens; used for GPT-3 and Codex. cl100k_base is a completely retrained vocabulary with 100,277 tokens, designed for multilingual text and code simultaneously; used for GPT-3.5-turbo, GPT-4, and text-embedding-ada-002. The jump from 50k to 100k tokens means each token encodes roughly 1.7× more information — shorter sequences, lower cost, higher quality for non-English languages.

The API is minimal. enc.encode(text) returns a list of integer token IDs. enc.decode(ids) converts back to a string. enc.encode_ordinary(text) is the same but never emits special tokens — useful when you want to count tokens without triggering any special-token handling. The most useful utility function for practitioners: len(enc.encode(text)) to measure how many tokens a string costs before sending it to the API.

tiktoken_demo.py 
import tiktoken

enc = tiktoken.get_encoding("cl100k_base")
# ↑ load GPT-4's tokenizer (100,277 tokens, best for multilingual)

# Encoding: string → list of integer token IDs
ids = enc.encode("Hello, world! 🌍")
# ↑ returns something like [9906, 11, 1917, 0, 11410, 234, 228]
print(ids)

# Decoding: list of integer IDs → string
text = enc.decode(ids)
# ↑ perfect round-trip; decode(encode(x)) == x always
print(text)

# Count tokens (critical for API cost estimation)
print(len(enc.encode(prompt)))
# ↑ GPT-4 charges per token; always count before sending

# Inspect individual token bytes
for token_id in ids:
    token_bytes = enc.decode_single_token_bytes(token_id)
    # ↑ each token ID maps to a specific byte sequence
    print(f"{token_id:6d} → {token_bytes}")

# Count how a model's context window fills up
context_window = 128_000   # GPT-4o context length
# ↑ GPT-4o can process 128,000 tokens (~96,000 words) at once
tokens_used = len(enc.encode(document))
print(f"Used {tokens_used} / {context_window} tokens")
Token counting matters. OpenAI's API charges by token. A typical English sentence uses 10–20 tokens. A 4,000-word essay uses ~5,000 tokens. GPT-4o at $5/1M tokens means a million-word novel costs about $6 to process. Always count before sending; always cache results when possible.

2.5

Special Tokens

A tokenizer needs tokens that do not correspond to text — they carry structural meaning: "this is where a document ends", "this is where the user's message starts", "this position should be ignored". These are special tokens, and they are added to the vocabulary after BPE training so they are always single tokens, never split.

GPT-2 uses exactly one special token: <|endoftext|> (ID 50256). During training, every document in the corpus is concatenated to the next with this token in between. This teaches the model that when it sees <|endoftext|>, the current "story" has ended — a fresh context begins. At generation time, placing this token in the prompt creates a clean slate. Because BPE can encode any byte sequence, GPT-2 technically doesn't need an unknown-token — but it does need a document boundary.

Chat models need more structure. The ChatML format (used by GPT-3.5/4) introduces <|im_start|> and <|im_end|> to delimit turns. LLaMA-3 uses <|begin_of_text|>, <|start_header_id|>, <|end_header_id|>, and <|eot_id|>. BERT-style encoder models use [CLS] (a classification token whose embedding is used as the sequence representation), [SEP] (separator between sentence pairs), [MASK] (replaced tokens during masked language modeling), and [PAD] (padding to equal length).

<|endoftext|> — ID 50256

GPT-2/GPT-3: document boundary. Separates training documents. At inference: "start fresh" signal. BPE makes it impossible to accidentally generate from its components.

[CLS] / [SEP] / [MASK]

BERT-family encoders. [CLS] at position 0 — its output vector represents the full sequence for classification. [SEP] separates premise from hypothesis in NLI tasks. [MASK] for MLM training.

<|im_start|> / <|im_end|>

ChatML format (GPT-3.5/4). Each conversation turn is wrapped: <|im_start|>user\nHello<|im_end|>. The model is trained to predict <|im_end|> as its stopping condition.

<|pad|> / [PAD]

Padding token: fills shorter sequences in a batch to equal length. Masked out in the loss function — model is not penalized for what it "predicts" at padding positions. GPT-2 uses EOT as pad.

Prompt injection via special tokens. If a user can inject <|im_end|> into their input, they can trick the model into "ending" the system prompt early and continuing as if a new role started. Always encode user input as ordinary text (never with special tokens enabled in user-controlled fields). tiktoken's encode_ordinary() strips all special-token handling.

ENDOFTEXT = "<|endoftext|>"   # token ID 50256 in GPT-2

documents = [
    "The quick brown fox jumps over the lazy dog.",
    "A second document about machine learning and neural networks.",
    "Third document: tokenization is the first step in NLP.",
    "Final document: transformers changed the field of NLP completely.",
]

# Step 1: concatenate all documents with the boundary token
training_stream = ENDOFTEXT.join(documents) + ENDOFTEXT
print("Training stream (first 120 chars):")
print(f"  {training_stream[:120]!r}...")
print(f"  Total chars: {len(training_stream)}")

# Step 2: naive word tokenizer (real GPT-2 uses tiktoken)
def tokenize(text):
    import re
    tokens = []
    for part in re.split(r'(<\|endoftext\|>)', text):
        if part == ENDOFTEXT:
            tokens.append(ENDOFTEXT)
        elif part:
            tokens.extend(re.findall(r'\S+|\s+', part))
    return [t for t in tokens if t.strip() or t == ENDOFTEXT]

tokens = tokenize(training_stream)
print(f"\nTokenized stream ({len(tokens)} tokens):")
for i, tok in enumerate(tokens):
    marker = " ← boundary!" if tok == ENDOFTEXT else ""
    print(f"  {i:3d}: {tok!r}{marker}")
    if i > 22:
        print(f"  ... ({len(tokens) - i - 1} more tokens)")
        break

# Step 3: sliding window creates (input, target) pairs
CONTEXT_LEN = 10

for start in range(0, min(30, len(tokens) - CONTEXT_LEN), CONTEXT_LEN):
    x = tokens[start : start + CONTEXT_LEN]

SentencePiece and Alternatives

BPE is not the only subword algorithm. Three others matter in practice: WordPiece, Unigram Language Model, and SentencePiece (which is a framework that can run either BPE or Unigram).

WordPiece (Schuster & Nakamura 2012, popularized by BERT) starts like BPE — character-level vocabulary — but chooses merges differently. Instead of merging the pair that appears most frequently, it merges the pair that most increases the likelihood of the training corpus under a unigram language model. In practice this produces similar but not identical vocabularies to BPE. The key surface difference: WordPiece marks sub-word continuations with a ## prefix, so "playing" might tokenize as ["play", "##ing"]. BERT, RoBERTa, DistilBERT, and mBERT all use WordPiece.

SentencePiece (Kudo & Richardson 2018) is a language-agnostic tokenization library that treats the raw Unicode string as input directly — no pre-whitespace-splitting step. This makes it suitable for Japanese, Chinese, Thai, and other languages that don't delimit words with spaces. It implements both BPE and Unigram objectives. LLaMA 1/2 (and their derivatives: Mistral, Vicuna, WizardLM) use SentencePiece with BPE. T5, mT5, and all Google models use SentencePiece with Unigram. Gemini and PaLM also use SentencePiece.

The Unigram LM algorithm (Kudo 2018) takes a different approach entirely: start with a large initial vocabulary, then iteratively remove tokens that decrease corpus likelihood the least. It converges to a smaller vocabulary by pruning, not merging. Unigram vocabularies are probabilistic — the same string can tokenize differently at inference (it samples the most likely segmentation), which adds some robustness to rare inputs.

If you are fine-tuning LLaMA or Mistral, use their SentencePiece model (do not retrain the tokenizer — your fine-tuned weights won't work with a different vocabulary). Only retrain a tokenizer if you are pretraining a model from scratch on a domain with highly unusual vocabulary (code, chemistry, genomics).

import math
from collections import Counter

def get_stats(vocab):
    pairs = Counter()
    for word, freq in vocab.items():
        syms = word.split()
        for i in range(len(syms) - 1):
            pairs[(syms[i], syms[i+1])] += freq
    return pairs

def count_tokens(vocab):
    """Count individual token frequencies for unigram probability."""
    counts = Counter()
    for word, freq in vocab.items():
        for tok in word.split():
            counts[tok] += freq
    return counts

def wordpiece_score(pair, pair_count, token_counts):
    """WordPiece score: pair_freq / (freq_A * freq_B) — prefers rare component pairs."""
    a, b = pair
    if token_counts[a] == 0 or token_counts[b] == 0:
        return 0
    return pair_count / (token_counts[a] * token_counts[b])

def merge_vocab(pair, vocab):
    src = ' '.join(pair)
    dst = ''.join(pair)
    return {w.replace(src, dst): f for w, f in vocab.items()}

corpus = {

From Raw Text to Clean Corpus

The quality of training data is the single most important factor in the quality of the resulting model. "Garbage in, garbage out" is not a metaphor — Llama 3 8B outperforms Llama 2 70B on many benchmarks despite being 8× smaller, primarily because it trained on 15T tokens of aggressively curated data versus 2T tokens of less-filtered data. The data pipeline deserves as much engineering attention as the model architecture.

The raw source for most large models is Common Crawl — a monthly snapshot of the web containing roughly 3.4 billion web pages, 80TB of uncompressed HTML. Common Crawl is free, enormous, and nearly unusable without cleaning: it contains spam, duplicate content, machine-generated text, personal information, pornography, and content in every language ever written. The filtering pipeline that converts raw CC into a usable corpus is where most of the IP lives.

A standard cleaning pipeline runs these steps in order: (1) Language detection — fastText's language classifier on the first 200 characters; discard non-target-language documents. (2) Rule-based quality filters — minimum word count (GPT-2 requires ≥50 words after filtering), max fraction of lines ending in ellipsis (spam signals), minimum stop-word coverage (random character strings fail), max symbol-to-word ratio. (3) Deduplication — MinHash + LSH to find near-duplicate documents (~3-4% of CC is near-duplicate; deduplication improves perplexity by ~10%). (4) Toxicity and PII filtering — classifier-based removal of hate speech, explicit content; regex-based removal of emails, phone numbers, SSNs. (5) Quality scoring — some pipelines train a fastText classifier to distinguish "Wikipedia-quality" from "low-quality" text using those as positive/negative examples.

def quality_score(text):
    words = text.lower().split()
    if len(words) < 10:                               # too short
        return 0.0, "too short"

    # symbol ratio check
    symbols = sum(1 for c in text if not c.isalnum() and c not in " .,!?;:'\"-")
    if symbols / len(text) > 0.15:                    # too many symbols
        return 0.0, "high symbol ratio"

    # stop word coverage (real text has ~25-35% stop words)
    sw_count = sum(1 for w in words if w in STOP_WORDS)
    sw_ratio = sw_count / len(words)
    if sw_ratio < 0.05:                               # too few stop words
        return 0.0, "low stop word coverage"

    # repetition: most common word shouldn't dominate
    most_common_freq = Counter(words).most_common(1)[0][1] / len(words)
    if most_common_freq > 0.3:
        return 0.0, "high repetition"

    # exclamation marks (spam signal)
    excl_ratio = text.count('!') / max(len(words), 1)
    if excl_ratio > 0.1:
        return 0.0, "spam signals"

    # base score: stop word ratio (proxy for "natural language-ness")
    score = min(1.0, sw_ratio * 3.0) * (1 - most_common_freq)
    return round(score, 3), "pass"

Sliding-Window Sampling

Once we have a clean corpus tokenized into a flat sequence of integers, we need to produce training examples. The core insight: language modeling is next-token prediction. Given the tokens so far, predict the next one. Every position in every sequence is simultaneously an input and a label — just offset by one.

The mechanics: take your full flat token stream (all documents concatenated with <|endoftext|> separators). Extract a window of length T+1. The first T tokens are the input x; the last T tokens (shifted by one) are the target y. Advance the window by stride positions and repeat. This is called sliding window sampling.

The choice of stride controls data efficiency and training cost. Stride = T (non-overlapping, GPT-2's approach): simple, no repeated tokens, but some cross-document relationships are never seen. Stride < T (overlapping): each token appears in multiple windows with different context prefixes — more training signal per token, at the cost of more passes over the data. For a corpus of N tokens, stride=T gives ⌊N/T⌋ samples; stride=1 gives N−T samples (much more, but each is only slightly different from the last). GPT-2 trained on 40GB of WebText with T=1024 and stride=1024, yielding roughly 40 million training samples.

def sliding_window_samples(tokens, context_len, stride):
    """
    Generate (x, y) pairs from a flat token stream.
    x = input sequence of length context_len
    y = target sequence = x shifted right by 1
    """
    samples = []
    n = len(tokens)
    for start in range(0, n - context_len, stride):
        x = tokens[start : start + context_len]
        y = tokens[start + 1 : start + context_len + 1]
        samples.append((x, y))
    return samples

The DataLoader Pattern

Individual samples are not fed to the model one at a time — they are batched. A batch of B sequences each of length T creates an input tensor of shape (B, T): B rows of T token IDs. The model processes all B sequences simultaneously, producing B × T next-token predictions in a single forward pass. For GPT-2's training configuration: B=512, T=1024 → 524,288 predictions per batch.

The DataLoader pattern (formalized in PyTorch) solves two problems: (1) it shuffles the dataset so every epoch sees samples in a different order, preventing the model from memorizing the presentation sequence; (2) it prefetches the next batch on the CPU while the GPU is processing the current one, keeping GPU utilization close to 100%. For a trillion-token corpus, the dataset doesn't fit in RAM — the DataLoader streams from disk, often from pre-tokenized binary files (GPT-2 stored its data in a flat binary file of uint16 token IDs, 2 bytes each).

Memory analysis is important for understanding why batches are sized as they are. The input batch (B, T) of int64 IDs uses 512 × 1024 × 8 = 4MB — trivial. After the embedding lookup, the tensor becomes (B, T, d_model) = (512, 1024, 768) of float32 → 1.6GB per batch, just for the embedding output. Activations from all transformer layers, optimizer states, and gradients can push GPU memory requirements to 10–20GB per batch for a 124M-parameter model. This is why batch size is tuned per hardware configuration.

import torch
from torch.utils.data import Dataset, DataLoader
import numpy as np

class TokenDataset(Dataset):
    def __init__(self, data_path, context_len):
        self.tokens = np.memmap(data_path, dtype='uint16', mode='r')
        # ↑ memory-map: file stays on disk, OS pages it in on demand
        #   2 bytes per token × 9B tokens = 18GB file, but uses no RAM upfront
        self.context_len = context_len

    def __len__(self):
        return len(self.tokens) - self.context_len
        # ↑ every position except the last is a valid start

    def __getitem__(self, idx):
        chunk = self.tokens[idx : idx + self.context_len + 1]
        # ↑ grab T+1 tokens so we have both x (T) and y (T, shifted)
        chunk = torch.tensor(chunk.astype(np.int64), dtype=torch.long)
        x = chunk[:-1]    # input tokens: positions 0..T-1
        y = chunk[1:]     # targets: positions 1..T (each x[i] predicts y[i])
        # ↑ y is x shifted one step to the right
        return x, y

    pin_memory=True,        # ↑ faster CPU→GPU transfer via pinned memory

def batch_generator(tokens, context_len, batch_size, shuffle=True):
    """Yield (x, y) batches like PyTorch's DataLoader."""
    n = len(tokens)

Token Embeddings

A token ID is just an integer. The model can't do anything meaningful with an integer — it needs a vector of real numbers it can run matrix multiplications through. The token embedding table is the translation layer: a matrix of shape [vocab_size × d_model] where row i is the learned vector representation of token i.

The lookup operation is conceptually simple: convert token ID i to a one-hot vector of length vocab_size, then multiply by the embedding matrix. Practically, this is implemented as a direct row lookup: E[i]. For GPT-2 small: E is [50,257 × 768]. That is 50,257 × 768 = 38.6 million parameters — 31% of GPT-2 small's total 124M parameters, all in the embedding table.

What do embeddings learn? The distributional hypothesis (Harris 1954): words that appear in similar contexts get similar vectors. After training, cosine_similarity(E["king"], E["queen"]) ≈ 0.7 and the classic arithmetic E["king"] − E["man"] + E["woman"] ≈ E["queen"] holds approximately. This is not explicitly trained — it emerges from predicting co-occurrences. Every token, including punctuation and spaces, gets its own 768-dimensional vector. The embedding for a space before a capital letter is different from the embedding for a capital letter without a space — GPT-2's byte-level BPE distinguishes these.

    """Create a random embedding table (simulating initialized weights)."""
    random.seed(seed_offset)
    # Normal(0, 0.02) initialization like GPT-2
    return [[random.gauss(0, 0.02) for _ in range(d_model)]
            for _ in range(vocab_size)]

def lookup(E, token_id):
    """Embedding lookup: just return row token_id of the table."""
    return E[token_id]  # O(1) — no matrix multiplication needed

def dot(a, b):
    return sum(x*y for x, y in zip(a, b))

def norm(v):
    return math.sqrt(sum(x*x for x in v))

def cosine_sim(a, b):
    return dot(a, b) / (norm(a) * norm(b))

def vec_add(a, b):
    return [x+y for x, y in zip(a, b)]

def vec_sub(a, b):
    return [x-y for x, y in zip(a, b)]

def nearest(E, query, vocab, k=3):
    """Find k nearest tokens by cosine similarity."""
    sims = [(cosine_sim(query, E[i]), i) for i in range(len(E))]
    sims.sort(reverse=True)
    return [(vocab[i], s) for s, i in sims[:k]]

2.11

Positional Encoding

Token embeddings have a critical blind spot: they contain no information about where in the sequence a token appears. The embedding for "cat" at position 0 is identical to the embedding for "cat" at position 500. But position matters enormously — "the cat ate the fish" means something entirely different from "the fish ate the cat". Self-attention compounds this: it computes a weighted sum over all tokens simultaneously, with no built-in notion of order. Without explicit position information, a transformer is a bag-of-tokens model — permutation invariant.

Positional encoding solves this by adding a position-specific vector to each token embedding before the first transformer block. The model can then distinguish "cat at position 0" from "cat at position 500" because their input vectors differ. Four approaches are in wide use, each with different trade-offs:

Sinusoidal (Vaswani 2017)

Fixed mathematical patterns: PE(pos, 2i) = sin(pos / 10000^(2i/d)). No learned parameters. Theoretically generalizes to any length. Encodes absolute position only.

Original Transformer · BERT (variant) · T5

Learned Absolute (GPT-2)

A second nn.Embedding of shape [T_max × d_model]. GPT-2: 1,024 × 768 = 786K parameters. Trained jointly with the model. Cannot extrapolate beyond T_max.

GPT-2 · GPT-3 · GPT-J · early LLMs

RoPE (Su et al. 2021)

Rotates Q and K vectors by a position-dependent angle in attention. Encodes relative position, not absolute. No separate embedding table. Efficiently extendable with YaRN/RoPE scaling.

LLaMA 1/2/3 · Mistral · Falcon · Qwen

ALiBi (Press et al. 2022)

No learned positions — instead adds −m×|i−j| to each attention score (m is a head-specific constant). Penalizes attending to distant tokens. Extrapolates to 2× training length freely.

BLOOM · MPT · OPT-175B variants
SINUSOIDAL PE (Vaswani et al. 2017) PE(pos, 2i) = sin(pos / 10000^(2i/d_model)) PE(pos, 2i+1) = cos(pos / 10000^(2i/d_model)) Even dims use sin, odd dims use cos · different frequencies let model distinguish positions

Why sinusoidal? The frequency decreases geometrically — position 0 vs 1 differs a lot in the high-frequency (small i) dimensions, but is almost identical in the low-frequency (large i) dimensions. This creates a positional "fingerprint" that is unique for every position and, crucially, allows the model to compute relative positions via dot products: PE(pos+k)·PE(pos) depends only on k, not on the absolute value of pos. GPT-2 uses learned absolute PE instead — simpler, and the data regime (1024-token training) makes extrapolation unnecessary. Modern models use RoPE because it naturally handles relative position and scales better to long contexts.

RoPE in three sentences. Represent position as a rotation angle. Rotate Q and K vectors by their respective position angles before computing Q·Kᵀ. The dot product of rotated vectors QK naturally captures relative position (rotation of Q by pos_q minus rotation of K by pos_k = relative rotation of pos_q−pos_k). No separate position embedding layer, no parameter cost.

import math

def sinusoidal_pe(pos, d_model=16):
    """Generate the sinusoidal positional encoding for one position."""
    pe = []
    for i in range(d_model):
        freq = 1.0 / (10000 ** (2 * (i // 2) / d_model))
        if i % 2 == 0:
            pe.append(math.sin(pos * freq))   # even dims: sin
        else:
            pe.append(math.cos(pos * freq))   # odd dims: cos
    return pe

def dot(a, b):
    return sum(x*y for x,y in zip(a,b))

def norm(v):
    return math.sqrt(sum(x*x for x in v))

# Generate PE for positions 0..7
D = 16
positions = list(range(8))
PE = [sinusoidal_pe(p, D) for p in positions]

print("Sinusoidal PE — first 6 dimensions across positions:")
print(f"{'pos':>4}", end="")
for i in range(6):
    print(f"  dim{i:d}  ", end="")
print()
for p in positions:
    print(f"{p:>4}", end="")
    for i in range(6):
        print(f" {PE[p][i]:>7.3f}", end="")
    print()

print()
# Key property: dot(PE[a], PE[b]) depends only on |a-b|
print("Dot products (should depend only on distance, not absolute positions):")
for offset in [0, 1, 2, 3]:
    sims = []
    for base in range(0, len(positions) - offset):
        s = dot(PE[base], PE[base + offset]) / (norm(PE[base]) * norm(PE[base + offset]))
        sims.append(s)
    avg = sum(sims) / len(sims)
    std = math.sqrt(sum((s - avg)**2 for s in sims) / len(sims))
    print(f"  Offset {offset}: cosine sims = {[f'{s:.3f}' for s in sims]}  (std={std:.4f})")

print()

The Full Input Pipeline

Every concept in this chapter connects into a single pipeline. Raw text enters one end; a tensor of shape (B, T, d_model) exits the other end, ready for the first transformer block. Understanding this pipeline end-to-end is essential for debugging data issues, understanding model limitations, and designing preprocessing for fine-tuning.

The GPT-2 code for the final step — converting a batch of token IDs into the embedding tensor — is eight lines:

gpt2.py (model.py forward pass, first 8 lines) 
def forward(self, idx, targets=None):
    B, T = idx.size()
    # ↑ idx is the batch of token IDs, shape (B, T)

    # 1. Create position indices: [0, 1, 2, ..., T-1] for each sequence
    pos = torch.arange(0, T, device=idx.device).unsqueeze(0)
    # ↑ shape: (1, T) — same positions for every sequence in the batch

    # 2. Look up token embeddings: one 768-dim vector per token ID
    tok_emb = self.transformer.wte(idx)
    # ↑ wte: shape [50257, 768]; idx selects rows → output (B, T, 768)

    # 3. Look up positional embeddings: one 768-dim vector per position
    pos_emb = self.transformer.wpe(pos)
    # ↑ wpe: shape [1024, 768]; pos selects rows → output (1, T, 768)
    #   broadcasts to (B, T, 768) — same position vector for all batch items

    # 4. Add token and positional embeddings — elementwise sum
    x = self.transformer.drop(tok_emb + pos_emb)
    # ↑ x shape: (B, T, 768). This is the input to the first transformer block.
    #   Dropout applied here at training time (p=0.1 in GPT-2)

    # 5. Pass through N transformer blocks
    for block in self.transformer.h:
        x = block(x)
    # ↑ 12 blocks for GPT-2 small, each refining x in-place via residuals

The parameter count of the input pipeline for GPT-2 small: token embedding 50,257 × 768 = 38.6M, positional embedding 1,024 × 768 = 0.8M. Together, 39.4M parameters — 32% of the model's total. The remaining 68% lives in the transformer blocks. This ratio is why weight tying (sharing the token embedding with the output LM head) is so impactful: it converts 38.6M parameters from the input into free supervision for the output layer.

The +: the most important operator in GPT-2. tok_emb + pos_emb is an elementwise addition of two (B, T, 768) tensors. No concatenation, no gating, no learned combination — just addition. The model must learn, through training, to encode "this token's meaning" in one part of the 768-dim space and "this position's identity" in another part, such that gradient descent through the combined representation finds both signals useful.
