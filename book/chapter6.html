6.1

Transfer Learning

A model pretrained on 300 billion tokens of text has absorbed something that cannot be easily replicated: a general model of language. It knows that "cat" and "feline" are semantically similar, that subject-verb agreement is required, that "I can't believe how" usually precedes an emotional statement, and that certain phrases signal spam. This knowledge exists in the weights — specifically, in the patterns learned across all 12 transformer blocks. Fine-tuning transfers this knowledge to a new task by showing the model a small set of labeled examples and adjusting its weights slightly.

The key insight is that most of what makes a classifier good — understanding syntax, recognising entities, modelling sentiment — is exactly what language modeling learns. A model that can predict "The syringe contained a lethal dose of" is already tracking semantic relationships that matter for medical text classification. Fine-tuning on 2,000 labeled examples therefore doesn't start from scratch — it starts from a foundation that has already seen billions of examples of the problem structure.

Three paradigms exist, each with different trade-offs. Feature extraction: freeze all pretrained weights, train only the new classification head. Fast, low memory, works even with 100 examples, but can't adapt to domain-specific language. Partial fine-tuning: unfreeze the last few transformer blocks and the new head. Better performance, needs 500+ examples. Full fine-tuning: all weights trainable. Best performance with enough data (2,000+), but requires careful LR selection to avoid catastrophic forgetting.

6.2

The Classification Head

Modifying GPT-2 for classification requires replacing one layer. Instead of the LM head (Linear(768 → 50,257) that predicts the next token), we add a classification head: Linear(768 → num_classes). The rest of the model — all 12 transformer blocks, the embeddings, the LayerNorms — remains unchanged. This is the elegance of the residual stream architecture: the final hidden state is a rich representation of the entire input, and any linear probe on top of it can extract a classification signal.

For a GPT-style (decoder-only, causal) model, the standard practice is to use the hidden state at the last token position as the input to the classification head. Unlike BERT (which uses the [CLS] token at position 0), GPT's causal attention means the last token can attend to the entire sequence — it has the most complete contextual information. For a sequence of length T, we take hidden_state[:, -1, :] (shape: batch × d_model) and project it to num_classes.

The classification head has 768 × num_classes + num_classes parameters. For binary spam detection (2 classes): 768 × 2 + 2 = 1,538 parameters. Compare that to the full GPT-2: 124 million. This means only 0.0012% of the model is new when doing feature extraction — every prediction comes almost entirely from pretrained knowledge, redirected by a tiny linear transformation.

Key insight: The classification head adds essentially no parameters relative to the pretrained model. Its job is not to add new knowledge — it's to learn which direction in the 768-dimensional residual stream corresponds to the target class. The pretrained model has already encoded that direction; the head just finds it.
classifier.py
class GPTClassifier:
    def __init__(self, pretrained_gpt, num_classes, freeze_backbone=True):
        self.backbone = pretrained_gpt         # ↑ full GPT-2 (without lm_head)
        self.head = Linear(768, num_classes)   # ↑ NEW: 768 → num_classes, randomly initialised
        if freeze_backbone:
            for param in self.backbone.parameters():
                param.requires_grad = False    # ↑ freeze: only head updates during training

    def forward(self, token_ids):
        # Run GPT backbone to get hidden states for every position
        hidden = self.backbone.get_hidden_states(token_ids)   # ↑ (batch, seq_len, 768)
        last   = hidden[:, -1, :]                            # ↑ last token position: (batch, 768)
        # The last position has attended to all preceding tokens — richest representation
        logits = self.head.forward(last)                      # ↑ (batch, num_classes) — raw class scores
        return logits

import math, random
random.seed(42)

# Simulate: pretrained model produces 4-dimensional embeddings
# "spam" texts cluster around one region, "ham" around another
def make_embeddings(n, center, noise=0.3):
    random.seed(42)
    return [[center[i] + random.gauss(0, noise) for i in range(len(center))]
            for _ in range(n)]

spam_emb = make_embeddings(20, [1.2, -0.8, 0.5, 1.1])
ham_emb  = make_embeddings(20, [-0.9, 0.7, -0.3, -0.8])

# Simple linear classifier: w · x + b, trained with gradient descent
w = [0.0, 0.0, 0.0, 0.0]
b = 0.0
lr = 0.1

def sigmoid(x): return 1/(1+math.exp(-max(-30,min(30,x))))
def dot(a, b): return sum(x*y for x,y in zip(a,b))

data = [(e,1) for e in spam_emb] + [(e,0) for e in ham_emb]
random.shuffle(data)

for epoch in range(30):
    loss = 0
    for x, y in data:
        p = sigmoid(dot(w,x)+b)
        err = p - y
        for i in range(len(w)):
            w[i] -= lr * err * x[i]
        b -= lr * err
        loss += -y*math.log(p+1e-9) - (1-y)*math.log(1-p+1e-9)

correct = sum(1 for x,y in data if round(sigmoid(dot(w,x)+b))==y)
print(f"Accuracy after 30 epochs: {correct}/{len(data)} = {100*correct/len(data):.0f}%")
print(f"Weights (classification direction): {[round(wi,3) for wi in w]}")
print("These weights define the direction in embedding space that separates classes.")
6.3

Freezing Layers

Freezing a layer means setting requires_grad = False on its parameters. During the backward pass, PyTorch skips computing gradients for frozen parameters and skips updating them in the optimizer step. The parameters simply do not change. This is computationally cheaper (no gradient storage for frozen layers) and prevents catastrophic forgetting — the phenomenon where fine-tuning on a small dataset causes the model to forget the general-purpose representations learned during pretraining.

With only 100 labeled examples, full fine-tuning is almost guaranteed to overfit and degrade the pretrained representations. Freezing all blocks and training only the head avoids this: the head sees gradient signal from the classification task, but the backbone — which holds 124M parameters of carefully learned language representations — stays intact. As dataset size grows, progressively unfreezing more layers generally helps: unfreeze the last 4 blocks with a small learning rate, then the full model with an even smaller rate.

Layer-wise learning rate decay (LLRD) is a complementary technique: instead of the same learning rate for all layers, use a decay factor (e.g. 0.9 per layer from top to bottom). The last block gets lr=1e-4, the block before it gets 0.9×1e-4, the one before 0.81×1e-4, and so on. This reflects the insight from §6.9: early layers learn general syntax (low LR = don't change much); late layers learn task patterns (higher LR = adapt more).

Key insight: Freezing is not a binary choice. Progressive unfreezing — thaw layers from top to bottom, one epoch at a time — is almost always better than either fully frozen or fully unfrozen fine-tuning. Howard & Ruder's ULMFiT paper demonstrated a 3-point accuracy gain on text classification benchmarks using this approach.
freezing.py
# Freeze all backbone parameters
def freeze_backbone(model):
    for name, param in model.backbone.named_parameters():
        param.requires_grad = False    # ↑ no gradient computed or stored for these
    # Only model.head.parameters() will be updated by the optimizer

# Progressive unfreezing: thaw last N blocks
def unfreeze_last_n_blocks(model, n):
    blocks = model.backbone.blocks  # ↑ list of 12 TransformerBlock instances
    for block in blocks[-n:]:        # ↑ last n blocks only
        for param in block.parameters():
            param.requires_grad = True    # ↑ re-enable gradient flow

# Layer-wise learning rate decay
def get_layer_lrs(model, base_lr=1e-4, decay=0.9):
    param_groups = []
    blocks = model.backbone.blocks
    for i, block in enumerate(reversed(blocks)):   # ↑ start from last block (highest LR)
        lr = base_lr * (decay ** i)                    # ↑ each block gets decay× less LR
        param_groups.append({"params": block.parameters(), "lr": lr})
    param_groups.append({"params": model.head.parameters(), "lr": base_lr})
    return param_groups    # ↑ pass this to optimizer: Adam(param_groups)

def gpt2_component_params(d=768, n_heads=12, n_layers=12, vocab=50257, ctx=1024):
    d_ff = 4 * d
    per_block = (2*2*d + 3*d*d + 3*d + d*d + d + d*d_ff + d_ff + d_ff*d + d)
    head_cls  = d * 2 + 2  # binary classification head
    emb       = vocab*d + ctx*d
    final_ln  = 2*d
    backbone  = emb + n_layers*per_block + final_ln
    return {
        "per_block": per_block,
        "backbone":  backbone,
        "head":      head_cls,
        "total":     backbone + head_cls
    }

6.4

Building a Spam Detector

The UCI SMS Spam Collection is an ideal benchmark: 5,574 English SMS messages, 747 spam (13.4%) and 4,827 ham (86.6%). The class imbalance is realistic — production spam filters see similar ratios. The messages are short (average 80 characters), which fits easily within GPT-2's 1,024-token context window. The task is binary: given a message, output 0 (ham) or 1 (spam).

The pipeline: tokenize each message with GPT-2's tokenizer → pad or truncate to 128 tokens → DataLoader with batch_size=32 → forward pass through GPT-2 classifier → cross-entropy loss against label → optimizer step. After 3 epochs of full fine-tuning, typical performance is accuracy 98.4%, F1 0.964, AUC 0.997. Feature extraction alone (frozen backbone, only head trained) achieves 95.2% accuracy, 0.89 F1 — substantially worse, but still far above a naive keyword filter (typical accuracy ~85%).

Two important implementation details. First: GPT-2's tokenizer produces variable-length outputs. SMS messages tokenize to 10–200 tokens. You must either pad all sequences to the same length within a batch (using a pad token and attention mask) or sort sequences by length and batch similarly-sized ones together. Second: the class imbalance (86.6% ham) means a model that predicts all-ham achieves 86.6% accuracy — this is the accuracy baseline trap. Always compare against this baseline and report F1, not just accuracy.

Key insight: A pretrained GPT fine-tuned on 4,457 SMS messages (80% train split) outperforms a logistic regression model trained on all 5,574 examples with TF-IDF features. The transfer learning head start — 300B tokens of language modeling — is worth thousands of labeled examples.
spam_detector.py
# Full fine-tuning pipeline for spam detection
import torch
from torch import nn

class SpamDataset(torch.utils.data.Dataset):
    def __init__(self, texts, labels, tokenizer, max_len=128):
        self.data = []
        for text, label in zip(texts, labels):
            ids = tokenizer.encode(text)[:max_len]        # ↑ truncate if > 128 tokens
            ids += [tokenizer.eos_id] * (max_len - len(ids))  # ↑ pad to max_len with eos token
            self.data.append((ids, label))

    def __getitem__(self, i): return self.data[i]
    def __len__(self): return len(self.data)

def train_classifier(model, train_loader, val_loader, epochs=3, lr=1e-4):
    optimizer = torch.optim.AdamW(model.parameters(), lr=lr, weight_decay=0.01)
    # ↑ AdamW: Adam with decoupled weight decay — standard for fine-tuning
    criterion = nn.CrossEntropyLoss()   # ↑ for 2-class: equivalent to binary cross-entropy

    for epoch in range(epochs):
        model.train()
        for token_ids, labels in train_loader:
            logits = model.forward(token_ids)       # ↑ (batch, 2) raw class scores
            loss   = criterion.forward(logits, labels)
            optimizer.zero_grad()
            loss.backward()                         # ↑ compute gradients for trainable params
            torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
            # ↑ gradient clipping: prevent exploding gradients from overriding pretrained weights
            optimizer.step()

        val_acc = evaluate(model, val_loader)
        print(ff"Epoch {epoch+1}: val_acc={val_acc:.3f}")

6.5

Accuracy, Precision, Recall, F1

Every classification metric is derived from four counts. True Positive (TP): the model predicted spam and was right. True Negative (TN): predicted ham, was right. False Positive (FP): predicted spam, but it was ham — a legitimate email incorrectly blocked. False Negative (FN): predicted ham, but it was spam — a spam email that slipped through. The relative cost of FP vs FN is business-specific: a spam filter might accept a high FP rate (annoying but safe) to minimise FNs (spam reaching the inbox), while a fraud detector accepts high FPs to minimise FNs (missed fraud).

Precision answers: "Of everything I labelled spam, how much was actually spam?" High precision means few legitimate emails blocked. Recall (also called sensitivity or TPR) answers: "Of all actual spam, how much did I catch?" High recall means few spam messages slip through. These metrics trade off: lowering the classification threshold increases recall (catch more spam) but decreases precision (block more legitimate mail). The F1 score is the harmonic mean of both, penalising extreme imbalances.

Why harmonic mean rather than arithmetic mean? If Precision=1.0 and Recall=0.0 (model labels nothing as spam), arithmetic mean = 0.5. Harmonic mean = 0.0 — correctly signalling that a metric with recall=0 is useless. The harmonic mean punishes extreme values much more severely. F1 = 0.9 means both Precision and Recall are reasonably high; there is no way to achieve F1=0.9 with Recall=0.1 regardless of how good precision is.

In spam detection, a high-recall low-precision model (catches all spam, blocks some ham) is usually preferable to a low-recall high-precision one (never blocks ham, but misses spam). Choose threshold based on the cost ratio of FP vs FN in your application.

▶ Cell 5 — Compute all classification metrics from a confusion matrix 
def classification_metrics(tp, fp, fn, tn):
    accuracy  = (tp + tn) / (tp + fp + fn + tn)
    precision = tp / (tp + fp) if (tp+fp) > 0 else 0
    recall    = tp / (tp + fn) if (tp+fn) > 0 else 0
    f1        = 2*precision*recall/(precision+recall) if (precision+recall) > 0 else 0
    return accuracy, precision, recall, f1

6.6

ROC-AUC

F1, precision, and recall all depend on a chosen classification threshold (typically 0.5). But the model outputs a probability, not a binary label. The ROC (Receiver Operating Characteristic) curve evaluates the model at every possible threshold simultaneously, giving a complete picture of the precision-recall trade-off that's independent of threshold choice.

At each threshold τ (from 1 down to 0), we classify all examples with probability ≥ τ as positive. We plot the True Positive Rate (TPR = TP/(TP+FN) = Recall) on the Y axis against the False Positive Rate (FPR = FP/(FP+TN)) on the X axis. As τ decreases, we catch more positives (TPR rises) but also flag more negatives as positive (FPR rises). A perfect classifier has a curve that jumps to TPR=1 before FPR rises at all, hugging the top-left corner. A random classifier follows the diagonal (TPR = FPR at every threshold).

AUC (Area Under the Curve) collapses the entire ROC curve to a single number. AUC = 0.5: random. AUC = 1.0: perfect. AUC has a probabilistic interpretation: it equals the probability that the model assigns a higher score to a randomly chosen positive example than to a randomly chosen negative example. AUC = 0.97 means: if you pick one spam and one ham at random, there's a 97% chance the spam gets a higher spam-probability than the ham.

Key insight: AUC measures ranking quality, not calibration. A model that consistently ranks positives above negatives has AUC = 1.0 even if the probabilities are poorly calibrated (e.g., outputting 0.3 for spam and 0.1 for ham — not well-calibrated, but perfectly ranked). For spam filtering where you'll choose a threshold post-deployment, AUC is the most informative single metric.

def roc_auc(preds):
    # Sort by probability descending
    sorted_preds = sorted(preds, key=lambda x: -x[0])
    P = sum(1 for _, y in preds if y == 1)
    N = sum(1 for _, y in preds if y == 0)

    points = [(0.0, 0.0)]
    tp = fp = 0
    for prob, label in sorted_preds:
        if label == 1: tp += 1
        else:          fp += 1
        points.append((fp/N, tp/P))  # (FPR, TPR)

    # AUC via trapezoid rule
    auc = sum((points[i][0]-points[i-1][0]) * (points[i][1]+points[i-1][1])/2
              for i in range(1, len(points)))
    return auc, points

auc, pts = roc_auc(predictions)
print(f"AUC = {auc:.4f}  (random = 0.5, perfect = 1.0)\n")
print("ROC curve sample points (FPR, TPR):")

6.7

Detecting Overfitting

Overfitting occurs when a model learns the training data too well — including noise and idiosyncrasies that don't generalise. The tell-tale sign is a divergence between training and validation loss: training loss keeps decreasing, but validation loss plateaus or begins rising. The model has memorised the training set rather than learning transferable patterns. This is especially common when fine-tuning large models on small datasets.

Three factors determine overfitting risk in fine-tuning: (1) Model capacity vs dataset size. GPT-2 124M fine-tuned on 500 examples is deeply at risk. Use feature extraction or LoRA instead of full fine-tuning. (2) Number of epochs. After 10 epochs on 500 examples, the model has seen each example 10 times. With a learning rate of 1e-4, that's plenty of iterations to overfit. 3 epochs is the standard for text fine-tuning. (3) Learning rate. A high LR causes rapid, dramatic weight changes that can both overfit faster and destroy pretrained representations (catastrophic forgetting).

The correct toolset for detecting and fighting overfitting: plot both training and validation loss at every epoch. Add dropout (GPT-2 already has 10% dropout in attention and FFN). Use weight decay (L2 regularisation, AdamW default 0.01). For extreme data scarcity, data augmentation (back-translation, synonym substitution) or retrieval augmentation can help. Never report test set performance until the final model is selected.

Early Stopping

Early stopping is the automatic application of the §6.7 insight: monitor validation loss, and stop training when it stops improving. The key parameter is patience: how many consecutive epochs without improvement before stopping. Patience = 3 means: if validation loss hasn't improved for 3 consecutive epochs, halt training and restore the weights from the best epoch. This prevents the common mistake of training for a fixed epoch count and returning the final (possibly overfit) model rather than the best one.

The delta parameter prevents stopping on noise: val_loss must improve by at least delta to count as an improvement. A common setting is delta = 0.001 — changes smaller than this are treated as flat. Without a delta, tiny fluctuations in validation loss (due to batch sampling randomness) can trigger premature stopping. With patience=3 and delta=0.001, training stops only when val_loss has been genuinely stagnant for 3 consecutive epochs.

Best practices: always checkpoint the model with the lowest validation loss, not the model at the final epoch. Use a separate held-out test set that's never seen during training or validation. The test set is only evaluated once, on the final model — using it for hyperparameter selection invalidates its estimate of generalisation performance.

Key insight: Early stopping is a form of regularisation. By limiting the number of gradient updates to those that actually improve generalisation, it constrains the effective model complexity without reducing the model size. Combined with a good learning rate and weight decay, it's the simplest and most reliable defence against overfitting in fine-tuning.
early_stopping.py
class EarlyStopping:
    def __init__(self, patience=3, delta=0.001):
        self.patience    = patience   # ↑ epochs to wait before stopping
        self.delta       = delta      # ↑ minimum improvement to count as better
        self.best_loss   = float('inf')
        self.counter     = 0          # ↑ epochs without improvement
        self.best_weights = None

    def step(self, val_loss, model):
        if val_loss < self.best_loss - self.delta:
            self.best_loss    = val_loss
            self.counter      = 0         # ↑ genuine improvement: reset patience counter
            self.best_weights = model.state_dict()  # ↑ save the best weights
            return False  # don't stop yet
        else:
            self.counter += 1            # ↑ no improvement: increment patience counter
            if self.counter >= self.patience:
                model.load_state_dict(self.best_weights)   # ↑ restore best weights before returning
                return True  # stop training
            return False

What Gets Transferred

Transfer learning is not magic — it is measurable. Tenney et al. (2019) trained linear classifiers on each BERT layer's hidden states and evaluated them on NLP tasks (part-of-speech, named entities, coreference). The results revealed a clear hierarchy: the NLP pipeline is physically encoded in the transformer layers from bottom to top. Early layers encode morphology and part-of-speech; middle layers encode semantic roles and named entities; late layers encode discourse-level patterns and task-specific associations.

This hierarchy has a direct implication for fine-tuning strategy. When transferring to a classification task, the late layers (9–12 in GPT-2) need to change the most — they must redirect from "what comes next in general web text" to "what class does this text belong to." The early layers (1–4) are already doing what classification needs — detecting syntax and morphology — and should be disturbed as little as possible. Middle layers (5–8) are partially task-specific and partially general; they benefit from a small learning rate update.

An important limitation: transfer learning breaks down when the fine-tuning domain is very different from pretraining. A GPT-2 pretrained on English web text fine-tuned on medical radiology reports will struggle because the vocabulary, syntax, and concepts are largely out-of-distribution. In this case, domain-adaptive pretraining — continuing pretraining on the target domain before fine-tuning — often recovers 2–5 F1 points. This was demonstrated by Gururangan et al. (2020) across 8 domains and 4 tasks.

Key insight: Tenney et al. found that BERT layers 1–4 are essentially identical to the classical NLP pipeline: tokenisation → POS → parsing → semantics → discourse. This means fine-tuning for any classification task is borrowing 4 free NLP pipeline steps that took decades of NLP research to build. You're paying for them with 300B tokens of pretraining compute — amortised across every user of the pretrained model.

Fine-Tuning Best Practices

After five chapters of theory, here are the practical decisions that determine fine-tuning success. These are empirically validated defaults — not rules, but strong priors that hold across most text classification tasks. Treat them as starting points, not constraints.

Learning rate: 2e-5 to 5e-5 for full fine-tuning; 1e-3 to 1e-4 for classification head only. These are the ranges that win on most benchmarks. Too high (1e-3 for full fine-tuning) causes catastrophic forgetting of pretrained weights. Too low (1e-6) causes slow convergence with no improvement over feature extraction. Use a linear warmup of 6% of total steps before applying cosine decay. Batch size: 16 or 32. Smaller batches are noisier but more updates per epoch — often better for small datasets. Epochs: 3–5 for most tasks with thousands of examples; 10–20 for very small datasets (200–500 examples) with early stopping enabled.

Data quality dominates. 1,000 high-quality, consistently-labeled examples outperform 10,000 noisy examples in almost every benchmark. Before adding more data, audit a sample: are the labels consistent? Are there near-duplicates that should be deduplicated? Is the label distribution representative of production? For imbalanced datasets, set class weights in the loss: weight = [1.0, total/(n_positive * 2)] — this upweights the minority class contribution to the gradient. For severely imbalanced data (99/1 ratio), oversample the minority class or use a focal loss.

Key insight: The single most impactful change in most fine-tuning projects is not the model architecture, loss function, or optimizer — it's data quality. Spending 4 hours auditing and cleaning labels typically produces a larger F1 gain than spending 4 hours tuning hyperparameters.
