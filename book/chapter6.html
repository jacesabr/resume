6.1

Transfer Learning

In 2018 Jeremy Howard and Sebastian Ruder published ULMFiT (Universal Language Model Fine-Tuning, arxiv 1801.06146) and demonstrated something that seemed almost too good: a model pretrained on general English text, then fine-tuned on a small task-specific dataset, could beat much larger models trained from scratch on that same task. On the IMDb sentiment benchmark they achieved 4.6% error with only 100 labeled examples — a regime where a model trained from scratch produces near-random outputs. Their key finding was quantitative: fine-tuning needs 18× less labeled data than training from scratch to reach equivalent performance. The pretraining is doing 95% of the work; fine-tuning redirects it.

The mechanism is representational reuse. A model pretrained to predict the next token must learn syntax (so it can model grammatical continuations), morphology (so it understands word forms), semantics (so it knows that "cat" and "feline" predict similar continuations), and discourse (so it can follow multi-sentence arguments). All of these are exactly the representations that a text classifier needs. GPT-2 fine-tuned on SST-2 (Stanford Sentiment Treebank) achieves 91.4% accuracy; the identical architecture trained from scratch on SST-2 alone achieves only 85.3%. That 6.1-point gap is the value of pretrained representations — it is knowledge extracted from hundreds of billions of tokens, transferred for free into a task the model has never seen.

Three strategies occupy the freeze/unfreeze spectrum. Feature extraction: freeze all pretrained weights, train only the new classification head. Fast, CPU-compatible, works with as few as 100 examples, but cannot adapt to domain-specific vocabulary. Partial fine-tuning: freeze early layers, unfreeze the last 2–4 transformer blocks plus the head. Better domain adaptation with 500+ examples. Full fine-tuning: all weights trainable. Best performance with 2,000+ clean examples, but requires a carefully tuned learning rate to avoid catastrophic forgetting. ULMFiT introduced a fourth option — gradual unfreezing — where one layer group is unfrozen per epoch from top to bottom, giving each layer time to adapt before the next is released. Their ablation showed this beats both fully frozen and fully unfrozen approaches on every benchmark tested.

6.2

The Classification Head

The language-model head in GPT-2 is a single linear layer: lm_head = Linear(768, 50257). It maps the 768-dimensional residual stream at each token position to a probability distribution over the 50,257-token vocabulary. For classification we replace it with a classification head: Linear(768, num_classes). Everything else — all 12 transformer blocks, the token and positional embeddings, the final LayerNorm — remains unchanged. The residual stream at the output is not discarded; we simply project it to a different target space.

Which token position to read matters. GPT uses causal (unidirectional) attention: every token can attend to itself and all preceding tokens, but not future tokens. This means the hidden state at position T-1 (the last token) has attended to the entire input sequence — it is the most information-rich position. For sequence classification we take hidden_states[:, -1, :] (shape: batch × 768) as the input to the classification head. BERT uses a dedicated [CLS] token at position 0 that was pretrained specifically to aggregate sequence-level representations; GPT has no equivalent, so the last position is the natural choice. A dropout layer (10–20%) immediately before the head provides additional regularisation; GPT-2 already uses resid_pdrop=0.1 throughout, and the extra dropout before the head reduces variance in the gradient signal.

The head adds almost no parameters. A 5-class sentiment classifier on GPT-2 small (768-dimensional hidden states): Linear(768, 5) = 768 × 5 + 5 = 3,845 parameters. The full GPT-2 small has 117,000,000 parameters. The head is 0.0033% of the model. For binary classification (spam / ham): Linear(768, 2) = 1,538 parameters. When doing pure feature extraction with a frozen backbone, those 1,538 parameters are the only ones that change during the entire training run. The pretrained model contributes 99.9987% of the computation; the head finds which linear direction in the existing representation space separates the classes.

Key insight: The classification head does not add knowledge — it locates the direction in the pretrained representation space that separates your target classes. That direction almost certainly already exists in the 768-dimensional space; the head is a search algorithm, not a learning algorithm.

import torch.nn as nn

class GPTClassifier(nn.Module):
    def __init__(self, backbone, num_classes, dropout=0.1):
        super().__init__()
        self.backbone = backbone                    # ↑ GPT-2 pretrained weights (no lm_head)
        self.drop = nn.Dropout(p=dropout)          # ↑ drop 10% of activations before head
        self.head = nn.Linear(768, num_classes)    # ↑ 768 hidden dims -> num_classes logits
        # GPT-2 small: 768-dim. Medium: 1024. Large: 1280. XL: 1600.

    def forward(self, input_ids, attention_mask=None):
        out = self.backbone(                        # ↑ run full 12-layer transformer
            input_ids=input_ids,
            attention_mask=attention_mask,
            output_hidden_states=True
        )
        hidden = out.hidden_states[-1]             # ↑ last block output: (batch, seq_len, 768)
        last_tok = hidden[:, -1, :]               # ↑ last token position: (batch, 768)
        last_tok = self.drop(last_tok)             # ↑ regularise before projection
        logits   = self.head(last_tok)             # ↑ (batch, num_classes) raw scores
        return logits                              # ↑ pass to CrossEntropyLoss (no softmax here)

6.3

Freezing Layers

In PyTorch, freezing a parameter means setting param.requires_grad = False. The backward pass skips gradient computation for frozen parameters entirely — no gradient is stored, no optimizer update is applied. The weights simply do not change. This is not only a memory and compute saving; it is a mechanism for preserving the pretrained representations that make transfer learning valuable. Fine-tuning GPT-2 (117M parameters) on 500 labeled examples leaves the model with 117M degrees of freedom but only 500 gradient signals. Given enough epochs it will memorise those 500 examples perfectly while generalising to nothing.

Howard & Ruder's gradual unfreezing schedule addresses this directly. Start with all backbone weights frozen; train only the classification head for one epoch until its random initialisation settles. Then unfreeze the last transformer block and train for another epoch. Continue unfreezing one block per epoch from top to bottom. Their ablation across six text classification benchmarks showed this outperformed both fully frozen feature extraction and fully unfrozen fine-tuning by an average of 3.1 F1 points. The intuition is that late layers encode task patterns (they need to change substantially), while early layers encode universal syntax (they should be disturbed as little as possible).

Discriminative learning rates (LLRD — layer-wise learning rate decay) complement gradual unfreezing. Each layer group receives a different learning rate, decaying by a factor of roughly 2.6× per layer from the classification head downward. If the head uses lr=1e-3: block 12 gets 3.85e-4, block 11 gets 1.48e-4, and so on down to block 1 at approximately 1.5e-6. The practical rule: with fewer than 1,000 labeled examples, freeze all backbone weights and train only the head; with 1,000–10,000, unfreeze the top 2–4 blocks; with 10,000+, full fine-tuning with LLRD is safe and usually optimal.

Key insight: Every unfrozen parameter is a parameter that can overfit. Freeze aggressively at first; unfreeze incrementally as you accumulate evidence that the model is generalising. The goal is to change as little as possible while adapting as much as necessary.

# Strategy A: Feature extraction (freeze everything except head)
def freeze_backbone(model):
    for param in model.backbone.parameters():
        param.requires_grad = False    # ↑ no gradient computed; no optimizer update
    # model.head parameters still have requires_grad=True by default

# Strategy B: Partial fine-tune (unfreeze top N blocks)
def unfreeze_top_n(model, n=4):
    freeze_backbone(model)                        # ↑ start fully frozen
    blocks = model.backbone.transformer.h          # ↑ list of 12 Block objects
    for block in blocks[-n:]:                      # ↑ last n (closest to output)
        for p in block.parameters(): p.requires_grad = True

# Strategy C: LLRD - different LR per layer group
def llrd_optimizer(model, head_lr=1e-3, decay=1/2.6):
    groups = [{"params": model.head.parameters(), "lr": head_lr}]
    blocks = model.backbone.transformer.h
    for i, block in enumerate(reversed(blocks)):  # ↑ enumerate from top (block 11) to bottom
        lr = head_lr * (decay ** (i + 1))          # ↑ each lower block gets decay x less LR
        groups.append({"params": block.parameters(), "lr": lr})
    return torch.optim.AdamW(groups, weight_decay=0.01)

# Check how many parameters are trainable after each strategy
def trainable_pct(model):
    total = sum(p.numel() for p in model.parameters())
    train = sum(p.numel() for p in model.parameters() if p.requires_grad)
    return 100 * train / total      # ↑ feature extract = 0.003%; full = 100%

Building a Spam Detector

The UCI SMS Spam Collection (Almeida et al. 2011) is the standard benchmark for binary text classification. It contains 5,574 SMS messages: 4,827 labeled “ham” (legitimate) and 747 labeled “spam” — an 86.6% / 13.4% split. This severe imbalance is deliberate; it mirrors real-world spam filter distributions. The critical consequence: a model predicting “ham” for every single message achieves 86.6% accuracy without learning a single pattern. Accuracy is not the right metric here. F1 on the spam class and PR-AUC are the metrics that matter.

The training pipeline has four stages. Tokenisation: each SMS is tokenized with the GPT-2 tokenizer. SMS messages are short (median ≈70 characters, ≈15 tokens after subword splitting), so max_length=128 covers all but a handful of outliers. Padding: sequences shorter than max_length are padded with the pad token ID; an attention mask (1 for real tokens, 0 for padding) tells the model to ignore padded positions when computing attention. Batching: 32 examples per batch is standard — fits comfortably on a 4GB GPU, provides stable gradient estimates. Learning rate: 2e-5 for full fine-tuning, 1e-3 for head-only. The pretraining learning rate for GPT-2 was approximately 6e-4; fine-tuning at 1/300th of that rate ensures the optimizer does not take steps large enough to overwrite pretrained representations. Two to five epochs is sufficient; more risks catastrophic forgetting.

To counteract the class imbalance, weight the loss by the inverse class frequency: the spam class weight = total / (2 × spam_count) ≈ 3.73; the ham class weight ≈ 0.58. Every spam example then contributes 3.73/0.58 ≈ 6.4× more gradient signal than a ham example, effectively equalising the model's exposure to both classes. A fine-tuned GPT-2 with this weighting achieves F1 ≈ 0.97–0.99 on the spam class. The performance comes almost entirely from the pretrained representations: the model already understands phrases like “FREE entry” and “claim your prize”; it just needs to learn that those patterns map to class 1.

Key insight: The SMS Spam Collection is 86.6% ham. A model predicting ham always achieves 86.6% accuracy, 0% spam recall, and F1 = 0.000. Never report accuracy as the primary metric on an imbalanced dataset. The baseline to beat is F1 ≈ 0.00, not accuracy 86.6%.

Accuracy, Precision, Recall, F1

Every binary classification result decomposes into four counts. A model produces a predicted label for each example; comparing predictions to ground truth yields: TP (true positives — spam flagged as spam), FP (false positives — ham incorrectly flagged as spam), FN (false negatives — spam that slipped through), and TN (true negatives — ham correctly passed). These four numbers are the atoms from which every classification metric is derived. Plotting them in a 2×2 confusion matrix immediately reveals where a model fails: too many FNs means spam is reaching the inbox; too many FPs means legitimate messages are being blocked.

Accuracy = (TP+TN)/(TP+FP+FN+TN). Correct for balanced datasets; dangerously misleading for imbalanced ones. For SMS spam (86.6% ham), all-ham prediction achieves 86.6% accuracy. Precision = TP/(TP+FP): of all messages the model flagged as spam, what fraction were genuinely spam? Measures the cost to the user of being falsely accused. Recall = TP/(TP+FN): of all actual spam messages, what fraction did the model catch? Measures the failure to protect. F1 = 2×P×R/(P+R): the harmonic mean. The harmonic mean is the correct choice (rather than arithmetic mean) because it is dominated by the smaller value: if precision is 0.99 but recall is 0.01, F1 = 0.02, correctly signalling near-uselessness. Macro-F1 averages F1 per class (unweighted); micro-F1 pools TP/FP/FN globally before computing, weighting by class frequency.

For spam detection, the cost of a false negative (spam reaches inbox) usually exceeds the cost of a false positive (legitimate email blocked), so recall is prioritised over precision. However, in a business context the calculus may reverse (a blocked customer order is expensive). This cost asymmetry is why choosing the right metric requires understanding the application before evaluating the model. A fine-tuned GPT-2 on SMS Spam typically achieves precision ≈ 0.97, recall ≈ 0.95, F1 ≈ 0.96 on the spam class — compared to F1 = 0.00 for the all-ham baseline.

def metrics(tp, fp, fn, tn):
    total = tp + fp + fn + tn
    acc  = (tp + tn) / total
    prec = tp / (tp + fp) if (tp + fp) > 0 else 0.0
    rec  = tp / (tp + fn) if (tp + fn) > 0 else 0.0
    f1   = 2*prec*rec / (prec+rec) if (prec+rec) > 0 else 0.0
    return acc, prec, rec, f1

ROC-AUC

Precision, recall, and F1 all depend on a threshold (typically 0.5) applied to the model's output probability. But the model produces a score, not a binary label, and the threshold is a deployment decision that should not be baked into model evaluation. The ROC (Receiver Operating Characteristic) curve evaluates the model at every possible threshold simultaneously. The name comes from WWII radar analysis: operators faced the same trade-off between sensitivity (catching real signals) and specificity (ignoring noise), and the ROC curve was developed to compare operator performance independent of their chosen operating point.

At each threshold τ from 1 down to 0, every example with score ≥ τ is labelled positive. Plot the True Positive Rate (TPR = TP/(TP+FN) = Recall) on the Y-axis against the False Positive Rate (FPR = FP/(FP+TN)) on the X-axis. As τ decreases, TPR rises (we catch more spam) but FPR also rises (we flag more ham). A perfect classifier jumps to TPR=1 before FPR leaves 0, hugging the top-left corner. A random classifier follows the diagonal (TPR = FPR at every threshold, AUC = 0.5). AUC (Area Under the ROC Curve) collapses the entire curve to a single number with a powerful probabilistic interpretation: AUC equals the probability that the model assigns a higher score to a randomly chosen positive example than to a randomly chosen negative example. AUC = 0.97 means: pick one spam and one ham at random; there is a 97% chance the spam gets a higher spam-probability score.

For severely imbalanced datasets, PR-AUC (Precision-Recall AUC) is often more informative than ROC-AUC. The random baseline for PR-AUC equals the class prevalence (0.134 for SMS spam), whereas the random ROC-AUC baseline is always 0.5 regardless of imbalance. A model with ROC-AUC = 0.95 on 1% prevalence data can have PR-AUC as low as 0.20 because the massive number of true negatives inflates the ROC curve. Fine-tuned BERT on IMDb (50/50 split) achieves ROC-AUC ≈ 0.98; fine-tuned GPT-2 achieves ≈ 0.96. On SMS spam (imbalanced), PR-AUC is the more honest metric.

Key insight: AUC measures ranking quality across all thresholds, not calibration at any single threshold. A model that consistently ranks positives above negatives can have AUC = 1.0 even if its probabilities are poorly calibrated. This makes AUC ideal for comparing models; F1 at the chosen deployment threshold is what matters for reporting final performance.

def roc_auc(preds):
    preds_s = sorted(preds, key=lambda x: -x[0])  # sort by score descending
    P = sum(y for _, y in preds)
    N = len(preds) - P
    pts = [(0.0, 0.0)]
    tp = fp = 0
    for prob, label in preds_s:
        if label == 1: tp += 1
        else:          fp += 1
        pts.append((fp/N, tp/P))
    # Trapezoidal rule for AUC
    auc = sum((pts[i][0]-pts[i-1][0])*(pts[i][1]+pts[i-1][1])/2
              for i in range(1, len(pts)))
    return auc, pts

Detecting Overfitting

Overfitting occurs when a model learns the training data too well — including its noise and idiosyncrasies — and fails to generalise. The canonical diagnostic signal is a divergence between training and validation loss: training loss continues to decrease, but validation loss plateaus or begins rising. The model has memorised the training labels rather than learning transferable patterns. In the context of fine-tuning, this is especially dangerous because the pretrained model has millions of parameters that can easily memorise a small fine-tuning dataset.

Three factors determine overfitting risk. Model capacity vs dataset size: GPT-2 (117M params) fine-tuned on 500 examples has 234,000 parameters per training example. The model can trivially memorise the dataset. Feature extraction (training only the 1,538-parameter head) eliminates this risk. Epoch count: after 10 epochs on 500 examples, the model has seen each example 10 times. With a learning rate of 1e-4 that is more than sufficient to overfit. The standard for text classification fine-tuning is 2–5 epochs. Learning rate: a high LR (1e-3 for full fine-tuning) causes rapid weight changes that both overfit faster and destroy pretrained representations — this is catastrophic forgetting, where the fine-tuning gradient overwrites the general-purpose representations built during pretraining. Keep the learning rate at 1e-5 to 5e-5 for full fine-tuning.

The generalisation gap — val_loss − train_loss — is a useful scalar summary. A healthy gap is less than 0.1 for most classification tasks. If the gap exceeds 0.2 and continues to grow, overfitting is occurring. Standard countermeasures: dropout (GPT-2 already uses 0.1 throughout), weight decay (AdamW default λ=0.01), label smoothing (ε=0.1 — softens one-hot targets from hard 0/1 to 0.05/0.95, preventing the model from becoming overconfident). If both training and validation loss are high, you are underfitting — the model is not learning at all, usually because the learning rate is too low or the dataset is too small for the chosen strategy.

Early Stopping

Early stopping monitors the validation metric after each epoch and halts training when it stops improving. The mechanism is simple: track the best validation loss seen so far; if the current epoch's validation loss is not better than the best by at least δ (a small tolerance like 0.001), increment a patience counter; if the counter reaches the patience limit, stop training and restore the saved checkpoint. The δ threshold prevents stopping on noise — a validation loss of 0.2413 vs 0.2411 is not a meaningful improvement and should not reset the counter.

Patience is the main hyperparameter. Patience = 2 is aggressive (stop after 2 epochs without improvement); patience = 5 is conservative. For fine-tuning classification models, patience = 3 is a robust default. The saved checkpoint is the model at the epoch with the best validation loss, not the final epoch's weights — restoring the best checkpoint at the end is critical; without this, you get the overfitted final-epoch weights. ReduceLROnPlateau is a gentler alternative: instead of stopping, reduce the learning rate by a factor (typically 0.5) when the validation metric has plateaued for patience epochs. This often squeezes out another 1–2% improvement before eventually stopping.

One important rule: never look at the test set until the final model has been selected using the validation set. Using test performance to make training decisions (number of epochs, hyperparameter choices) leaks test-set information into the model selection process, causing optimistically biased performance estimates. The workflow is: train on the training set, tune all decisions on the validation set, report final numbers once on the test set at the very end.

Key insight: Early stopping is not just about avoiding overfitting — it is about finding the sweet spot where the model has learned the task-specific patterns without beginning to memorise the training set's idiosyncrasies. The best checkpoint is almost never the last one.

class EarlyStopping:
    def __init__(self, patience=3, delta=0.001, mode="min"):
        self.patience = patience      # ↑ epochs to wait before stopping
        self.delta    = delta          # ↑ minimum improvement to count as real progress
        self.mode     = mode           # ↑ "min" for loss, "max" for accuracy/F1
        self.best     = float("inf") if mode == "min" else float("-inf")
        self.counter  = 0
        self.best_epoch = 0

    def step(self, val_metric, epoch):
        improved = (
            val_metric < self.best - self.delta  if self.mode == "min"
            else val_metric > self.best + self.delta   # ↑ "max" mode
        )
        if improved:
            self.best = val_metric
            self.counter = 0             # ↑ reset counter on genuine improvement
            self.best_epoch = epoch
            return False, True           # ↑ (should_stop=False, is_best=True)
        self.counter += 1
        should_stop = self.counter >= self.patience  # ↑ stop if counter hits patience
        return should_stop, False

What Gets Transferred

Transfer learning is measurable. Tenney et al. (2019, ICLR, arxiv 1905.05950) trained linear classifiers on the hidden states of each individual BERT layer and evaluated them on a suite of NLP tasks: part-of-speech tagging, chunking, named entity recognition, semantic role labeling, coreference resolution, and others. The results revealed a clear hierarchy that mirrors the classical NLP processing pipeline — a hierarchy that engineers had developed over decades was independently re-discovered by the model during unsupervised pretraining. Layer 2–3: morphology and POS tagging peak. Layer 7–8: named entity recognition peaks (layer 8: 74.4% F1). Layer 10: semantic role labeling peaks (64.4% F1). Layer 12: task-specific and discourse-level patterns.

This has concrete implications for fine-tuning strategy. When adapting GPT-2 for spam classification, the late layers (9–12) need the most change: they were optimised for “predict next token given general web text” and need to redirect toward “map this input to a class label.” The early layers (1–4) are already doing exactly what classification needs — detecting word forms, syntactic roles, and basic semantics — and should be disturbed as little as possible. This is why discriminative learning rates (lower LR for lower layers) work: they let late layers adapt substantially while keeping early layers nearly frozen.

Transfer also breaks down when the fine-tuning domain is sufficiently different from pretraining. Gururangan et al. (2020, arxiv 2004.10964) showed that domain-adaptive pretraining — continuing pretraining on the target domain before task fine-tuning — recovers 2–5 F1 points across 8 domains (biomedical, CS, news, reviews) and 4 tasks. The intuition: if the model has never seen clinical radiology reports during pretraining, its early layers have not learned the domain-specific morphology and the late layers cannot build domain-accurate semantic representations. Continued pretraining on unlabeled domain text fills this gap without requiring any additional labels.

Key insight: Tenney et al. found that BERT's layers 1–4 re-implement the classical NLP pipeline: tokenisation → POS → parsing → semantics → discourse. This means every fine-tuned classifier gets four NLP pipeline stages for free — stages that took the NLP community decades to build. You are paying for them with pretraining compute, amortised across all users of the shared pretrained model.

Fine-Tuning Best Practices

Learning rate is the single most important hyperparameter for fine-tuning. The safe range for full fine-tuning is 1e-5 to 5e-5; for the classification head alone, 1e-3 to 1e-4. Rates above 1e-4 for full fine-tuning risk catastrophic forgetting within the first epoch. Use a linear warmup for the first 6–10% of training steps, then linear or cosine decay to zero. The warmup prevents the optimizer from taking large steps when the classification head is still randomly initialised, which would corrupt the backbone weights in the first few iterations. Smith (2017) showed that a learning rate finder — running one epoch with exponentially increasing LR and plotting loss vs LR — identifies the right range faster than grid search.

Batch size: 16 or 32 is standard for text classification. Smaller batches produce noisier gradients but more updates per epoch, which can help on very small datasets by exposing the model to more parameter update opportunities. Larger batches (64+) require gradient accumulation if memory-constrained but stabilise training when data is abundant. Max sequence length: most SMS messages need at most 128 tokens; most product reviews fit in 256; most documents in 512. Truncating to 256 instead of 512 halves the memory and compute cost of attention (which scales O(n²)). Data augmentation for small datasets: back-translation (English → German → English with a translation model) generates paraphrases that preserve semantic content. Wei & Zou (2019, arxiv 1901.11196) showed that synonym replacement, random insertion, random swap, and random deletion each independently improve classification performance on small datasets by 0.5–3.0% F1.

Ensemble methods: fine-tuning the same pretrained model five times with different random seeds and taking the majority vote across seeds improves accuracy by 1–2 points on most benchmarks at the cost of 5× inference compute. Class weights: for any imbalanced dataset, set CrossEntropyLoss(weight=[...]) with weights proportional to the inverse class frequency. This is equivalent to oversampling the minority class and is dramatically easier to implement. Data quality dominates: spending four hours cleaning and re-labeling ambiguous examples in your training set typically produces a larger F1 gain than four hours of hyperparameter tuning.

Key insight: The most impactful change in most real-world fine-tuning projects is not the model, loss function, or optimizer — it is data quality. 1,000 clean, consistently labeled examples outperform 10,000 noisy examples on nearly every benchmark. Audit a random sample before tuning anything else.

Multi-Label and Multi-Class Problems

Text classification splits into two fundamentally different problem types based on whether the classes are mutually exclusive. Multi-class (also called single-label): each input belongs to exactly one class. The correct output layer is Linear(d, C) followed by softmax over all C classes, with cross-entropy loss. The C scores are constrained to sum to 1 (they form a probability distribution), and the highest score is the predicted class. Examples: sentiment (positive / negative / neutral), topic classification (sports / politics / entertainment / technology), spam (spam / ham). Multi-label: each input can simultaneously belong to zero, one, or multiple classes. The correct output layer is Linear(d, C) followed by sigmoid applied independently to each of the C outputs, with binary cross-entropy (BCE) computed per class. The C scores are independent probabilities; each is thresholded separately (default: 0.5). Examples: document tagging (a news article may be simultaneously tagged “politics” AND “economy”), toxic comment detection (a comment may be simultaneously “toxic” AND “threat”).

The choice of output activation — softmax vs sigmoid — encodes your assumptions about class structure. Softmax says: “exactly one class is true; the model's confidence in one class should come at the expense of others.” Sigmoid says: “each class is an independent binary question; confidence in one class tells you nothing about the others.” Applying softmax to a multi-label problem will actively hurt performance: the loss function will penalise the model for assigning high probability to two simultaneously correct classes, even when both are genuinely present. Conversely, applying sigmoid to a single-label problem loses the mutual-exclusivity constraint and will produce poorly calibrated probabilities.

Thresholding in multi-label problems is a deployment decision, not a training decision. The default threshold of 0.5 is rarely optimal. For a multi-label toxic comment classifier, you might lower the threshold to 0.3 to catch more positives (higher recall) at the cost of more false positives. The optimal threshold per class can be found by sweeping the validation set: for each class, compute F1 at every threshold from 0.1 to 0.9 in steps of 0.05 and pick the threshold that maximises F1 on validation. This threshold optimisation adds no additional model training and typically improves macro-F1 by 2–5 points on multi-label benchmarks.

Key insight: Softmax enforces that scores sum to 1 — giving confidence to one class takes it from others. Sigmoid treats each class independently. If your classes are mutually exclusive, use softmax + cross-entropy. If multiple classes can be simultaneously true, use sigmoid + binary cross-entropy. Getting this wrong silently degrades your model.

Catastrophic Forgetting and How to Fight It

Catastrophic forgetting occurs when training a neural network on a new task causes it to lose performance on tasks it previously knew. In the fine-tuning context: updating GPT-2's weights on spam classification can, with a high enough learning rate or too many epochs, overwrite the general language representations the model learned during pretraining. The model stops being a good language model and becomes an overfit spam classifier that fails on any text that does not closely resemble its training distribution. The plasticity-stability dilemma is fundamental: the same property (weight updatability) that allows learning also allows forgetting.

Kirkpatrick et al. (2017, arxiv 1612.00796) proposed Elastic Weight Consolidation (EWC), a principled solution from the continual learning literature. EWC adds a regularisation term to the loss function that penalises large changes to parameters that were important for the previous task. “Importance” is measured by the diagonal of the Fisher Information Matrix — which estimates how much the model's output changes in response to each parameter being perturbed. Parameters with high Fisher Information (changing them a lot would hurt Task A performance) receive a high penalty for being modified during Task B fine-tuning. The EWC loss term is: λ ∑_i F_i (θ_i − θ*_i)², where F_i is the Fisher importance for parameter i, θ*_i is the Task A optimal value, and λ controls the regularisation strength. EWC demonstrated that a single neural network could sequentially learn 10 Atari games without forgetting earlier games.

For practical fine-tuning, EWC is rarely necessary because there are simpler countermeasures. Small learning rate (1e-5 to 2e-5) ensures that the optimizer takes small steps, making complete forgetting geometrically impossible in 3–5 epochs. Frozen early layers preserve the general-purpose representations that are most at risk. Early stopping prevents the model from training long enough to overfit. Replay: mix a small fraction of general pretraining data (e.g., Wikipedia text) into the fine-tuning batches; this provides a gradient signal that pushes back against forgetting. In practice, the combination of lr=2e-5 + frozen early layers + 3 epochs eliminates catastrophic forgetting for most text classification use cases without any explicit EWC regularisation.

Key insight: Catastrophic forgetting is a genuine risk but a solved problem in practice. The solution is simply a small learning rate. At lr=2e-5, even 5 full fine-tuning epochs do not cause measurable degradation on general NLP benchmarks. EWC is valuable for sequential multi-task learning; for single-task fine-tuning, keep the LR small.

Fine-Tuning with Limited Labels

Few-shot fine-tuning — learning from 1 to 100 labeled examples — is one of the most practically important problems in NLP. Labeling data is expensive, time-consuming, and often requires domain expertise. In many real-world applications, the annotation budget is measured in tens of examples, not thousands. Naive full fine-tuning on 10 examples will overfit catastrophically regardless of learning rate: the model has 117M parameters and 10 gradient signals. Feature extraction (training only the classification head) is much more stable but may not adapt to domain-specific language. The right approach depends on the amount of data and the nature of the task.

SetFit (Tunstall et al. 2022, arxiv 2209.11055) is currently the most effective approach for very low-label regimes (8–64 examples per class). SetFit fine-tunes a sentence embedding model using contrastive learning on sentence pairs: positive pairs (two sentences from the same class) are pushed together in embedding space; negative pairs (sentences from different classes) are pushed apart. This requires only class labels, no task-specific architecture, and converges with as few as 8 labeled examples per class. On 8-shot benchmarks, SetFit with a 110M sentence encoder matches or outperforms GPT-3 (175B) in-context learning. The key insight is that contrastive learning extracts more signal from each example than cross-entropy: a single labeled example generates one positive pair for every other example in the same class, and one negative pair for every example in different classes, producing n²/2 training pairs from n examples.

In-context learning (ICL) is the zero-parameter alternative: include labeled examples in the prompt and ask the model to classify new examples by analogy. GPT-3 with 10 labeled examples in the prompt achieves competitive performance on many classification benchmarks without any weight updates. ICL requires no training infrastructure, no held-out validation set, and can be updated instantaneously by changing the prompt. The trade-off: each inference call includes the full set of examples (10 examples × ~50 tokens each = 500 tokens of overhead per query), ICL performance is sensitive to example selection and ordering, and very large models (GPT-3+) are needed for reliable few-shot ICL. For deployments where low latency and low cost per query matter more than annotation savings, few-shot fine-tuning wins; for maximum label efficiency and minimal infrastructure, ICL wins.

Key insight: SetFit with 8 examples per class outperforms GPT-3 175B in-context learning on 7 of 8 few-shot classification benchmarks (Tunstall et al. 2022), using a model 1,590 times smaller. Label efficiency is not just about the number of examples — it is about the learning algorithm that extracts signal from each example.

def generate_pairs(data, n_positive_per_class=5, n_negative=20):
    pairs = []
    classes = list(data.keys())

    for cls, examples in data.items():

        a = random.choice(data[c1])
        b = random.choice(data[c2])
        pairs.append((a, b, 0, f"neg/{c1}-{c2}"))
    return pairs
