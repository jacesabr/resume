What Is a Large Language Model?

A large language model is, at its core, a function that maps a sequence of tokens to a probability distribution over the next token. That deceptively simple objective — predict what comes next — when applied to enough data with enough parameters, gives rise to systems that translate, reason, write code, and hold coherent conversations.

The "large" in LLM refers to both parameter count (billions) and training data (trillions of tokens). Neither alone is sufficient: scale interacts. A 7B-parameter model trained on 1T tokens outperforms a 70B model trained on 100B tokens on many tasks, as the Chinchilla scaling laws (§1.9) formalise.

The autoregressive loop. At inference time, the model is called repeatedly. The output token is appended to the input, and the model runs again — until an end-of-sequence token is produced or a length limit is hit. Every output token is a one-way commitment: there is no backtracking.

The choice of how to select the next token from that probability distribution matters enormously. Greedy decoding (always take the highest-probability token) is fast but produces repetitive text. Beam search, long the gold standard in machine translation, generates text that feels "hollow and artificial," as Holtzman et al. (2019) demonstrated. Nucleus (top-p) sampling — first truncate the distribution to the smallest set of tokens whose probabilities sum to at least p, then sample — empirically produces the most coherent and diverse text.

import numpy as np

def softmax(logits, temperature=1.0):
    # Temperature > 1 → flatter dist (more random)
    # Temperature < 1 → sharper dist (more greedy)
    logits = np.array(logits) / temperature
    e = np.exp(logits - logits.max())  # numerically stable
    return e / e.sum()

def top_p_sample(logits, p=0.9, temperature=1.0):
    probs = softmax(logits, temperature)
    sorted_idx = np.argsort(probs)[::-1]
    cumulative = np.cumsum(probs[sorted_idx])
    # Keep only tokens in the nucleus
    cutoff = np.searchsorted(cumulative, p) + 1
    nucleus_idx = sorted_idx[:cutoff]
    nucleus_probs = probs[nucleus_idx]
    nucleus_probs /= nucleus_probs.sum()
    return np.random.choice(nucleus_idx, p=nucleus_probs)

Tokenisation

Models don't see characters or words — they see tokens. A token is a subword unit discovered by a compression algorithm. The dominant approach is Byte-Pair Encoding (BPE), introduced by Sennrich et al. (2016). BPE starts with individual bytes and iteratively merges the most frequent adjacent pair until a target vocabulary size is reached. GPT-4 uses a vocabulary of 100,257 tokens (the cl100k_base encoding).

BPE has some surprising failure modes. The string "SolidGoldMagikarp" was a Reddit username that appeared so rarely it effectively became a single token, causing GPT-3 to output random text when prompted with it. Numbers tokenise inconsistently: 1234 might be one token, but 12345 might be three, which is why LLMs struggle with arithmetic. Languages with richer morphology (Turkish, Finnish) have much higher token fertility — more tokens per word — making them effectively "more expensive" to process.

WordPiece vs BPE vs SentencePiece. BERT uses WordPiece, which maximises the likelihood of the training corpus rather than frequency. SentencePiece (used by LLaMA, T5) operates on raw bytes and avoids language-specific pre-tokenisation, making it truly language-agnostic.

def get_pairs(vocab):
    """Count all adjacent symbol pairs across all words."""
    pairs = Counter()

        symbols = word.split()

Embeddings

Each token is mapped to a dense vector of floating-point numbers — its embedding. These vectors are learned during training. The geometry of the resulting space encodes semantic relationships: similar words cluster together, and vector arithmetic often mirrors conceptual arithmetic. The famous king − man + woman ≈ queen result (Mikolov et al., 2013) is a property that emerges automatically — nobody hardcoded it.

Modern LLMs use embedding dimensions of 4,096 (LLaMA-3 8B) to 12,288 (GPT-3 175B). The embedding table alone can be large: 100K tokens × 4,096 dims × 2 bytes (BF16) = 800MB, which is why tying input and output embeddings (using the same matrix for both) saves substantial memory.

The antonym paradox. "hot" and "cold" have high cosine similarity despite being opposites, because they appear in identical contexts ("it was very ___"). Embedding similarity captures distributional similarity, not conceptual similarity. This is a fundamental limitation worth understanding.

Transformer Architecture

The transformer (Vaswani et al., 2017) stacks L identical blocks, each containing two sublayers: multi-head self-attention and a position-wise feedforward network. Each sublayer is wrapped in a residual connection and followed by layer normalisation. The residual streams are the key architectural insight: gradients flow directly from output to input through additive skip connections, making training stable even at 100+ layers.

Modern decoder-only LLMs (GPT family, LLaMA, Mistral) use Pre-Norm (LayerNorm before the sublayer) rather than the original Post-Norm, which further stabilises training. Llama 3 uses RMSNorm instead of LayerNorm — it omits the mean-centering step, saving computation with negligible quality difference.

def count_params(d, d_ff, L, H, kv_heads, V, rope=True):
    """Approximate parameter count for a decoder-only LLM."""
    # Embedding
    embed = V * d

    # Per-layer attention: Q, K, V, O projections
    d_h = d // H       # head dimension
    q_proj  = d * d              # Q: d→d
    k_proj  = d * (kv_heads * d_h)  # K: d→kv_heads*d_h
    v_proj  = d * (kv_heads * d_h)  # V: same
    o_proj  = d * d              # O: d→d
    attn = q_proj + k_proj + v_proj + o_proj

    # Per-layer FFN (SwiGLU uses gate+up+down, 2/3 intermediate)
    d_int = int(d_ff)   # intermediate dim (already 2/3 * 4d for SwiGLU)
    ffn = d * d_int * 2 + d_int * d  # gate, up, down

    # Norms (RMSNorm: just one scale per dim, no bias)
    norm = d * 2 * L   # 2 norms per layer

    # LM head (often tied to embeddings, so not counted separately)
    total = embed + L * (attn + ffn) + norm
    return total

Self-Attention

Self-attention computes, for each token, a weighted average of all other token representations. The weights come from the similarity between that token's query vector and every other token's key vector. The value vectors are then averaged using these weights. The operation is differentiable end-to-end — the model learns which relationships to pay attention to purely from data.

The −∞ mask entries become zero after softmax, so future tokens receive zero attention weight. This is what makes the transformer autoregressive at training time — all positions are trained in parallel, but each token only attends to its left context. import numpy as np

def softmax(x, axis=-1):
    e = np.exp(x - x.max(axis=axis, keepdims=True))
    return e / e.sum(axis=axis, keepdims=True)

def attention(Q, K, V, mask=None):
    d_k = Q.shape[-1]
    scores = Q @ K.T / np.sqrt(d_k)   # (T, T)
    if mask is not None:
        scores = np.where(mask, scores, -1e9)
    weights = softmax(scores)
    return weights @ V, weights

Multi-Head Attention & GQA

Running attention once gives the model a single perspective. Multi-head attention (MHA) runs H separate attention operations in parallel, each with head dimension d_k = d_model/H. Each head learns to attend to different relationship types — one might track subject-verb agreement, another coreference chains, another syntactic dependencies. The outputs are concatenated and projected back to d_model via W_O.

For large models, MHA's memory cost is prohibitive during inference due to the KV cache (§1.11). Multi-Query Attention (MQA) (Shazeer, 2019) uses a single K and V head shared across all Q heads, reducing KV cache by H-fold with some quality cost. Grouped-Query Attention (GQA) (Ainslie et al., 2023, arXiv:2305.13245) is the sweet spot: G groups of Q heads share one K/V head each. LLaMA-3 uses 32 Q-heads and 8 KV-heads (4 Q-heads per KV group), reducing KV cache to 25% of MHA with negligible quality loss. Ainslie et al. showed GQA matches MHA quality while approaching MQA inference speed.

Positional Encoding & RoPE

Self-attention is permutation-invariant — if you shuffle all token positions, the attention weights change but the form of the computation is identical. Word order must be injected explicitly. The original transformer used fixed sinusoidal encoding: PE(pos,2i) = sin(pos/100002i/d), PE(pos,2i+1) = cos(…). The wavelengths form a geometric sequence from 2π to 10000·2π. The key property: a relative offset PE(pos+k) can be expressed as a linear function of PE(pos), which the model can learn to exploit.

Rotary Position Embedding (RoPE) (Su et al., 2022, arXiv:2104.09864) is the current standard, used in LLaMA, Mistral, GPT-NeoX, Falcon, and most recent models. RoPE encodes position by rotating Q and K vectors by an angle proportional to position: q_m = R(m)·q, k_n = R(n)·k. When the dot product Q·K is computed, the result depends only on the relative distance (m-n), not absolute positions. This gives zero-shot generalisation to unseen sequence lengths — a critical property for context extension.

YaRN extends LLaMA from 4K → 128K context using only 0.1% of original pretraining compute (Peng et al., 2023, arXiv:2309.00071). It rescales RoPE frequencies differently for low-frequency (high-dimension) components versus high-frequency ones, avoiding the “lost in the middle” failure mode of naive position interpolation. LLaMA-3 increases the RoPE base from 10,000 (standard) to 500,000 for its 128K-context variants — making the wavelengths dramatically longer and more robust to extended sequence lengths.

import math, random
random.seed(3)

def rope_rotate(x, pos, base=10000):
    # Apply RoPE to a vector x at position pos
    d = len(x)
    half = d // 2
    result = [0.0] * d
    for i in range(half):
        theta = 1.0 / (base ** (2*i / d))   # frequency for dimension i
        angle = pos * theta
        cos_a, sin_a = math.cos(angle), math.sin(angle)
        # Rotate pair (x[i], x[i+half]) by angle
        result[i]      = x[i] * cos_a - x[i+half] * sin_a
        result[i+half] = x[i] * sin_a + x[i+half] * cos_a
    return result

The Feedforward Sublayer

After attention routes information between positions, the feedforward network (FFN) processes each token independently and in parallel. The original FFN applies two linear projections with a nonlinearity: FFN(x) = W₂ · GELU(W₁ · x). The intermediate dimension is 4× d_model: for LLaMA-3 8B that's 4,096 → 16,384 → 4,096.

Geva et al. (2021, arXiv:2012.14913) made a striking discovery: FFN layers act as key-value memories. The first projection computes “keys”; the GELU selects which keys are active; the second projection reads out corresponding “values.” Specific factual knowledge (“Paris is the capital of France”) can be traced to specific neurons in specific FFN layers. Lower layers store syntactic patterns; upper layers store semantic facts. Meng et al. (2022, ROME) exploited this to directly edit factual associations by modifying a single FFN weight row.

SwiGLU (Shazeer, 2020, arXiv:2002.05202) replaces GELU with a gated variant: FFN(x) = W₂ · (SiLU(W₁·x) ⊗ W₃·x). The gate acts as a learned router. SwiGLU requires 3 matrices instead of 2, so the intermediate dimension is reduced to approximately 2/3 of the standard 4× value to keep parameter counts equal. LLaMA-3 8B uses d_ff=14,336 (≈ 2/3 × 4 × 4,096, rounded to a multiple of 256 for GPU efficiency). Shazeer showed GLU variants consistently outperform standard GELU/ReLU across model scales.

Pre-training at Scale & Chinchilla

LLMs are pretrained on vast, diverse text corpora by minimising next-token prediction loss. The datasets have grown dramatically: GPT-3 trained on ~300 billion tokens (mainly Common Crawl, WebText, Books1/2, Wikipedia); LLaMA-3 trained on 15 trillion tokens. Composition matters as much as size: code, books, and scientific papers consistently outperform raw web text per token because they contain denser reasoning, explicit structure, and factual content.

The Chinchilla scaling laws (Hoffmann et al., 2022, arXiv:2203.15556) upended the field. By training over 400 models from 70M to 16B parameters across 5B to 500B token budgets, they found that compute-optimal training requires roughly 20 tokens per parameter. GPT-3 (175B parameters) should have trained on 3.5 trillion tokens — it used only 300 billion, making it severely undertrained. The 70B Chinchilla model, trained on 1.4 trillion tokens, outperforms 280B Gopher on every evaluated benchmark while using identical compute. The message: “current large language models are significantly undertrained.”

# Coefficients from Hoffmann et al. 2022, Table 3 (Approach 3)
E     = 1.69    # irreducible entropy
A     = 406.4   # model-size scaling coefficient
B     = 410.7   # data-size scaling coefficient
alpha = 0.34    # model-size scaling exponent
beta  = 0.28    # data-size scaling exponent

def chinchilla_loss(N, D):
    return E + A / N**alpha + B / D**beta

def optimal_ND(C, steps=2000, lr=0.01):
    # Find compute-optimal N and D for budget C FLOPs

    D = C / (6 * N)
    return N, D

BERT vs GPT: Encoder vs Decoder

Both BERT and GPT are transformer-based, but they differ in attention pattern and training objective. GPT (decoder-only, Radford et al. 2018) uses causal (unidirectional) attention and is trained with causal language modelling (CLM): predict the next token given left context. This makes it naturally generative. BERT (encoder-only, Devlin et al. 2019, arXiv:1810.04805) uses bidirectional attention and is trained with masked language modelling (MLM): 15% of input tokens are randomly masked, and the model predicts them using context from both directions.

The bidirectional context gives BERT richer representations for classification tasks. But the autoregressive structure of GPT allows generation — and at scale, generation proves to be the more powerful paradigm. GPT-3 demonstrated that a large enough generative model performs classification by framing it as generation (few-shot prompting). By 2023, decoder-only models dominated essentially every NLP benchmark despite being “weaker” per token due to their restricted context window at each position.

T5 (Raffel et al., 2020, arXiv:1910.10683) tried a third approach: encoder-decoder with a unified text-to-text framework. Every task is framed as text-to-text conversion. T5-XXL (11B) was competitive with GPT-3 (175B) on many benchmarks — a 16× parameter efficiency advantage. But the encoder-decoder paradigm hasn't scaled as cleanly as pure decoders, and 2023–2024 saw near-universal consolidation around decoder-only architectures.

KV Cache & Inference Memory

During inference, each token's K and V projections are computed once and cached. When the next token is generated, only the new token's Q, K, V are computed — then the new K and V are appended to the cache, and attention is computed between the new Q and all cached K/V pairs. Without caching, each step requires O(N²) re-computation; with the cache, each step is O(N) in total across the context.

The memory cost is substantial. LLaMA-3 8B: 32 layers, 8 KV-heads, d_head=128, BF16 (2 bytes). One token occupies 32 × 8 × 128 × 2 × 2 = 131,072 bytes ≈ 128 KB. At 128K context, that's 16 GB just for the KV cache — equal to the model weights. At full MHA (32 KV-heads), it would be 64 GB. This is why GQA (§1.6) is critical for long-context serving, and why vLLM's PagedAttention (Kwon et al., 2023) manages KV cache like OS virtual memory pages, achieving 24× higher throughput by eliminating fragmentation.

def kv_cache_gb(batch, seq, n_layers, kv_heads, d_model, n_heads, bits=16):
    d_head = d_model // n_heads
    return batch * seq * n_layers * kv_heads * d_head * 2 * (bits//8) / 1e9

def model_gb(d, d_ff, L, H, kv, V, bits=16):

Flash Attention

Standard attention materialises the full N×N attention matrix in GPU HBM (high-bandwidth memory, the GPU's VRAM). For N=4,096 tokens and d=128, that's 4,096² × 4 bytes = 67 MB per head per layer. The bottleneck is not compute — it's memory bandwidth. Reading and writing this matrix to/from HBM dominates runtime. Dao et al. (2022, arXiv:2205.14135) observed that on-chip SRAM is 19 TB/s bandwidth vs 2 TB/s for HBM, but only ~50 MB total on an A100. Flash Attention tiles Q, K, V into SRAM blocks (typically 64–128 tokens per tile), computes attention within each tile, and never materialises the full N×N matrix.

Flash Attention is exact, not approximate — it produces bit-identical results to standard attention. The speedup is purely algorithmic: 2–4× end-to-end wall-clock reduction on GPT-2 training, 7.6× fewer HBM reads/writes per forward pass on A100. Flash Attention-2 (2023) reorganises the inner loop to better parallelise across GPU thread blocks, achieving another 2× speedup. Flash Attention-3 (2024) exploits H100 async tensor cores for further gains. The technique is now universal: PyTorch 2.0 uses FA-2 by default via F.scaled_dot_product_attention().

Online softmax is the algorithmic key. Computing attention requires the full row of dot-product scores to normalise with softmax. FA computes a running max and sum as it processes each tile, using the numerically stable identity: if new max m' > old max m, correct the accumulated sum by exp(m − m') before adding new terms. This allows exact softmax computation tile-by-tile with only O(N) memory instead of O(N²).

import math, random
random.seed(0)

def softmax_vec(v):
    mx = max(v)
    exps = [math.exp(x - mx) for x in v]
    s = sum(exps)
    return [e/s for e in exps]

def matmul(A, B):
    rA, cA, cB = len(A), len(A[0]), len(B[0])
    return [[sum(A[i][k]*B[k][j] for k in range(cA)) for j in range(cB)] for i in range(rA)]

def standard_attention(Q, K, V):
    T, d_k = len(Q), len(Q[0])
    scale = math.sqrt(d_k)
    Kt = [[K[j][i] for j in range(T)] for i in range(d_k)]
    S = matmul(Q, Kt)
    S = [[s/scale for s in row] for row in S]
    P = [softmax_vec(row) for row in S]
    return matmul(P, V)

def flash_attention_tiled(Q, K, V, B=2):
    # Flash-like: tiled computation, no full NxN materialised in memory
    T, d_k = len(Q), len(Q[0])
    d_v = len(V[0])
    scale = math.sqrt(d_k)
    O = [[0.0]*d_v for _ in range(T)]
    l = [0.0]*T       # running softmax denominator
    m = [-1e9]*T      # running max

    for j in range(0, T, B):         # tile over K/V
        Kj = K[j:j+B]
        Vj = V[j:j+B]
        for i in range(0, T, B):     # tile over Q
            Qi = Q[i:i+B]
            Sij = [[sum(Qi[a][k]*Kj[b][k] for k in range(d_k))/scale
                    for b in range(len(Kj))] for a in range(len(Qi))]
            for a in range(len(Qi)):
                row_max = max(Sij[a])
                m_new   = max(m[i+a], row_max)
                Pij_row = [math.exp(s - m_new) for s in Sij[a]]
                lij     = sum(Pij_row)
                # Correct accumulated O[i+a] for new max
                corr = math.exp(m[i+a] - m_new) if m[i+a] > -1e8 else 0.0
                l_new = corr * l[i+a] + lij
                for c in range(d_v):
                    O[i+a][c] = (corr * l[i+a] * O[i+a][c]
                                 + sum(Pij_row[b] * Vj[b][c] for b in range(len(Vj)))) / l_new
                m[i+a] = m_new
                l[i+a] = l_new
    return O

Alignment: RLHF, DPO & LoRA

Pre-training produces a model that predicts text; it does not produce a model that helpfully follows instructions. Bridging that gap — alignment — is the central engineering challenge of deploying LLMs. Three techniques dominate: Supervised Fine-Tuning (SFT), Reinforcement Learning from Human Feedback (RLHF), and the more recent Direct Preference Optimisation (DPO). Low-Rank Adaptation (LoRA) makes all three computationally feasible without retraining billions of parameters.

Stage 1 — SFT. A curated set of (prompt, ideal response) pairs is collected — InstructGPT used 13,000 prompts written by 40 contractors (Ouyang et al., 2022, arXiv:2203.02155). The pre-trained model is fine-tuned with standard cross-entropy loss on these examples. SFT teaches the response format but cannot capture nuanced human preferences about tone, safety, or helpfulness.

Stage 2 — Reward Model Training. Contractors rank model outputs for the same prompt. A separate model Rφ(x, y) is trained to predict the human preference score. For a preferred response yw vs rejected response yl, the reward model loss is:

Reward model loss: ℒ(φ) = −E(x,yw,yl)~D[ log σ(Rφ(x,yw) − Rφ(x,yl)) ]

This is a Bradley-Terry preference model: the probability that humans prefer yw equals σ(Rw − Rl). Training pushes the preferred response's score above the rejected one's.

Stage 3 — PPO. The SFT model is fine-tuned with proximal policy optimisation to maximise the reward model's score, subject to a KL-divergence penalty that prevents the model drifting too far from the SFT base. The PPO update is:

RLHF objective: maxπ Ex~D, y~π[ Rφ(x,y) ] − β · KL[π(y|x) ∥ πSFT(y|x)]

β is typically 0.01–0.1. Without the KL penalty the model collapses to reward-hacking — e.g., producing very long outputs that the reward model was not trained to penalise.

DPO — a simpler alternative. Rafailov et al. (2023, arXiv:2305.18290) showed that the RLHF objective has a closed-form optimal policy. Rather than training a separate reward model + running PPO, DPO directly fine-tunes the LLM on preference pairs:

DPO loss: ℒDPO(θ) = −E(x,yw,yl)[ log σ( β log πθ(yw|x)/πref(yw|x) − β log πθ(yl|x)/πref(yl|x) ) ]

Interpretation: DPO increases the probability of preferred responses relative to the reference model, while decreasing rejected responses — all in a single supervised loss. No PPO, no separate reward model, 2× less VRAM at training time.

LoRA — efficient fine-tuning. Full fine-tuning of a 70B model requires ~280 GB of GPU VRAM for fp16 weights alone. Hu et al. (2021, arXiv:2106.09685) introduced Low-Rank Adaptation: instead of updating W directly, add a low-rank decomposition Δ W = BA where B ∈ ℝd×r, A ∈ ℝr×k, and r ≪ min(d,k). For a 4096×4096 attention projection with r=16, LoRA uses 4096×16 + 16×4096 = 131,072 parameters instead of 16,777,216 — a 128× reduction. With r=64 on a 7B model, LoRA reduces trainable parameters from 7B to ~4M: a 1,750× reduction. Only the A,B matrices are updated; W is frozen. At inference, Δ W = BA is merged into W — zero latency overhead.

# DPO loss — single preferred/rejected pair
def dpo_loss(log_pi_w, log_ref_w, log_pi_l, log_ref_l, beta=0.1):
    # ↑ beta controls how far policy can deviate from reference
    ratio_w = log_pi_w - log_ref_w   # ↑ log-ratio for preferred response
    ratio_l = log_pi_l - log_ref_l   # ↑ log-ratio for rejected response
    logit   = beta * (ratio_w - ratio_l)   # ↑ margin between preferred and rejected
    loss    = -math.log(1 / (1 + math.exp(-logit)))   # ↑ -log σ(logit)
    return loss

Scale & Emergent Abilities

The Kaplan scaling laws (Kaplan et al., 2020, arXiv:2001.08361) were the first systematic study of how loss decreases with scale. They found:

Kaplan power laws: L(N) ≈ (Nc/N)αN    L(D) ≈ (Dc/D)αD    L(C) ≈ (Cc/C)αC

Where N=parameters, D=tokens, C=compute (FLOPs). Exponents: αN≈0.076, αD≈0.095, αC≈0.050. The laws hold across 7 orders of magnitude — from 768-parameter models to 1.5B parameters. Chinchilla (2022) later revised the compute-optimal frontier.

Emergent abilities are capabilities that appear abruptly at a threshold scale and are near-zero below it. Wei et al. (2022, arXiv:2206.07682) catalogued ~137 tasks that show emergence across PaLM (540B), GPT-3 (175B), and other large models. Examples: 3-digit addition emerges at ~100B parameters; chain-of-thought reasoning emerges at ~60B; word-in-context disambiguation emerges at ~130B. The abruptness is partly a measurement artifact: on accuracy metrics, capability transitions look sudden; on continuous loss metrics they look smoother. But the phenomenon is real — some capabilities genuinely require sufficient representational capacity before they can be expressed.

Srivastava et al.'s BIG-Bench (2022, arXiv:2206.04615) benchmark of 204 tasks specifically targets abilities believed to be beyond current models. Of 150 tasks evaluated, 65 showed emergence patterns — flat near random performance at small scale, sudden jump at large scale. The jump often occurs at 1023–1024 training FLOPs, corresponding roughly to GPT-3 scale.

Mixture of Experts (MoE)

A standard ("dense") transformer activates all parameters for every token. A Mixture-of-Experts model sparsely routes each token to a small subset of specialised FFN networks ("experts"), keeping compute constant while multiplying total capacity. Shazeer et al. (2017, arXiv:1701.06538) introduced MoE to NLP; Fedus et al. (2022, arXiv:2101.03961) scaled it to the Switch Transformer at 1.6 trillion parameters — the largest language model published to date — using only the compute of a 7B dense model.

Architecture. In an MoE layer, the standard FFN sublayer is replaced by E expert networks (typically E=8 or E=64) plus a learned router. The router is a small linear layer that produces a probability distribution over experts. For each token, the top-K experts (typically K=1 or K=2) are selected and their outputs are linearly combined:

MoE forward pass (Top-1 routing):
h = Router(x) = Softmax(x Wr)     [E-dimensional]
i* = argmax(h)     [selected expert index]
output = h[i*] · FFNi*(x)     [weighted expert output]

Mistral's Mixtral-8x7B (Jiang et al., 2024, arXiv:2401.04088) uses K=2 of E=8 experts per token. Total parameters: 46.7B. Active parameters per token: only 12.9B — matching a ~13B dense model's compute at 3.6× the capacity.

Load balancing. Without regularisation, the router collapses: one expert receives all tokens (routing collapse). The auxiliary load-balancing loss encourages uniform expert utilisation:

Auxiliary loss (Switch Transformer): ℒaux = α · E · ∑i fi · Pi

fi = fraction of tokens routed to expert i (discrete, non-differentiable)
Pi = average router probability for expert i (differentiable)
The product fi·Pi is differentiable w.r.t. router weights and minimized when all experts receive equal load.

Expert capacity. In distributed training, each expert is placed on a different device. A "capacity factor" C limits how many tokens can be processed per expert per batch — overflow tokens are "dropped" (passed through as residual unchanged). C=1.0 means no slack; C=1.25 is typical. Dropped tokens are a training inefficiency but acceptable at scale.

Real-world MoE models in production: Mixtral-8x7B (Mistral, 2024) — open-weight, outperforms LLaMA-2 70B at ~13B active params. GPT-4 — rumored 8 experts ×220B (1.76T total), though not officially confirmed. Gemini 1.5 Pro (Google, 2024, arXiv:2403.05530) — MoE with reported 1M-token context.

# Simplified MoE routing: softmax router, Top-K selection
def softmax(x):
    m = max(x)
    e = [math.exp(v - m) for v in x]
    s = sum(e)
    return [v/s for v in e]

def top_k_routing(token_vec, router_weights, k=2):
    # ↑ router_weights: (E, d) matrix; E experts, d-dim tokens
    E = len(router_weights)
    logits = [sum(token_vec[j] * router_weights[i][j]
               for j in range(len(token_vec)))
              for i in range(E)]   # ↑ dot product of token with each expert gate
    probs = softmax(logits)
    ranked = sorted(enumerate(probs), key=lambda x: -x[1])
    selected = ranked[:k]    # ↑ top-K experts selected
    total = sum(p for _, p in selected)
    weights = [(i, p/total) for i, p in selected]   # ↑ renormalize weights
    return weights

Quantisation & Deployment Efficiency

A full-precision (float32) parameter occupies 4 bytes; float16/bfloat16 cuts this to 2 bytes; int8 to 1 byte; int4 to 0.5 bytes. LLaMA-3 70B in float16 requires ~140 GB of GPU VRAM — four A100-40GB cards with nothing to spare. In 4-bit quantisation (GGUF, GPTQ, AWQ), the same model fits in ~35 GB — a single A100 or two 24 GB consumer GPUs. The quality cost is surprisingly small: perplexity typically degrades <1% at int8, <2–3% at int4 with modern algorithms.

Post-Training Quantisation (PTQ) converts a trained float16 model without re-training. Dettmers et al. (2022, arXiv:2208.07339) introduced LLM.int8(), using vector-wise quantisation with mixed-precision decomposition for emergent outlier features. Outlier features (a small fraction of dimensions with magnitudes 60–100× larger than average) blow up INT8 quantisation — LLM.int8() detects these dimensions and keeps them in float16 while quantising the rest to int8.

NF4 — Normal Float 4-bit. Dettmers et al. (2023, arXiv:2305.14314, QLoRA paper) introduced NF4, an information-theoretically optimal 4-bit data type for normally-distributed weights. Standard int4 has uniformly-spaced bins; NF4 places bins so that each quantile of a standard normal distribution occupies equal probability mass. For weights that are (approximately) normally distributed, NF4 minimises the expected quantisation error. QLoRA = NF4 quantisation + LoRA fine-tuning + double quantisation of the quantisation constants = fine-tuning a 65B model on a single 48 GB A40.

NF4 quantile levels — 16 values covering the standard normal distribution's quantiles (each covers 1/16 of the probability mass). Contrast with uniform int4 [−8,−7,...,+7]: NF4 clusters values near zero (where most weights lie) and spreads more near tails.

Dequantisation: each 4-bit code maps to its NF4 centroid, scaled by the block's absmax value. Block size = 64 (64 weights share one absmax constant). Double quantisation: the absmax constants themselves are further quantised to 8-bit, saving ~0.5 bits/parameter.

GGUF & llama.cpp. For CPU/GPU inference on consumer hardware, llama.cpp (Gerganov, 2023) implements its own quantisation formats: Q4_K_M means 4-bit, K-quant (groups of 256 with enhanced precision on important weights), M=medium (some layers kept at 6-bit). A llama.cpp Q4_K_M of LLaMA-3 70B fits in ~42 GB; Q5_K_M fits in ~52 GB with meaningfully better quality. Inference speed on an M2 MacBook Pro: ~8 tokens/second for 8B, ~1–2 t/s for 70B.

AWQ (Activation-Aware Weight Quantisation). Lin et al. (2023, arXiv:2306.00978) observed that not all weights matter equally — 0.1% of channels account for most quantisation error because input activations are large at those channels. AWQ protects these salient channels via per-channel scaling before quantisation, achieving better quality than GPTQ at similar speed.

The 2025–26 Landscape & Open Problems

As of mid-2025, the frontier has shifted from raw capability to efficiency, reasoning, and multimodality. The GPT-4 release (March 2023) established a new capability ceiling; subsequent frontier models — Claude 3 Opus, Gemini 1.5 Pro, Llama 3.1 405B — have approached or matched it on most benchmarks at lower cost. The open-weight ecosystem has compressed the capability gap dramatically: Llama 3.1 405B (released July 2024) performs on par with GPT-4-class models while being freely downloadable.

Context length. The practical context window has expanded from 4K tokens (GPT-3) to 128K (GPT-4 Turbo), 1M (Gemini 1.5 Pro), and 2M in research settings. Key enabling techniques: Ring Attention (Liu et al., 2023, distributes attention across devices), YaRN (Peng et al., 2023, RoPE scaling for extrapolation), and sliding window attention (Mistral). The bottleneck is now KV cache memory, not model architecture.

Reasoning models. OpenAI's o1/o3, Google's Gemini 2.0 Flash Thinking, and DeepSeek-R1 (January 2025, arXiv:2501.12948) implement chain-of-thought reasoning at inference time — generating and evaluating multiple solution paths before returning an answer. DeepSeek-R1 achieved GPT-4o-level math and coding performance through pure reinforcement learning from outcome feedback (no human preference data), a landmark demonstrating that RLHF is not necessary for reasoning alignment.

Multimodality. GPT-4V, Gemini 1.5 Pro, Claude 3.5 Sonnet, and Llama 3.2 90B Vision accept images alongside text. The dominant architecture is a ViT (Vision Transformer) encoder whose patch embeddings are projected into the LLM's residual stream — the same architecture as CLIP (Radford et al., 2021). Native image generation within a unified model (not a separate diffusion model) remains an active research area (Chameleon, Transfusion, 2024).

Open problems at the 2025 frontier:

Hallucination & Calibration
LLMs generate plausible-sounding falsehoods with high confidence. RAG, grounding via tool use, and constitutional AI partially mitigate but don't solve. Factuality remains the #1 deployment blocker for high-stakes applications (medicine, law).
Long-Context Faithfulness
"Lost in the middle" (Liu et al., 2023): LLMs reliably use information at the start and end of a 128K context but systematically miss information buried in the middle. This is an attention sink problem, not a context-length problem per se.
Sample Efficiency
Chinchilla-optimal training requires 20 tokens/parameter. GPT-3 175B needed 300B tokens; LLaMA-3 405B used 15T tokens. Humans acquire language from ~100M words (≈ 1B tokens) over childhood. Current pre-training is 10,000× less sample-efficient than biological learning.
Test-Time Compute Scaling
DeepSeek-R1 and OpenAI-o3 show that spending more inference compute (extended chain-of-thought, self-consistency) improves performance on hard tasks. A second scaling law axis beyond training compute. Optimal allocation of training vs inference budget is an open research question.

import math

# --- Training compute budget breakdown ---
# Rule of thumb: C ≈ 6 * N * D FLOPs (Kaplan 2020)
# Factor of 6 = 2 (forward) + 4 (backward, ~2x forward)
def training_flops(N_params, D_tokens):
    return 6 * N_params * D_tokens

# --- Inference compute per token ---
# ~2 * N FLOPs per token (matrix-vector products dominate)
def inference_flops_per_token(N_params):
    return 2 * N_params

print("Training compute (in FLOPs) for selected models:")
models = [
    ("GPT-3 175B", 175e9, 300e9),
