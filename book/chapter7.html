7.1

From Pretraining to Instruction Following

A pretrained language model is, at its core, a completion engine. Feed it the prefix "The capital of France is" and it will confidently predict "Paris." Feed it the opening of a Hemingway novel and it will continue in Hemingway's voice. The model has absorbed a staggering amount of world knowledge — facts, reasoning patterns, stylistic registers, common sense — but it has learned exactly one skill: predict the next token given the preceding context. That is not the same as following instructions.

The gap becomes obvious the moment you ask a raw pretrained model a direct question. Ask "What is the boiling point of water?" and it will often respond by generating more questions in the same format, because that is what Q&A pages on the internet look like: one question followed by another. The model has no concept of "I am the assistant, you are the user, my job is to answer." That role structure, that turn-taking convention, that helpful disposition — none of it exists in the pretrained weights. It has to be taught, and the process of teaching it is called instruction tuning or supervised fine-tuning (SFT).

The history of this insight unfolded in three distinct eras. The first era (GPT-1, GPT-2) showed that pretraining on web text yielded surprisingly capable zero-shot performance on downstream tasks — but the models were unreliable, inconsistent, and prone to ignoring what you actually asked. The second era began with the FLAN paper (Wei et al., 2021), which showed that fine-tuning on a mixture of tasks described in natural language dramatically improved instruction following and generalization to unseen tasks. The third era arrived with InstructGPT (Ouyang et al., 2022) and ChatGPT: combine SFT with human preference learning (RLHF), and you get a model that is not only helpful but actively aligned with human expectations about tone, safety, and format. The open-source ecosystem — LLaMA, Mistral, Falcon, Qwen — has since replicated and extended these techniques, making instruction tuning accessible to anyone with a GPU and a dataset.

What makes SFT work? The pretrained model already knows how to write coherent prose, reason about facts, and handle edge cases. SFT does not teach it new knowledge — it teaches it a new mode of operation. By showing the model thousands of examples where a system prompt, a user message, and a well-formed assistant response co-occur, the model learns to condition its generation on the instruction-following frame. It learns that when it sees a certain token pattern (e.g., <|im_start|>user) it should generate a response to whatever follows, not a continuation of the same pattern. The knowledge is already there; SFT is the key that unlocks it in the right direction.

Key insight: Pretraining gives the model everything it knows. SFT gives it a job description. The model's capabilities are largely fixed after pretraining — SFT reshapes how those capabilities are expressed and when they activate.

7.2

SFT Dataset Formats

Before you can fine-tune a model, you need data in the right shape. The field has converged on three dominant formats, each representing a different philosophy about what the training signal should look like. Understanding the tradeoffs between them is not academic: the wrong format choice can cause the model to learn the template structure but not the content, or to confuse user turns with assistant turns, or to fail entirely on multi-turn conversations.

The Alpaca format emerged from Stanford's Alpaca project (2023), which used GPT-3 to generate 52,000 instruction-following examples using a "self-instruct" procedure. It is simple: a JSON object with three fields — instruction, input, and output. The input field is optional context (a passage to summarize, code to debug), and may be empty for pure instruction-following tasks. Alpaca is excellent for curating domain-specific fine-tunes because the format is human-readable and easy to write by hand. Its limitation is that it is inherently single-turn: there is no mechanism for conversation history.

The ShareGPT format captures multi-turn conversations as an alternating sequence of human and assistant turns. Each example is a JSON object with a conversations list, where each entry has a from field ("human" or "gpt") and a value field containing the text. This format was popularized by the ShareGPT dataset — real conversations scraped from ChatGPT users with their permission — and it enables the model to learn how to maintain context across turns, refer back to earlier parts of the conversation, and handle follow-up questions naturally. The tradeoff is that curation is harder: a bad turn anywhere in the sequence contaminates the example.

The ChatML format, developed by OpenAI and adopted by most modern open-source models including Mistral, Llama-3, and Qwen, uses special tokens to delineate roles at the token level. Rather than relying on string matching, the tokenizer has dedicated vocabulary entries for role boundaries. This makes the format unambiguous even when the content itself contains text that looks like role markers. ChatML is the industry standard for production fine-tuning today. Llama-3 uses a slightly different flavor with <|begin_of_text|> and <|eot_id|> tokens, but the principle is identical.

Why format matters: The model learns the template structure as much as the content. If your dataset has inconsistent formatting — some examples using "### Instruction:" and others using "<|im_start|>" — the model will learn to output both interchangeably, which is not what you want at inference time.
Python — Generate a random Alpaca-format training example 
import random, json

INSTRUCTIONS = [
    "Explain {topic} to a 10-year-old.",
    "Write a Python function that {task}.",
    "Summarize the key idea behind {topic} in two sentences.",
    "What are the three most important things to know about {topic}?",
    "Compare and contrast {topic} and {other}.",
]
TOPICS = ["gradient descent", "attention mechanisms", "tokenization",
          "regularization", "backpropagation", "embeddings", "LoRA"]
TASKS  = ["computes cosine similarity between two vectors",
          "flattens a nested list", "implements a simple tokenizer"]

def make_alpaca_example():
    instr_tmpl = random.choice(INSTRUCTIONS)
    topic = random.choice(TOPICS)
    other = random.choice([t for t in TOPICS if t != topic])
    task  = random.choice(TASKS)
    instruction = (instr_tmpl
        .replace("{topic}", topic)
        .replace("{other}", other)
        .replace("{task}", task))
    example = {
        "instruction": instruction,
        "input": "",
        "output": f"[Model would generate a response about {topic!r} here]"
    }
    return example

for i in range(3):
    ex = make_alpaca_example()
    print(f"--- Example {i+1} ---")
    print(json.dumps(ex, indent=2))
    print()

7.3

Chat Templates and Masked Loss

Fine-tuning a model on instruction data requires two things that are easy to get wrong: a consistent way to serialize the conversation into a flat token sequence (the chat template), and a training objective that only punishes the model for mistakes on the tokens it is supposed to generate (masked loss). Getting either wrong is enough to produce a model that either ignores instructions or degrades into incoherent output after a few turns.

The chat template defines how roles, messages, and special tokens are arranged in the input sequence. In the ChatML convention used by Mistral and OpenHermes, a conversation is serialized as: <|im_start|>system\n{system_prompt}\n<|im_end|>\n<|im_start|>user\n{user_message}\n<|im_end|>\n<|im_start|>assistant\n. The model generates tokens starting after the final assistant\n until it emits <|im_end|>, at which point generation stops. Llama-3 uses a different set of special tokens — <|begin_of_text|>, <|start_header_id|>, <|end_header_id|>, <|eot_id|> — but the concept is identical. HuggingFace's tokenizer.apply_chat_template() handles this serialization automatically once you supply the correct Jinja2 template for your model family.

Masked loss is the mechanism that makes SFT actually work as instruction tuning rather than general text modeling. The cross-entropy loss is computed over the entire token sequence, but only certain positions contribute gradients: the assistant turns. System prompts and user messages are masked out by setting their label values to -100 — a sentinel value that PyTorch's nn.CrossEntropyLoss silently ignores. Without this masking, the model receives gradient signal from every token in the sequence, which teaches it to predict user messages given prior user messages. This is precisely the wrong objective: you end up with a model that completes user queries rather than responding to them. The masking forces the model to learn: "Given everything before my turn, what should I say?"

chat_template.py 
from transformers import AutoTokenizer

# Load a tokenizer that knows the ChatML template
tokenizer = AutoTokenizer.from_pretrained("mistralai/Mistral-7B-Instruct-v0.2")

messages = [
    {"role": "system",    "content": "You are a helpful assistant."},
    {"role": "user",      "content": "What is backpropagation?"},
    {"role": "assistant", "content": "Backpropagation is the algorithm..."},
]

# Serialize to a flat token sequence — handles special tokens automatically
flat = tokenizer.apply_chat_template(
    messages,
    tokenize=False,       # ↑ return string, not token IDs — easier to inspect
    add_generation_prompt=False
)
print(flat)

# ------ Masked loss: only train on assistant tokens ------
IGNORE_INDEX = -100  # ↑ PyTorch CrossEntropyLoss ignores this sentinel value

def mask_labels(input_ids, tokenizer):
    """Return labels tensor: -100 everywhere except assistant response tokens."""
    labels = list(input_ids)  # ↑ start with a copy of all token IDs

    # Find the assistant start token position
    assistant_token = tokenizer.encode("<|im_start|>assistant", add_special_tokens=False)

    in_assistant = False
    i = 0
    while i < len(labels):
        # Check if we're entering an assistant span
        span = labels[i:i+len(assistant_token)]
        if span == assistant_token:
            in_assistant = True
            labels[i:i+len(assistant_token)] = [IGNORE_INDEX]*len(assistant_token)
            i += len(assistant_token)
            continue
        # Check if we're leaving an assistant span
        if in_assistant and labels[i] == tokenizer.convert_tokens_to_ids("<|im_end|>"):
            in_assistant = False
        # Mask non-assistant tokens
        if not in_assistant:
            labels[i] = IGNORE_INDEX  # ↑ gradient will be zero at this position
        i += 1

    return labels
Without masking: The model trains to predict user messages given prior context — essentially a general next-token predictor. With masking, it trains to generate responses given full conversational context. This single change is the difference between a completion model and an instruction-following model.
Python — Implement masked cross-entropy loss on a toy sequence 
import math

def softmax(logits):
    m = max(logits)
    exps = [math.exp(x - m) for x in logits]
    s = sum(exps)
    return [e / s for e in exps]

def cross_entropy(logits, target):
    """Standard cross-entropy: -log(prob[target])"""
    probs = softmax(logits)
    return -math.log(probs[target] + 1e-9)

IGNORE_INDEX = -100

# Toy sequence: 6 positions, vocab size 5
# Positions 0-1 are system/user (masked), positions 2-5 are assistant (trained)
logits = [
    [2.1, 0.5, 0.3, 0.1, 0.0],  # pos 0: system token    (masked)
    [0.1, 3.2, 0.2, 0.1, 0.1],  # pos 1: user token      (masked)
    [0.2, 0.1, 4.1, 0.3, 0.2],  # pos 2: assistant tok 1 (trained)
    [0.1, 0.2, 0.1, 3.8, 0.1],  # pos 3: assistant tok 2 (trained)
    [0.3, 0.1, 0.2, 0.1, 2.9],  # pos 4: assistant tok 3 (trained)
    [1.1, 0.2, 3.1, 0.1, 0.1],  # pos 5: EOS token       (trained)
]
targets = [IGNORE_INDEX, IGNORE_INDEX, 2, 3, 4, 2]  # -100 = masked

print("Position  Label     Loss")
print("-" * 40)
total_loss = 0.0
num_active = 0
for i, (lg, tgt) in enumerate(zip(logits, targets)):
    if tgt == IGNORE_INDEX:
        print(f"  {i}       MASKED    ---  (no gradient)")
    else:
        loss = cross_entropy(lg, tgt)
        total_loss += loss
        num_active += 1
        print(f"  {i}       {tgt}         {loss:.4f}")

avg_loss = total_loss / num_active
print(f"\nAverage loss (active positions only): {avg_loss:.4f}")
print(f"Without masking, loss would include {len(logits)} positions.")
print(f"With masking, only {num_active} positions contribute gradients.")

7.4

LoRA in Depth

Fine-tuning all the parameters of a large language model is prohibitively expensive. A 7B-parameter model in 16-bit precision requires 14 GB just to store the weights; optimizer states (Adam maintains two moments per parameter) add another 28 GB, and activations during backpropagation can push the total well above 80 GB. This puts full fine-tuning out of reach for anyone without access to multiple high-end GPUs. Low-Rank Adaptation (LoRA), introduced by Hu et al. in 2021, offers an elegant solution: instead of updating the full weight matrix W, learn a low-rank decomposition of the update.

The key observation is that when you fine-tune a pretrained model on a downstream task, the weight changes tend to be low-rank — they live in a small subspace of the full parameter space. LoRA exploits this by parameterizing the weight update as the product of two small matrices: A of shape (d, r) and B of shape (r, k), where r << min(d, k) is the rank. The forward pass becomes h = Wx + (B · A · x) · (α/r), where W is frozen and only A and B are updated during training. At initialization, A is drawn from a Gaussian distribution and B is zero, so the adapter contributes nothing at the start of training — you begin exactly where the pretrained model left off.

The parameter savings are dramatic. Consider the query projection in a GPT-2 attention head: d = k = 768, so W has 768 × 768 = 589,824 parameters. With LoRA at rank 8, A has 768 × 8 = 6,144 parameters and B has 8 × 768 = 6,144 parameters, for a total of 12,288 — just 2.1% of the original. Across all attention layers in a 7B model, LoRA at r=16 typically introduces 4–8 million trainable parameters out of 7 billion, reducing optimizer memory by 99% and enabling fine-tuning on a single 24 GB GPU.

Which layers should you adapt? The standard practice is to apply LoRA to the query and value projection matrices (Wq and Wv) in every attention head. Some practitioners also adapt the key matrix Wk and the output projection Wo. Adapting the feed-forward network (FFN) layers adds more parameters but can improve performance on knowledge-intensive tasks. What you should almost never adapt are the embedding layers: they are large (vocab_size × d_model), sparse in their gradients, and slow to train. The practical recommendation for most fine-tunes is Wq + Wv with r=16 and α=32 as a starting point.

After training, LoRA adapters can be merged back into the base weights with zero inference overhead: W_merged = W0 + (α/r) · B · A. The merged model is identical in architecture to the original — it can be served by any standard runtime without adapter-aware infrastructure. This merge-at-inference property is one of LoRA's most practical advantages over other PEFT methods like prefix tuning, which require permanent changes to the inference pipeline.

lora_adapter.py 
import math

class LoRALinear:
    """LoRA adapter wrapping a frozen linear layer (no PyTorch for clarity)."""

    def __init__(self, d_in, d_out, rank=8, alpha=16):
        self.d_in  = d_in     # ↑ input dimension (e.g. 768 for GPT-2)
        self.d_out = d_out    # ↑ output dimension
        self.rank  = rank     # ↑ r — the bottleneck rank, controls param count
        self.scale = alpha / rank  # ↑ α/r — prevents large jumps at high rank

        # A: Gaussian init — starts contributing to the forward pass immediately
        std = 1 / math.sqrt(rank)
        self.A = [[_gauss(std) for _ in range(d_out)]
                   for _ in range(rank)]

        # B: zero init — adapter outputs zero at training start (stable baseline)
        self.B = [[0.0 for _ in range(rank)]
                   for _ in range(d_in)]

    def adapter_params(self):
        """Count trainable parameters in this LoRA adapter."""
        a_params = self.rank * self.d_out   # ↑ A matrix
        b_params = self.d_in * self.rank    # ↑ B matrix
        return a_params + b_params

    def base_params(self):
        """Count frozen parameters in the original weight matrix."""
        return self.d_in * self.d_out        # ↑ W0 is never updated

import random
def _gauss(std): return random.gauss(0, std)

7.5

QLoRA: Quantized Fine-Tuning

LoRA reduces the number of trainable parameters dramatically, but the frozen base model still needs to be held in memory during training. A 7B model in 16-bit floating point requires 14 GB of VRAM just for the weights, plus activations and optimizer states for the LoRA parameters. QLoRA, introduced by Dettmers et al. in 2023, pushes this boundary further by quantizing the frozen base model to 4 bits while keeping the LoRA adapters in 16-bit precision. The result is remarkable: a 65B model fine-tune fits on a single 48 GB A6000 GPU, and a 7B fine-tune fits on a 10 GB consumer GPU.

The 4-bit format used by QLoRA is called NF4 (Normal Float 4). Unlike uniform 4-bit quantization, which divides the weight range into 16 equally spaced bins, NF4 places its 16 quantization levels non-uniformly — they are spaced to match the normal distribution that neural network weights approximately follow. The densest part of the quantization grid is in the center of the distribution, where most weights live, and the levels spread out toward the tails. This information-theoretically optimal placement means NF4 loses less information than uniform 4-bit for the same number of bits.

QLoRA also introduces double quantization, which quantizes the quantization constants themselves. Standard quantization stores a scale and zero-point per block of 64 weights (32-bit floats). Double quantization treats these constants as another set of values to be quantized, reducing their storage cost from 32 bits to 8 bits and saving an additional 0.37 bits per parameter — roughly 3 GB across a 65B model. Finally, paged optimizers handle the case where GPU memory is temporarily exhausted by gradient checkpointing: the CUDA unified memory system automatically pages optimizer states to CPU RAM and back, preventing out-of-memory crashes at the cost of some latency.

Practical rule of thumb: With QLoRA, divide the model's parameter count by 2 to get the approximate VRAM requirement in GB. A 7B model needs ~4–6 GB, a 13B model needs ~8–10 GB, a 70B model needs ~40–48 GB. Full fine-tuning needs roughly 20× the model size in VRAM.
Python — Compute VRAM requirements for different training methods 
def vram_gb(params_b, method, lora_rank=16):
    """Estimate VRAM requirement in GB for a given model size and method."""
    params = params_b * 1e9

    if method == "full_fp16":
        weights   = params * 2        # 2 bytes per param (fp16)
        gradients = params * 2        # same size as weights
        adam      = params * 2 * 4   # Adam: m + v, each fp32 (4 bytes)
        activations = params * 1.5   # rough estimate
        return (weights + gradients + adam + activations) / 1e9

    elif method == "lora_fp16":
        lora_params = 2 * 4096 * lora_rank * 32 * 2  # 32 layers, Q+V
        weights   = params * 2          # frozen base in fp16
        lora_adam = lora_params * 4 * 2 # Adam states on lora params only
        return (weights + lora_adam) / 1e9

    elif method == "qlora_nf4":
        lora_params = 2 * 4096 * lora_rank * 32 * 2
        weights   = params * 0.5       # 4-bit = 0.5 bytes per param
        lora_adam = lora_params * 4 * 2
        overhead  = params * 0.05      # paged optimizer overhead
        return (weights + lora_adam + overhead) / 1e9

7.6

Direct Preference Optimization (DPO)

Supervised fine-tuning teaches the model to follow instructions, but it does not teach it to prefer good responses over mediocre ones. A model trained only on SFT will generate plausible-sounding text, but it has no mechanism to learn that one response is more helpful, more honest, or less harmful than another. Reinforcement Learning from Human Feedback (RLHF) addresses this by training a reward model on human preference judgments and then optimizing the language model to maximize reward using policy gradient methods. The approach works — it produced ChatGPT — but it is notoriously complex, requiring four models (reference, reward, policy, critic), careful hyperparameter tuning for PPO, and significant infrastructure overhead.

Direct Preference Optimization (DPO), introduced by Rafailov et al. in 2023, reformulates the RLHF objective in a way that eliminates the reward model entirely. The key insight is mathematical: the optimal language model policy under the RLHF objective has a closed-form relationship to the reward function, which means you can express the reward implicitly in terms of the policy and the reference model. Substituting this expression into the Bradley-Terry model for human preferences gives a classification loss that can be minimized directly on a dataset of (prompt, chosen_response, rejected_response) triples — no RL, no reward model, no PPO.

The DPO loss has an intuitive interpretation: it increases the log-probability of chosen responses relative to the reference model and decreases the log-probability of rejected responses relative to the reference model, where the β parameter controls how aggressively to deviate from the reference. A small β (e.g., 0.1) allows large divergence from the reference policy — the model can change significantly — while a large β keeps the aligned model close to the SFT checkpoint. Too small and you get mode collapse; too large and the preference signal is barely absorbed. In practice, β = 0.1 is a good starting point for models fine-tuned with SFT first.

DPO LOSS LDPO(πθ) = − E(x, yw, yl) [log σ( β · log(πθ(yw|x) / πref(yw|x)) − β · log(πθ(yl|x) / πref(yl|x)) )] σ = sigmoid. yw = chosen response. yl = rejected response. β controls KL penalty strength (typically 0.1). πref = frozen SFT checkpoint.
DPO in one sentence: Maximize the gap between how much more likely the model is to generate the chosen response over the rejected response, relative to a frozen reference model, measured in log-probability space.
Python — Implement DPO loss on toy log-probabilities 
import math

def sigmoid(x):
    return 1.0 / (1.0 + math.exp(-x))

def dpo_loss(log_pi_chosen, log_pi_ref_chosen,
             log_pi_reject, log_pi_ref_reject, beta=0.1):
    """
    Compute DPO loss for a single (prompt, chosen, rejected) triple.
    All inputs are log-probabilities (sum over tokens).
    """
    # Implicit reward for chosen: how much more likely vs reference?
    reward_chosen = beta * (log_pi_chosen - log_pi_ref_chosen)
    # ↑ positive = model prefers chosen MORE than reference did (good)

    # Implicit reward for rejected
    reward_reject = beta * (log_pi_reject - log_pi_ref_reject)
    # ↑ negative = model prefers rejected LESS than reference did (good)

    # Loss = -log(sigmoid(reward_chosen - reward_reject))
    margin = reward_chosen - reward_reject
    loss   = -math.log(sigmoid(margin) + 1e-9)
    return loss, reward_chosen, reward_reject, margin

RLHF vs DPO

The choice between RLHF and DPO is not purely academic — it affects your infrastructure requirements, the stability of training, the quality of the resulting model, and how much expertise you need on your team. Both approaches share the same goal (align a language model to human preferences using paired data) and both use a reference model to define a KL constraint that prevents the policy from drifting too far from the SFT checkpoint. The difference is in how they get there.

RLHF, as implemented in InstructGPT and LLaMA-2-Chat, requires four distinct model instances: the reference model (frozen SFT checkpoint), the reward model (trained from human comparisons), the policy model (the model being aligned), and, in PPO-style training, a critic model that estimates the value function. Training requires coordinating all four simultaneously, which means the GPU memory footprint is 4× a single model, plus the overhead of RL optimization (which introduces hyperparameters like the PPO clip ratio, the advantage estimation coefficient, and the KL penalty weight that interact in non-obvious ways). Reward hacking is a persistent risk: the policy can find inputs that fool the reward model into assigning high scores without being genuinely helpful. OpenAI's experience building ChatGPT required significant engineering to mitigate these failure modes.

DPO collapses this to two models: the frozen reference model and the policy being trained. The loss is a simple binary cross-entropy that can be implemented in a standard supervised training loop. Training is stable, reproducible, and compatible with standard gradient accumulation, mixed precision, and LoRA. The practical results are compelling: Zephyr-7B (trained with DPO on top of Mistral-7B SFT) outperformed LLaMA-2-Chat-70B on MT-Bench despite being 10× smaller. The field has largely converged on DPO for open-source instruction tuning, with RLHF reserved for cases where you need fine-grained reward control (e.g., safety-critical applications) or have the infrastructure to run it reliably.

Python — Implement KL divergence (used in both RLHF and DPO) 
import math

def kl_divergence(p, q):
    """
    KL(p || q) — how much q diverges from p, in nats.
    Used as the penalty term in both RLHF and DPO to keep
    the aligned policy close to the reference distribution.
    """
    total = 0.0
    for pi, qi in zip(p, q):
        if pi > 1e-10:
            total += pi * math.log(pi / (qi + 1e-10))
            # ↑ pi * log(pi/qi): each token's contribution to divergence
    return total

def normalize(probs):
    s = sum(probs)
    return [x / s for x in probs]

# Toy 6-token vocabulary
# Reference model (SFT checkpoint) distribution over next token
ref   = normalize([0.3, 0.25, 0.2, 0.15, 0.07, 0.03])

# Slightly aligned policy — prefers token 0 more, token 3 less
mild  = normalize([0.4, 0.25, 0.2, 0.08, 0.05, 0.02])

# Aggressively aligned policy — mode collapse toward token 0
aggro = normalize([0.85, 0.08, 0.04, 0.01, 0.01, 0.01])

print("Policy comparison vs. reference:")
print(f"  Mild alignment:       KL = {kl_divergence(mild,  ref):.4f} nats")
print(f"  Aggressive alignment: KL = {kl_divergence(aggro, ref):.4f} nats")
print(f"  (reference vs itself: KL = {kl_divergence(ref,   ref):.4f} nats)")
print()
print("RLHF KL penalty: reward -= beta * KL(policy || reference)")
print("DPO implicit KL: controlled via beta hyperparameter in loss")
print()

# Demonstrate how beta scales the KL penalty
reward_base = 2.5  # reward model score
print(f"{'Beta':>6}  {'KL Penalty':>12}  {'Net Reward (mild)':>20}")

for beta in [0.0, 0.05, 0.1, 0.2, 0.5, 1.0]:

Evaluation: MMLU, MT-Bench, Elo

Building a fine-tuned model without a principled evaluation strategy is like calibrating a scale by checking whether it feels right. The field has developed three complementary evaluation paradigms that together give a reasonably complete picture of a model's capabilities: MMLU for knowledge breadth, MT-Bench for instruction-following quality, and Chatbot Arena Elo for human preference at scale. Each captures something the others miss, and each has well-documented failure modes that practitioners should understand before trusting the numbers.

MMLU (Massive Multitask Language Understanding, Hendrycks et al. 2021) consists of 14,042 multiple-choice questions spanning 57 subjects, from elementary mathematics and US history to professional medicine and international law. Models are evaluated in a 5-shot setting: five example questions are provided in the context before the test question, and the model must select the correct letter (A/B/C/D). MMLU is excellent for measuring factual knowledge and reasoning across domains, and it scales well — you can run it without GPT-4 as a judge. Its limitation is saturation: GPT-4 scores 86.4%, and the gap between models is increasingly compressed at the top. Small improvements in MMLU do not reliably predict differences in user experience.

MT-Bench (Zheng et al. 2023) addresses MMLU's blind spot by testing instruction following and multi-turn reasoning. It consists of 80 carefully crafted questions across 8 categories (writing, roleplay, extraction, reasoning, math, coding, STEM, humanities), each followed by a mandatory follow-up question that requires the model to build on its own previous answer. Each response is graded by GPT-4 on a 1–10 scale. This LLM-as-judge paradigm is powerful — it can evaluate open-ended responses that multiple-choice benchmarks cannot — but it introduces a circularity: a model trained to be similar to GPT-4 will tend to score higher. Chatbot Arena Elo sidesteps this by using human judges in anonymous head-to-head comparisons, applying the Bradley-Terry model to compute Elo-style ratings. It is the ground truth that practitioners trust most, but it requires thousands of evaluations to converge and cannot be run cheaply on every checkpoint.

Python — Implement Elo rating with Bradley-Terry model 
import random, math

def expected_score(rating_a, rating_b):
    """Bradley-Terry: probability model A wins given ratings."""
    return 1.0 / (1.0 + 10 ** ((rating_b - rating_a) / 400))

def update_elo(rating_a, rating_b, result_a, k=32):
    """Update Elo ratings. result_a=1 if A wins, 0 if B wins, 0.5 if tie."""
    exp_a  = expected_score(rating_a, rating_b)
    exp_b  = 1 - exp_a
    new_a  = rating_a + k * (result_a     - exp_a)
    new_b  = rating_b + k * ((1-result_a) - exp_b)
    return new_a, new_b

# Simulate Chatbot Arena: 3 models, 150 head-to-head comparisons
random.seed(42)
models = {
    "GPT-4":           {"rating": 1200, "true_strength": 0.85},
    "Zephyr-7B (DPO)": {"rating": 1200, "true_strength": 0.62},
    "LLaMA-2-7B":      {"rating": 1200, "true_strength": 0.45},
}
names = list(models.keys())

for _ in range(150):
    a, b = random.sample(names, 2)
    # Probability A wins is proportional to true_strength difference
    sa = models[a]["true_strength"]
    sb = models[b]["true_strength"]
    prob_a_wins = sa / (sa + sb)
    result_a = 1 if random.random() < prob_a_wins else 0
    ra, rb = update_elo(models[a]["rating"], models[b]["rating"], result_a)
    models[a]["rating"] = ra
    models[b]["rating"] = rb

print("Arena Elo Ratings after 150 matchups:")
print("-" * 45)
ranked = sorted(models.items(), key=lambda x: -x[1]["rating"])
for rank, (name, info) in enumerate(ranked, 1):
    bar = "#" * int((info["rating"] - 1050) / 15)
    print(f"  #{rank} {name:<22} {info['rating']:>7.1f}  {bar}")

Deploying a Local Assistant

Once you have a fine-tuned model — whether merged weights or a base model plus LoRA adapter — you need a runtime that can serve it efficiently. Production deployments at scale use vLLM or Text Generation Inference (TGI) with PagedAttention and continuous batching, but for local deployment, personal projects, or development environments, two tools dominate: llama.cpp and Ollama. Both are purpose-built for running quantized models on consumer hardware with minimal setup.

llama.cpp, written in C++ by Georgi Gerganov, was the project that first made LLM inference accessible on CPUs and consumer GPUs. It introduced the GGUF format — a self-describing binary file that bundles quantized weights, tokenizer vocabulary, and model configuration into a single portable file. GGUF supports a range of quantization levels: Q2_K (smallest, most degraded), Q4_K_M (recommended sweet spot), Q5_K_M (excellent quality, slightly larger), Q8_0 (near-lossless, 2× the 4-bit size). The performance hit from Q4 quantization is real but modest: typical MMLU degradation is 1–3 percentage points relative to the 16-bit model, in exchange for a 4× reduction in model size and the ability to run on CPUs.

Ollama wraps llama.cpp with a higher-level interface: a model library (with one-line ollama pull commands), a REST API compatible with the OpenAI client, and automatic model management. You can run any GGUF model with ollama run model_name and interact with it via http://localhost:11434. For production local deployments, Ollama is typically the right choice; for maximum control over quantization parameters and sampling settings, llama.cpp directly is more powerful. Both support streaming output, context management, and the full suite of sampling hyperparameters (temperature, top-p, repetition penalty).

async def stream_ollama(model, messages, temperature=0.7):
    """Proxy streaming tokens from Ollama to the client via SSE."""
    payload = {
        "model":       model,
        "messages":    messages,        # ↑ OpenAI-compatible message list
        "stream":      True,            # ↑ token-by-token SSE streaming
        "options":     {"temperature": temperature},
    }
    async with httpx.AsyncClient(timeout=None) as client:
        async with client.stream(
            "POST",
            f"{OLLAMA_URL}/api/chat",
            json=payload
        ) as resp:
            async for line in resp.aiter_lines():
                if line:
                    chunk = json.loads(line)       # ↑ each line is a JSON chunk
                    token = chunk.get("message", {}).get("content", "")
                    if token:
                        yield f"data: {json.dumps({'token': token})}\n\n"
                    if chunk.get("done"):
                        yield "data: [DONE]\n\n"  # ↑ signal end of stream
                        return

@app.post("/chat")
async def chat(body: dict):
    return StreamingResponse(
        stream_ollama(
            model=body.get("model", "my-assistant"),
            messages=body["messages"],
            temperature=body.get("temperature", 0.7),
        ),
        media_type="text/event-stream"
    )

import time, random

def simulate_token_stream(text, avg_ms=60):
    """Simulate token-by-token streaming with realistic timing."""
    # Split into word-level "tokens" for demo (real tokenizer is subword)
    tokens = text.split()
    for tok in tokens:
        delay = random.gauss(avg_ms, 15) / 1000  # stochastic GPU timing
        yield tok + " ", max(0, delay)

def display_stream(token_gen):
    """Accumulate and display tokens as they arrive."""
    accumulated = []
    total_time  = 0.0
    for token, delay in token_gen:
        accumulated.append(token)
        total_time += delay

    full_text   = "".join(accumulated)
    token_count = len(accumulated)
    tps         = token_count / total_time if total_time > 0 else 0
    return full_text, token_count, total_time, tps

The Complete Fine-Tuning Pipeline

Every technique in this chapter — SFT, masked loss, LoRA, QLoRA, DPO, evaluation, deployment — is a piece of a larger pipeline. Understanding each piece in isolation is necessary but not sufficient; the hard part is knowing how they fit together and how to make decisions at each step based on what you actually have: the size of your dataset, the VRAM available, the nature of the task, and the quality bar you need to hit. This section ties the pieces into an end-to-end walkthrough.

The pipeline begins with your base model. A strong pretrained model makes everything downstream easier: better knowledge, better instruction-following latent capability, more signal per training step. For most applications, a Mistral-7B, Llama-3-8B, or Qwen2-7B base is the right starting point. From there, the decision tree branches primarily on dataset size. With fewer than 500 instruction examples, the risk of overfitting is high; use LoRA with a small rank (r=4 to r=8) and early stopping. Between 500 and 5,000 examples, LoRA (r=16) followed by DPO on high-quality preference pairs is the standard recipe. Above 5,000 examples, a full QLoRA fine-tune with a moderate rank (r=64) is worth considering; above 50,000, full fine-tuning on 16-bit weights if your hardware supports it.

The open-source ecosystem has matured rapidly. HuggingFace PEFT handles LoRA configuration and adapter management. TRL (Transformer Reinforcement Learning) provides the SFTTrainer and DPOTrainer classes that wrap all the complexity of masked loss, chat template application, and preference pair sampling. Axolotl provides a YAML-configurable wrapper over TRL for reproducible, multi-GPU fine-tuning with sensible defaults. LLaMA-Factory extends this with a web UI. For most practitioners, Axolotl is the right tool: it handles gradient checkpointing, LoRA rank selection, chat template selection, and evaluation integration, all from a single configuration file.

The most important step is data quality, not model size. A 7B model trained on 2,000 meticulously curated examples in your domain will outperform a 70B model trained on 50,000 noisy examples. Filter aggressively: remove examples with broken formatting, factual errors, inconsistent instruction style, or responses shorter than two sentences.

Closing: What You've Built

Seven chapters ago, a language model was a black box that somehow produced coherent text when you typed into a chat interface. Now you have the full picture, from first principles to production deployment. You started with raw text — bytes of web-scraped documents — and you understand how a BPE tokenizer converts them into integer sequences that a neural network can process. You built a transformer from scratch, and you understand why every component is there: the attention mechanism that lets tokens communicate across sequence positions, the positional encodings that give the model a sense of order, the feed-forward sublayers that perform per-token computation, the layer normalization that prevents gradient explosion, the residual connections that make deep networks trainable.

You saw how pretraining works at scale: the cross-entropy objective, the Adam optimizer, the learning rate schedule that warms up and decays, the data pipeline that never lets the GPU starve. You understand why scale matters (Chinchilla scaling laws), why data quality matters more than data quantity at the tail of the compute budget, and why the pretrained model is not the same as the assistant you interact with — the gap that this entire chapter was dedicated to closing. You learned how SFT bridges that gap by showing the model its job description in the form of instruction-response pairs, how masked loss ensures the model trains to respond rather than to repeat, and how LoRA and QLoRA make the whole process tractable on consumer hardware. You saw how DPO replaces an entire reinforcement learning pipeline with a stable supervised objective, and how to evaluate and deploy the result.

What comes next is an open frontier. Multimodal models (LLaVA, GPT-4V, Gemini) add vision encoders that project image patches into the same embedding space as text tokens, enabling reasoning over images, diagrams, and documents. Retrieval-augmented generation (RAG) augments fixed model knowledge with dynamic retrieval from external databases, addressing one of the most persistent limitations of parametric memory. Mixture-of-Experts (MoE) architectures — Mixtral, Grok-1, DeepSeek-MoE — dramatically increase model capacity without proportionally increasing inference cost by routing each token to a small subset of specialized feed-forward layers. Speculative decoding uses a small draft model to generate candidate tokens that a large model verifies in parallel, achieving 2–4× inference speedup with identical output quality. Reasoning models (o1, DeepSeek-R1) add an extended "chain of thought" generation phase that the model trains to use before producing a final answer, dramatically improving performance on math and logic tasks.

The direction the field is moving is unmistakable: smaller models trained on better data. Phi-3-mini (3.8B parameters) outperforms models 10× larger on many benchmarks, trained on carefully curated "textbook quality" data. Long-context models now support 128K–1M token contexts, enabling reasoning over entire codebases and books. The compute requirements for competitive performance at the research frontier have grown, but the compute requirements for building a genuinely useful domain-specific assistant have fallen dramatically. A single A100 GPU, a week of QLoRA training, and a well-curated 5,000-example dataset is enough to produce a model that outperforms GPT-3.5 on your specific task.

The papers that changed everything: Attention Is All You Need (Vaswani et al. 2017) gave us the transformer. Scaling Laws for Neural Language Models (Kaplan et al. 2020) gave us the compute-optimal recipe. InstructGPT (Ouyang et al. 2022) gave us alignment. LoRA (Hu et al. 2021) and QLoRA (Dettmers et al. 2023) gave us accessibility. DPO (Rafailov et al. 2023) gave us simplicity. Every model you interact with today is downstream of these six papers.
