5.1

Cross-Entropy Loss

Every language model needs a training signal: a number that says how wrong the current weights are, and which direction to push them. For language modeling, that signal is cross-entropy loss. It is the dominant objective in pretraining, fine-tuning, and reinforcement-learning-from-human-feedback alike. Understanding it from first principles is non-negotiable.

Cross-entropy comes from information theory. Claude Shannon, in his 1948 paper "A Mathematical Theory of Communication," defined entropy as the average number of bits needed to encode an event drawn from a probability distribution. If you have a fair coin, entropy is 1 bit. If one side is weighted 99:1, entropy is much lower — you can use a short code because the outcome is predictable. Cross-entropy measures how many bits you actually need when you encode events from the true distribution p using a code designed for the model's distribution q. The more q diverges from p, the more bits you waste — and the higher the loss.

In language modeling, the "true distribution" is a one-hot vector: the correct next token gets probability 1, every other token gets 0. So the cross-entropy formula collapses to a single log term: only the probability the model assigned to the correct token matters. If the model assigns probability 0.5 to the correct token, loss = -log(0.5) ≈ 0.69. If it assigns 0.01, loss = -log(0.01) ≈ 4.6. Perfect confidence gives loss = 0.

CROSS-ENTROPY (single token, one-hot target) L = -log( p_correct ) p_correct = softmax(logits)[true_token_index]

Over a full sequence of T tokens, we average the per-token losses. This is exactly the negative log-likelihood of the sequence under the model — the quantity the model is trained to minimize:

SEQUENCE CROSS-ENTROPY L = -(1/T) Σt=1..T log P( xt | x1...xt-1 ) Each token is conditioned on all prior tokens. The model predicts position t+1 from position 1..t.

At random initialization, the model distributes probability roughly uniformly across all vocabulary tokens. For GPT-2's vocabulary of 50,257 tokens, that's -log(1/50257) ≈ 10.82. A well-trained GPT-2 small achieves around 3.1 nats on its WebText test set. Every unit of loss reduction represents an enormous increase in the model's predictive precision — because of the exponential in the log.

Intuition check: A loss of 3.0 means the model gives the correct token roughly e-3.0 ≈ 5% probability on average. A loss of 1.0 means ~37% average probability. Perfect text memorization gives loss → 0. These numbers ground every training curve you will ever read.
cross_entropy.py 
import math

def softmax(logits):
    # ↑ convert raw scores to probabilities that sum to 1
    max_l = max(logits)                        # ↑ numerical stability: subtract max before exp
    exps = [math.exp(x - max_l) for x in logits]
    total = sum(exps)
    return [e / total for e in exps]

def cross_entropy_loss(logits, target_idx):
    # ↑ logits: raw model output (one float per vocab token)
    # ↑ target_idx: index of the correct next token
    probs = softmax(logits)
    p_correct = probs[target_idx]              # ↑ probability assigned to the right answer
    return -math.log(p_correct + 1e-12)        # ↑ +epsilon avoids log(0) = -inf

def sequence_cross_entropy(logits_list, targets):
    # ↑ average loss over an entire token sequence
    losses = [cross_entropy_loss(lg, tgt)
              for lg, tgt in zip(logits_list, targets)]
    return sum(losses) / len(losses)            # ↑ mean over T tokens

5.2

The Training Loop

The training loop is the engine that turns a loss function and a dataset into a trained model. Every deep learning framework wraps the same seven-step cycle, repeated for millions of iterations. Understanding each step — not just their order but why they exist — lets you debug training instabilities, interpret loss curves, and reason about computational cost.

The most subtle step is why we call zero_grad() at the start of each iteration. PyTorch, by design, accumulates gradients — each call to .backward() adds new gradient values to whatever is already stored in .grad. This is useful for gradient accumulation across micro-batches, but catastrophic if forgotten: without zeroing, you are computing the gradient of the sum of all past losses, not just the current batch. The model would behave as if it had seen every previous batch simultaneously, with increasingly stale data.

Validation is equally important as training loss monitoring. A model that memorizes its training set will show a decreasing training loss while validation loss plateaus or climbs. This divergence is the definition of overfitting for language models. In practice, pretraining of large models rarely overfits on web-scale data — the dataset is so large the model never sees the same sequence twice — but the validation check remains essential for detecting hardware faults, runaway learning rates, and dataset contamination.

1
Load Batch
Draw a batch of token sequences (input_ids, target_ids) from the DataLoader. Input is tokens 0..T-1, target is tokens 1..T (shifted by one).
2
Forward Pass
Run input_ids through the model. Each position predicts the next token. Output shape: (batch, seq_len, vocab_size) logits.
3
Compute Loss
Cross-entropy between predicted logits and target token IDs. PyTorch's F.cross_entropy() handles softmax internally.
4
Zero Gradients
optimizer.zero_grad() — clears accumulated gradients from the previous step. Forgetting this is one of the most common bugs in PyTorch.
5
Backward Pass
loss.backward() — automatic differentiation traces the computation graph, computing ∂L/∂w for every parameter w via the chain rule.
6
Clip Gradients
clip_grad_norm_(model.parameters(), max_norm=1.0) — prevents gradient explosion. Applied before the optimizer step, not after.
7
Optimizer Step
optimizer.step() — applies Adam/AdamW update rule: w ← w - lr · m̂ / (√v̂ + ε), where m̂ and v̂ are bias-corrected moment estimates.
Gradient accumulation: If your GPU can only fit batch_size=4 but you want effective_batch=32, run 8 forward+backward passes without calling zero_grad() or optimizer.step(). Divide the loss by 8 each time so gradients are averaged, not summed. Then clip and step once. This simulates a larger batch without extra memory.
train_loop.py 
import torch
from torch import nn

def train_one_epoch(model, dataloader, optimizer, scheduler, device,
                    grad_clip=1.0, val_loader=None, eval_every=200):
    model.train()
    total_loss = 0.0
    step = 0

    for input_ids, target_ids in dataloader:         # ↑ Step 1: get batch
        input_ids  = input_ids.to(device)
        target_ids = target_ids.to(device)

        logits = model(input_ids)                      # ↑ Step 2: forward pass → (B, T, V)
        B, T, V = logits.shape
        loss = nn.functional.cross_entropy(            # ↑ Step 3: compute loss
            logits.view(B * T, V),                     # ↑ reshape to (B*T, V) for F.cross_entropy
            target_ids.view(B * T)                     # ↑ targets flattened to (B*T,)
        )

        optimizer.zero_grad()                          # ↑ Step 4: clear old gradients
        loss.backward()                                # ↑ Step 5: backpropagate
        nn.utils.clip_grad_norm_(model.parameters(),   # ↑ Step 6: clip gradient norm
                                 max_norm=grad_clip)
        optimizer.step()                               # ↑ Step 7: update weights
        scheduler.step()                               # ↑ advance LR schedule

        total_loss += loss.item()
        step += 1

        if val_loader and step % eval_every == 0:
            val_loss = evaluate(model, val_loader, device)
            print(f"step {step:>6} | train {total_loss/step:.4f} | val {val_loss:.4f}")

    return total_loss / step

def evaluate(model, dataloader, device):
    # ↑ validation: no gradient computation needed (saves memory)
    model.eval()
    total_loss, steps = 0.0, 0
    with torch.no_grad():                              # ↑ disables autograd engine entirely
        for input_ids, target_ids in dataloader:
            logits = model(input_ids.to(device))
            B, T, V = logits.shape
            loss = nn.functional.cross_entropy(
                logits.view(B*T, V), target_ids.view(B*T).to(device))
            total_loss += loss.item()
            steps += 1
    model.train()
    return total_loss / steps

5.3

Perplexity

Cross-entropy loss is measured in nats (natural-log units) or bits (log-2 units), and a loss of 3.1 is hard to reason about intuitively. Perplexity converts this number into something more interpretable: the effective vocabulary size the model is choosing from at each step. A perplexity of 100 means the model is, on average, as confused as if it had to pick uniformly among 100 equally likely options. A perplexity of 1 means perfect prediction every time.

PERPLEXITY FROM CROSS-ENTROPY PPL = exp( L ) where L = -(1/T) Σ log P(xt | x<t) A loss drop of 1.0 nat corresponds to PPL / e ≈ PPL / 2.718 (multiplicative, not additive)

Concretely: GPT-2 small at random initialization has PPL ≈ exp(10.82) ≈ 50,000 (essentially the full vocabulary). After training on WebText, it reaches PPL ≈ exp(3.1) ≈ 22. GPT-4, benchmarked on Penn Treebank with the right tokenization, achieves single-digit perplexity. Each unit of loss improvement is a dramatic collapse in the model's effective uncertainty.

There is a critical benchmarking trap here. Perplexity comparisons are only valid when two conditions hold: the models use identical tokenization, and they are evaluated on identical test sets. A model with a larger vocabulary will assign different probability masses per-token even if the underlying language model is equally powerful. Bengio et al. (2003) first noted this in their seminal neural language model paper, and it remains a source of faulty comparisons in published benchmarks today. When you see two perplexity numbers in a paper, check the footnotes for tokenizer alignment before drawing conclusions.

PPL drop vs. loss drop: Because PPL = exp(L), a loss improvement from 4.0 → 3.0 (delta = 1.0) reduces PPL from 54.6 → 20.1 (delta = 34.5). A further improvement from 3.0 → 2.0 reduces PPL from 20.1 → 7.4 (delta = 12.7). Equal loss improvements become smaller perplexity improvements as the model gets better — a natural consequence of the exponential scale.

5.4

Learning Rate Scheduling

A fixed learning rate is a poor choice for neural network training. In the first steps, the model's gradients are large and noisy — the weights are random, the loss surface is steep, and any large step risks bouncing off cliffs or into saddle points. Later in training, as the model converges toward a minimum, large steps cause oscillation rather than precise descent. Learning rate scheduling solves both problems: start small, peak, then decay.

Linear warmup addresses the early instability. By ramping the learning rate from nearly zero up to lr_max over the first warmup_steps, we allow the model to take many small exploratory steps before committing to a direction. GPT-3 uses 375 million tokens of warmup (roughly 1,500 gradient steps with batch size 0.5M tokens). The warmup phase also gives Adam's moment estimates time to warm up — Adam's bias correction is less effective in the first few steps, so a small LR limits the damage.

Cosine decay is the dominant schedule after warmup. After reaching lr_max, the learning rate follows a half-cosine curve descending to lr_min (typically 10% of lr_max). Cosine decay is smooth and schedule-free — no hyperparameters beyond T_max and lr_min — and empirically outperforms linear or step decay at large scale. The formula is: lr(t) = lr_min + 0.5 · (lr_max − lr_min) · (1 + cos(π · t / T)), where t is the current step after warmup and T is the total post-warmup steps.

COSINE DECAY SCHEDULE lr(t) = lr_min + 0.5 · (lr_max - lr_min) · (1 + cos(π · t / T)) t = steps since end of warmup. T = total decay steps. At t=0: lr=lr_max. At t=T: lr=lr_min.
GPT-3 schedule: lr_max = 6×10-4, warmup = 375M tokens (~1,500 steps), cosine decay to 10% (6×10-5) over 300B tokens total. Batch size = 3.2M tokens. Total steps ≈ 93,750. Note: at 175B parameters, even one step processes an enormous amount of compute — the schedule is tuned per-run, not universal.
▶ Python 3 · lr_schedule.py 
import math

def get_lr(step, warmup_steps, total_steps, lr_max, lr_min):
    """Cosine schedule with linear warmup."""
    if step < warmup_steps:
        # linear warmup from 0 to lr_max
        return lr_max * (step / warmup_steps)
    # cosine decay from lr_max to lr_min
    progress = (step - warmup_steps) / (total_steps - warmup_steps)
    cosine = 0.5 * (1 + math.cos(math.pi * progress))
    return lr_min + cosine * (lr_max - lr_min)

warmup_steps = 100
total_steps  = 1000
lr_max       = 6e-4
lr_min       = 6e-5   # 10% of max

print(f"{'Step':>6}  {'LR':>12}  {'Phase':<14}  {'Bar'}")
print("-" * 55)

for step in checkpoints:
    lr = get_lr(step, warmup_steps, total_steps, lr_max, lr_min)
    phase = "warmup" if step < warmup_steps else "cosine decay"
    bar_len = int((lr / lr_max) * 30)

5.5

Gradient Clipping

Even with careful learning rate scheduling, training deep networks occasionally produces runaway gradients. A single batch with an unusual token distribution, a poorly initialized layer, or a rare idiom in the data can cause the gradient norm to spike by orders of magnitude. Without protection, this spike updates weights so aggressively that the model "forgets" everything it has learned, and training effectively restarts from a corrupted state. Gradient clipping is the circuit breaker that prevents this.

The standard approach is global norm clipping. Compute the L2 norm of the concatenation of all gradient vectors across all parameters. If this global norm exceeds a threshold (typically 1.0), scale every gradient proportionally downward so the total norm exactly equals the threshold. This preserves the direction of the gradient — you are still descending in the right direction — but limits the step size. It is analogous to anti-lock brakes: you can steer and slow down simultaneously, because the clipping allows you to maintain traction rather than locking up on sudden deceleration.

The max_norm threshold of 1.0 is nearly universal across GPT-2, GPT-3, LLaMA, and Mistral. Raising it to 5.0 or 10.0 allows more aggressive steps and can speed up convergence in stable phases, but dramatically increases the risk of loss spikes. Lowering it to 0.5 produces smoother but slower training. In practice, monitoring the gradient norm itself — logging it every step — is informative: persistent norms above 0.5 suggest the learning rate may be too high.

GLOBAL GRADIENT NORM CLIPPING g_norm = √( Σp Σi gp,i2 )
if g_norm > max_norm: gp ← gp · (max_norm / g_norm) All parameter gradients are scaled by the same factor, preserving gradient direction.
Clipping vs. normalization: Gradient clipping only acts when the norm exceeds the threshold — it is a safety valve, not a constant normalization. Gradient normalization (setting norm=1.0 always) would prevent the model from taking larger, well-justified steps during stable phases. Clipping gives you the best of both: unconstrained updates when things are fine, safety when things are not.
gradient_clipping.py 
import math

def compute_grad_norm(gradients):
    # ↑ gradients: list of lists (one list of values per parameter tensor)
    total_sq = 0.0
    for grad in gradients:
        for g in grad:
            total_sq += g * g                  # ↑ accumulate squared gradient values
    return math.sqrt(total_sq)                 # ↑ global L2 norm across all parameters

def clip_gradients(gradients, max_norm):
    # ↑ returns scaled gradients and the pre-clip norm (for logging)
    norm = compute_grad_norm(gradients)
    if norm > max_norm:
        scale = max_norm / (norm + 1e-6)         # ↑ +epsilon prevents division by zero
        gradients = [[g * scale for g in param]
                     for param in gradients]      # ↑ uniform scale across all params
    return gradients, norm                       # ↑ return pre-clip norm for monitoring

5.6

Mixed Precision: BF16 vs FP32

A 7-billion-parameter model stored in FP32 requires 28 GB of GPU memory (4 bytes per parameter). The same model in BF16 requires 14 GB. But the memory savings are only the beginning: modern NVIDIA GPUs (A100, H100) execute BF16 matrix multiplications at roughly 3× the throughput of FP32. Mixed-precision training — running the forward and backward passes in BF16 while keeping a FP32 master copy of weights for the optimizer — makes large-model training feasible on hardware that would otherwise be impossible.

Understanding why BF16 is preferred over the older FP16 requires looking at the exponent bits. FP16 has 5 exponent bits, giving it a maximum representable value of 65,504. During a transformer's forward pass — especially through layer normalization and attention softmax — intermediate activations can easily exceed this range, producing NaN (Not a Number) values that cascade through the entire model. BF16, despite also being 16 bits, uses 8 exponent bits — the same as FP32 — giving it a dynamic range of roughly ±3.4×1038. The cost is reduced mantissa precision (7 bits vs FP32's 23), which introduces small rounding errors that are acceptable for training but catastrophic for FP16's overflow.

Mixed-precision training uses BF16 for the forward and backward passes (fast computation, low memory), but maintains a FP32 master copy of the weights that the optimizer updates. The FP32 weights are cast to BF16 before each forward pass. This preserves the small gradient signals that would be lost if the optimizer worked in BF16 directly — the Adam update involves subtracting numbers of very similar magnitude, which requires full precision to avoid cancellation errors.

PyTorch autocast: with torch.autocast(device_type='cuda', dtype=torch.bfloat16): wraps your forward pass. PyTorch automatically selects which operations run in BF16 (matmuls, convolutions) and which stay in FP32 (reductions, layer norm). You do not need to manually cast anything. Combined with a GradScaler (for FP16) or just autocast alone (for BF16), this is the full recipe.

Gradient Checkpointing

Backpropagation through a deep neural network requires storing all intermediate activations from the forward pass — every layer's output is needed to compute that layer's gradients. For a 12-layer transformer with a sequence length of 2048 and a model dimension of 768, the activation memory per sample is roughly 12 × 2048 × 768 × 2 bytes (BF16) ≈ 38 MB. At a batch size of 8, that is 300 MB per training step just for activations, before any consideration of weights or optimizer state. For 70-billion-parameter models, this activation memory becomes the dominant memory cost, not the weights.

Gradient checkpointing (also called activation recomputation) breaks this linear relationship. Instead of storing all activations, you store only the activations at certain "checkpoint" boundaries — typically every N transformer blocks. During the backward pass, when a layer's gradient is needed, its input is fetched from the nearest checkpoint and the forward pass is re-run on that segment to recompute the needed activations. You are trading compute (a 30–40% increase in FLOPs) for memory (a 5–10× reduction in activation memory). For multi-hundred-billion-parameter models, this trade is non-negotiable.

In PyTorch, wrapping a module with torch.utils.checkpoint.checkpoint() implements this automatically. The function discards all intermediate activations during the wrapped forward pass and re-runs the forward computation during the backward pass. The key constraint: the checkpointed function must be deterministic and free of side effects, because it will be executed twice. For transformer blocks with dropout, the random number state must be preserved between the two runs — PyTorch handles this automatically.

Memory arithmetic: Without checkpointing, activation memory scales as O(n_layers × seq_len × d_model). With checkpointing at every layer, it scales as O(seq_len × d_model) — constant in depth. The cost: each layer's forward pass runs twice per training step. For a 96-layer GPT-4-scale model, checkpointing every layer would reduce activation memory by 96× at the cost of 2× FLOPs. In practice, checkpointing every 4–8 layers gives a better trade-off.

import torch
from torch.utils.checkpoint import checkpoint

class TransformerBlock(torch.nn.Module):
    def __init__(self, d_model, n_heads):
        super().__init__()
        self.attn  = torch.nn.MultiheadAttention(d_model, n_heads, batch_first=True)
        self.norm1 = torch.nn.LayerNorm(d_model)
        self.ff    = torch.nn.Sequential(
            torch.nn.Linear(d_model, d_model * 4),
            torch.nn.GELU(),
            torch.nn.Linear(d_model * 4, d_model)
        )
        self.norm2 = torch.nn.LayerNorm(d_model)

    def forward(self, x):
        attn_out, _ = self.attn(x, x, x)
        x = self.norm1(x + attn_out)
        x = self.norm2(x + self.ff(x))
        return x

class GPTWithCheckpointing(torch.nn.Module):
    def __init__(self, n_layers, d_model, n_heads, use_ckpt=True):
        super().__init__()
        self.blocks = torch.nn.ModuleList(
            [TransformerBlock(d_model, n_heads) for _ in range(n_layers)])
        self.use_ckpt = use_ckpt

    def forward(self, x):
        for block in self.blocks:
            if self.use_ckpt:
                x = checkpoint(block, x, use_reentrant=False)
                # ↑ discards activations; recomputes during backward
            else:
                x = block(x)               # ↑ stores all activations (full memory)
        return x

def estimate_activation_memory(n_layers, seq_len, d_model,
                                 batch_size, bytes_per_elem=2,  # BF16
                                 checkpoint_every=None):
    """
    Without checkpointing: store activations for all layers.
    With checkpointing every N layers: store only N checkpoint inputs.
    """
    # Per-layer activation size (simplified: attention + MLP intermediates)
    # attention: Q,K,V matrices + attention weights + output
    attn_mem = 3 * seq_len * d_model  + seq_len * seq_len + seq_len * d_model
    # MLP: intermediate 4*d_model
    mlp_mem  = 2 * seq_len * d_model + seq_len * 4 * d_model
    per_layer = (attn_mem + mlp_mem) * bytes_per_elem * batch_size

    if checkpoint_every is None:
        total = per_layer * n_layers
        strategy = "no checkpointing"
    else:
        checkpoints = n_layers // checkpoint_every
        total = per_layer * checkpoint_every + per_layer * checkpoints
        strategy = f"checkpoint every {checkpoint_every} layers"

    return total / (1024**2), strategy  # return in MB

Temperature Sampling

Once a language model is trained, generating text requires a sampling strategy. The model produces a vector of logits — one raw score per vocabulary token — and we must convert these into a token choice. The simplest approach is greedy decoding: always pick the token with the highest logit. Greedy decoding is deterministic and maximally confident, but it produces repetitive, generic text. Real language has natural variability; the same sentence beginning can end many reasonable ways, and a model that always picks the single most likely continuation sounds flat and mechanical.

Temperature controls how sharp or flat the probability distribution is before sampling. The trick is to divide all logits by a temperature T before computing the softmax. When T=1, the distribution is unchanged. When T approaches 0, the highest-logit token gets nearly all the probability mass — approaching greedy. When T>1, the distribution flattens; low-probability tokens get relatively more weight, and output becomes more varied (at the cost of occasional incoherence). Temperature does not change which tokens are possible, only how likely each is to be selected.

The intuition: imagine each logit as a temperature in Celsius. At T=0.5, you are dividing by 0.5 (multiplying by 2) — amplifying differences, making the distribution more extreme. At T=2, you are cutting each difference in half — making the model less decisive. For creative tasks like fiction writing, T≈0.8–1.2 produces natural, varied output. For factual tasks like code generation or question answering, T≈0.1–0.3 keeps the model focused on high-probability, correct tokens.

TEMPERATURE SAMPLING pi = exp(zi / T) / Σj exp(zj / T) z_i = logit for token i. T → 0: greedy. T = 1: standard softmax. T → ∞: uniform.

GREEDY (T≈0)

Always selects the single highest-probability token. Deterministic. Prone to repetition.

"The cat sat on the mat. The cat sat on the mat."

TEMP=0.8

Sharpened distribution. High-probability tokens dominate but variation exists. Best for most tasks.

import math, random

def softmax_temp(logits, temperature):
    """Apply temperature then softmax."""
    if temperature < 1e-8:
        # greedy: one-hot on argmax
        max_idx = max(range(len(logits)), key=lambda i: logits[i])
        return [1.0 if i == max_idx else 0.0 for i in range(len(logits))]
    scaled = [z / temperature for z in logits]
    max_s = max(scaled)
    exps = [math.exp(s - max_s) for s in scaled]
    total = sum(exps)
    return [e / total for e in exps]

def sample_token(probs):
    r = random.random()
    cumulative = 0.0
    for i, p in enumerate(probs):
        cumulative += p
        if r <= cumulative:
            return i
    return len(probs) - 1

Top-k and Nucleus Sampling

Temperature alone does not solve all sampling problems. Consider a token position where the model is very confident: the top token has 95% probability, and the remaining probability mass is spread across 50,000 tokens at nearly 0.001% each. With temperature=1.0, the model might still occasionally pick one of these fringe tokens — an absurd word that breaks the sentence. Top-k sampling addresses this by zeroing out all but the k most probable tokens before sampling, ensuring the model can only pick from a reasonable set of candidates.

Top-k has its own failure mode. At positions where the model is genuinely uncertain — say, a comma following a noun could be followed by any of dozens of plausible continuations — top-k=50 might arbitrarily cut off at the 50th most likely option even though tokens 51–150 are all plausible and nearly equally likely. The truncation is too rigid. Nucleus sampling (top-p) solves this by specifying a probability mass threshold rather than a token count. Keep the minimum number of tokens that together cover at least p=0.9 of the probability mass, then renormalize and sample from that set. When the model is confident, the nucleus is small (maybe 2–3 tokens). When the model is uncertain, the nucleus expands to include more options.

In practice, top-p=0.9 or top-p=0.95 combined with temperature=0.7 is the most common production configuration. Many modern models (Claude, GPT-4, Llama 2) use top-p sampling by default. The two can be combined: apply temperature first (rescale logits), then apply top-p to the rescaled probabilities. This gives you both the sharpening effect of temperature and the adaptive cutoff of nucleus sampling.

Why k=50? The k=50 default in many implementations is a heuristic from GPT-2 era experiments. For modern large models (7B+), the probability distribution is sharper, and k=20–40 is often sufficient. For creative tasks with smaller models, k=100 or even k=200 can produce more interesting output. Top-p=0.9 is more principled than top-k precisely because it adapts to the model's confidence automatically.
TOP-P (NUCLEUS) SAMPLING S = smallest set such that Σi ∈ S pi ≥ p_threshold
p'i = pi / Σj ∈ S pj for i ∈ S, else 0 S is the "nucleus." Size of S varies per step: 1 token when confident, many tokens when uncertain.

import math, random

def softmax(logits):
    m = max(logits)
    e = [math.exp(x - m) for x in logits]
    s = sum(e)
    return [x/s for x in e]

def top_k_filter(probs, k):
    """Zero out all but top-k probabilities, then renormalize."""
    # sort indices by probability descending
    indexed = sorted(enumerate(probs), key=lambda x: x[1], reverse=True)
    filtered = [0.0] * len(probs)
    for rank, (idx, p) in enumerate(indexed):
        if rank < k:
            filtered[idx] = p
    total = sum(filtered)
    return [p / total for p in filtered]

def top_p_filter(probs, p_thresh):
    """Keep minimum set of tokens covering p_thresh probability mass."""
    indexed = sorted(enumerate(probs), key=lambda x: x[1], reverse=True)
    filtered = [0.0] * len(probs)
    cumulative = 0.0
    for idx, p in indexed:
        filtered[idx] = p
        cumulative += p
        if cumulative >= p_thresh:
            break
    total = sum(filtered)
    nucleus_size = sum(1 for p in filtered if p > 0)
    return [p / total for p in filtered], nucleus_size

def sample(probs):
    r = random.random()
    cum = 0.0
    for i, p in enumerate(probs):
        cum += p
        if r <= cum:
            return i
    return len(probs) - 1

Loading Pretrained Weights

OpenAI released GPT-2 weights publicly in 2019, and they remain among the most studied and reimplemented model weights in existence. Loading them into a custom PyTorch model — rather than using a HuggingFace wrapper — is an instructive exercise because it forces you to confront precisely how weights map between architectures, where naming conventions differ, and why some weight matrices must be transposed. Andrej Karpathy's nanoGPT repository is the canonical reference for this process.

The core challenge is the state dict mismatch. PyTorch's nn.Linear stores its weight as shape (out_features, in_features) — the matrix is already transposed relative to the mathematical convention W where y = Wx. HuggingFace's GPT-2 weights follow the convention of TensorFlow's original release, where the weight is stored as (in_features, out_features). This means every projection matrix in attention (Q, K, V projections, output projection) and in the MLP must be transposed when loading from HuggingFace format. Miss this and your model will produce garbage output despite nominally correct architecture parameters.

Beyond transpositions, you must carefully map key names between the source and target state dicts. HuggingFace uses names like transformer.h.0.attn.c_attn.weight while your custom model might call it blocks.0.attn.qkv.weight. The weights themselves are identical; only the paths differ. A systematic mapping function that iterates both state dicts simultaneously and matches by shape and position is the cleanest approach. As an additional sanity check, always verify that the loaded model achieves approximately the same perplexity as the original on a reference text.

HuggingFace GPT-2 weight shapes to transpose: c_attn.weight (combined QKV, shape 768×2304 in TF convention), c_proj.weight (output projection), c_fc.weight (MLP first layer), mlp.c_proj.weight (MLP second layer). Embedding and LayerNorm weights do not need transposition.

import torch
from transformers import GPT2Model

def load_pretrained_weights(our_model, model_type="gpt2"):
    # ↑ our_model: custom GPT in PyTorch, must have same n_layers/d_model
    print(f"Loading weights from HuggingFace: {model_type}")
    hf_model = GPT2Model.from_pretrained(model_type)
    hf_sd = hf_model.state_dict()                    # ↑ HuggingFace state dict
    our_sd = our_model.state_dict()

    # Keys that need transposition: Conv1D -> Linear convention mismatch
    transpose_keys = ["attn.c_attn.weight", "attn.c_proj.weight",
                      "mlp.c_fc.weight",   "mlp.c_proj.weight"]

    copied = 0
    for our_key in our_sd.keys():
        hf_key = our_key                             # ↑ assume 1-to-1 key mapping

        # Map our key naming to HuggingFace naming if different
        hf_key = hf_key.replace("blocks.", "h.")
        hf_key = hf_key.replace("ln_final", "ln_f")

        if hf_key in hf_sd:
            src = hf_sd[hf_key]
            needs_t = any(hf_key.endswith(k) for k in transpose_keys)
            if needs_t:
                src = src.t()                        # ↑ transpose Conv1D weights to Linear shape
            our_sd[our_key].copy_(src)               # ↑ copy in-place into our model
            copied += 1

    print(f"Loaded {copied}/{len(our_sd)} weight tensors.")
    our_model.load_state_dict(our_sd)
    return our_model

def matmul(A, B):
    """Simple matrix multiply A (m×n) @ B (n×p) -> (m×p)."""
    m, n = len(A), len(A[0])
    p = len(B[0])
    C = [[0.0]*p for _ in range(m)]
    for i in range(m):
        for k in range(n):
            for j in range(p):
                C[i][j] += A[i][k] * B[k][j]
    return C

def transpose(M):
    rows, cols = len(M), len(M[0])
    return [[M[r][c] for r in range(rows)] for c in range(cols)]

Scaling Laws

One of the most surprising findings in deep learning is that language model performance obeys predictable power laws across many orders of magnitude of model size, data size, and compute budget. Kaplan et al. (2020) at OpenAI showed that cross-entropy loss decreases smoothly as you scale any of these three dimensions, and that the relationship is well-described by L ∝ 1/N0.076 for model parameters N and L ∝ 1/D0.095 for dataset tokens D. These small exponents mean you need roughly 10× more parameters to get a loss reduction equivalent to doubling your dataset size — but the relationship is reliable enough to project forward to untrained scales.

The more impactful implication from the Kaplan paper was about compute-optimal training. Given a fixed compute budget C (measured in FLOPs), what is the best allocation between model size N and training tokens D? Kaplan et al. concluded that it was usually better to train a very large model for a relatively short time than to train a small model exhaustively. This guided OpenAI's strategy for GPT-3: 175B parameters trained on 300B tokens — roughly 1.7 tokens per parameter.

Hoffmann et al. (2022) at DeepMind challenged this conclusion with the Chinchilla paper. They ran a more careful scaling experiment with many more data points at smaller scales and derived a different optimal ratio: approximately 20 tokens per parameter. The Chinchilla-optimal 70B model trained on 1.4T tokens (Chinchilla itself) outperformed GPT-3 on almost every benchmark despite being 2.5× smaller. This is the "Chinchilla finding" that reshaped the field: GPT-3 was severely undertrained, not because 300B tokens is small, but because 175B parameters demands 3.5T tokens to reach compute-optimality.

import math

def chinchilla_optimal(compute_flops):
    """
    Given a compute budget in FLOPs, compute the Chinchilla-optimal
    N (params) and D (tokens).

    From Hoffmann et al. 2022:
    C ≈ 6*N*D (FLOPs for forward+backward)
    Optimal: N ≈ sqrt(C / 120), D ≈ 20*N
    """
    N_opt = math.sqrt(compute_flops / 120)
    D_opt = 20 * N_opt
    return N_opt, D_opt

    print(f"{name:<22}  {C:>16.2e}  {fmt(N_opt):>13}  {fmt(D_opt):>13}  {ratio:>9.1f}x")

print()
print("Key insight: Chinchilla says optimal token:param ratio is ~20x.")
print("GPT-3 used 1.7x — was undertrained by a factor of ~12x in tokens.")

Summary

Chapter 5 in Brief

This chapter traced the full arc from loss function to trained model. Cross-entropy loss gives us a differentiable signal from raw text, requiring no labels — only the next-token prediction objective. The seven-step training loop (load, forward, loss, zero_grad, backward, clip, step) is the universal engine that processes this signal. Perplexity translates loss into an interpretable quantity but is only comparable across identical tokenization setups.

Four numerical stability systems — learning rate scheduling, gradient clipping, mixed precision (BF16), and gradient checkpointing — make training at scale feasible. Without them, large models either diverge, overflow, or exhaust GPU memory. At inference time, temperature, top-k, and nucleus sampling control the balance between confidence and creativity in generated text, with nucleus sampling being the most principled adaptive approach.

Loading pretrained weights requires careful attention to weight transposition conventions between TensorFlow-origin models and PyTorch. Finally, scaling laws — both the Kaplan power laws and the Chinchilla correction — show that the field's understanding of optimal resource allocation has shifted: not merely "bigger is better," but "bigger and longer-trained together, at the right ratio" is what determines quality per compute dollar.
